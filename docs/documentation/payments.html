Оплаты: как работают таблицы payment_intents, payables, payments, users_prices

Файл: /docs/documentation/payments.html

1) Зачем нужны эти таблицы

  • payables — что именно оплачивается (покупка/обязательство: месяц, клубный взнос, форма, лагерь).
  • payment_intents — технический трек попытки оплаты у провайдера (Robokassa и т.д.).
  • payments — финансовый факт: деньги реально поступили (запись для отчётов).
  • users_prices — состояние начислений и оплат по месяцам в разрезе группы (проекция для UI «Цены», витрины оплаты и отчёта «Задолженности»).
Ключевой принцип: разовые оплаты (club_fee/uniform/camp) не пишем в users_prices.
В users_prices отражаются только месячные начисления и их оплата (monthly_fee).
У ученика в нескольких группах — отдельная строка на каждую пару (user_id, team_id, new_month). См. §2.1 и ученик ↔ группы.

2) Что хранится в каждой таблице

payables (доменная покупка)

  • type: monthly_fee, club_fee, custom_payment_fee, lesson_package_fee, uniform, camp
  • amount/currency: сумма и валюта
  • status: pending → paid (дальше возможны refund-статусы)
  • month (только для monthly_fee): дата YYYY-MM-01
  • meta: доп. параметры (например {month: "2025-09-01"}, {team_id: …} для месячного абонемента, {user_period_price_id: …} для доп. платежа, {user_lesson_package_id: …} для назначенного абонемента, размер формы, смена лагеря…)

payment_intents (попытка оплаты у провайдера)

  • provider: например robokassa
  • provider_inv_id: внешний InvId у провайдера (по нему приходят webhooks)
  • tbank_payment_id / tbank_order_id (для provider=tbank): идентификаторы T‑Bank (PaymentId/OrderId) для связывания с уведомлениями
  • payable_id: ссылка на payables (что именно оплачивается)
  • status: pending / paid / failed
  • out_sum: ожидаемая сумма
  • payment_date: строковая метка платежа для провайдера (Shp_paymentDate у Robokassa), не тип datetime в БД. Примеры: 2026-06-01 (месячный абонемент), Клубный взнос, Дополнительный платеж, подпись абонемента lesson package. В отчёте «Платежные запросы» для UI форматируется в «месяц год» при распознавании даты (PaymentIntentReportController::formatPaymentDateForReport); факт оплаты — поле paid_at.
  • paid_at: timestamp успешной оплаты (nullable)
  • meta: диагностика интеграции

payments (финансовый журнал)

  • Смысл: фиксирует факт поступления денег.
  • Типично храним: partner_id, user_id, location_id (опциональный снимок при первом create, только если передан явно), team_id / team_title (снимок оплаченной группы при Init с payables.meta.team_id, §2.2), summ, operation_date, payment_number (обычно = provider_inv_id / InvId).
  • Важно: запись в payments появляется только после успешного webhook.
  • Локация в отчётах: payments.location_id — опциональный снимок при первом создании строки (только если location_id передан в атрибутах записи); дальше не перезаписывается. Используется в отчётах «Платежи», «Платежи по месяцам», LTV — см. reports-admin.html (§2.1). Отчёт «Задолженности»: фильтр локации — по группам ученика в pivot; фильтр группы для месячных долгов — по users_prices.team_id (§2.1).

users_prices (начисления и оплаты по месяцам и группам)

  • user_id — ученик
  • team_id — группа начисления (NOT NULL); UNIQUE (user_id, team_id, new_month) (миграция 2026_06_21_120000_add_team_id_to_users_prices_table.php)
  • new_month: дата YYYY-MM-01
  • price: начисленная сумма к оплате за месяц в этой группе (строка/число в рублях; для витрины оплаты и Init — источник истины после нормализации)
  • is_paid: автоматическая отметка из webhook (monthly_fee)
  • is_manual_paid, manual_paid_at, manual_paid_by: ручной override из «Цены» (setPrices.manualPaid.manage); для UI/отчётов — accessor effective_is_paid (ручной важнее автоматического)

2.1) Начисления по группам: «Цены», витрина, webhook

Источник группы — pivot team_user (см. student-team-membership). Legacy-колонка users.team_id в расчётах не участвует.
  • Админка «Цены» (/admin/setting-prices, вкладка users): список учеников выбранной группы; при открытии месяца для группы создаётся/читается строка users_prices с team_id этой группы (SettingPricesController, firstOrCreate по (user_id, team_id, new_month)).
  • Ручная оплата (setManualPaid): обязательны user_id, team_id, месяц; ученик должен состоять в группе (UserPriceTeamMembership::studentBelongsToTeam).
  • Dashboard → витрина: в форму оплаты передаётся hidden team_id из ячейки месяца (dashboard.blade.php → paymentUser.blade.php). У ученика с 2+ группами в пользовательском режиме консоли отображается только начисление выбранной группы (селект «Выбор группы», § dashboard-cabinet §4).
  • Resolver суммы — App\Services\Payments\UserPriceMonthlyFeePaymentResolver:
    • если передан team_id — проверка членства и чтение users_prices по (user_id, team_id, month);
    • если не передан и за месяц ровно одна строка с price > 0 — берётся её team_id;
    • если несколько строк — 422 «Укажите группу для оплаты за этот месяц.»;
    • если строк нет — fallback на UserPriceTeamMembership::primaryTeamIdForStudent (первая группа в pivot по order_by, title).
  • Старт оплаты (Robokassa / T‑Bank): в payables.meta.team_id сохраняется группа платежа для всех типов оплат (PayableTeamResolver, §2.3).
  • Webhook (успех / refund): UserPrice::markMonthlyPaid($userId, $month, $teamId) — updateOrCreate по (user_id, team_id, new_month); team_id из payables.meta, иначе primary group из pivot.
  • Отчёт «Задолженности»: месячные долги — одна строка на каждую неоплаченную запись users_prices; фильтр filter_team_id для месячной части — по users_prices.team_id (см. reports-admin §3).

2.2) Снимок группы в журнале payments (team_id / team_title)

Миграция: 2026_06_21_140000_add_team_id_to_payments_table.php — nullable payments.team_id + индекс. Снимок нужен, когда ученик состоит в нескольких группах: в отчёте «Платежи» и в пользовательском «Мои платежи» видна конкретная оплаченная группа, а не все группы ученика через запятую.
  • Когда заполняется: при первой записи строки в payments через PaymentLedgerRecorder (Robokassa ResultURL, T‑Bank webhook). Поля team_id и team_title задаются только при первом create (как location_id), дальше не перезаписываются.
  • Источник: сервис App\Services\Payments\PaymentLedgerTeamResolver::resolveFromPayable → делегирует в PayableTeamResolver::resolveFromPayable (§2.3):
    • для всех типов — сначала payables.meta.team_id (снимок на Init);
    • для custom_payment_fee — fallback на user_custom_payment.team_id;
    • для lesson_package_fee — fallback на user_lesson_packages.team_id или группу из календаря;
    • для monthly_fee без meta — primary group из pivot.
  • Отображение в отчётах: хелпер App\Support\Payments\PaymentTeamTitleDisplay::forRow — если у строки payments.team_id > 0, показывается название этой группы (paidTeam или снимок team_title); иначе — все группы ученика из pivot (старые платежи до миграции).
  • Фильтр «Группа» в админ-отчёте «Платежи» (filter_team_id): UserTeamQuery::applyPaymentLedgerTeamFilters — платёж попадает, если payments.team_id = … или (payments.team_id IS NULL и ученик состоит в группе по pivot). Подробно — reports-payments §1.1.
  • LTV и «Платежи по месяцам» по-прежнему фильтруют по членству ученика в pivot (applyReportTeamFilters), не по снимку payments.team_id.
  • Legacy: у старых club_fee без payables.meta.team_id журнал может остаться с payments.team_id = NULL (новые оплаты всегда пишут meta на Init).
  • T‑Bank Init: в DATA.team_id передаётся группа для всех типов оплат (обязательно для маршрутизации выплаты на юр. лицо).
  • Robokassa ResultURL: при восстановлении payable, если intent.payable_id пуст, выбирается pending-payable с тем же meta.team_id, что в payment_intents.meta (RobokassaController::resolveTeamIdFromIntentMeta).

2.3) Группа платежа и юр. лицо (PayableTeamResolver)

Сервис: App\Services\Payments\PayableTeamResolver. Используется при Init оплаты, в LegalEntityResolver::forPayable, в журнале payments.
Важно: без определённой группы новая оплата не инициируется; выплата не уходит на уровень партнёра без группы. ShopCode и реквизиты — только из активного юр. лица справочника (без fallback на partners).
Тип payableОткуда берётся team_id
monthly_feeusers_prices / payables.meta.team_id
club_feeформа: 1 группа — авто; 2+ — обязательный выбор; 0 групп — запрет оплаты
custom_payment_feeuser_custom_payment.team_id (задаётся в админке при создании)
lesson_package_feeпри ≥2 юр. лицах — user_lesson_packages.team_id при назначении; иначе календарь → primary group

Юр. лицо для выплаты: teams.legal_entity_id → активное юр. лицо с ShopCode. При одном активном юр. лице без привязки группы резолвер использует единственное юр. лицо. При ≥2 активных — без явной привязки Init T‑Bank запрещён (см. §2.3.5, admin-legal-entities §1.1).

Backfill (миграция 2026_06_29_160000_backfill_team_id_on_payment_assignments.php): для старых строк без team_id в user_custom_payment и user_lesson_packages — primary group ученика (для абонемента сначала группа из календаря). Сервис: PaymentAssignmentTeamBackfill.

Админка доп. платежей (/admin/setting-prices/custom-payments): при создании обязателен селект «Группа»; в таблице — колонка «Группа» (team_label в JSON /custom-payments/data).

Назначения абонементов (/admin/lesson-packages/assignments): при ≥2 юр. лицах — обязательный селект «Группа» при создании; в таблице — колонка «Группа» (team_label в JSON /assignments/data). См. lesson-packages §assignments-tab.

2.3.2) Юр. лицо и выплата (LegalEntityResolver)

Для новых оплат LegalEntityResolver::forPayable не вызывает forPartner без группы: если из payable не удалось получить team_id → LegalEntityResolution(entity: null) (Init / выплата без ShopCode по группе не строятся).
Цепочка: payables.meta.team_id → teams.legal_entity_id → partner_legal_entities → ShopCode (tinkoff_shop_code). Без активного юр. лица с ShopCode — T‑Bank недоступен. См. admin-legal-entities §1.1, account-partner-organization.

3) Основные воркфлоу

3.1 Месячная оплата (type=monthly_fee)

  1. Создаём payables:
    • type=monthly_fee
    • month=YYYY-MM-01 (+ дублируем в meta.month и meta.team_id)
    • status=pending
  2. Создаём payment_intents:
    • payable_id = payables.id
    • provider_inv_id = InvId (уходит в Robokassa)
    • status=pending
  3. Провайдер вызывает ResultURL (webhook) → при успехе в транзакции:
    • payment_intents.status=paid
    • payables.status=paid, paid_at
    • пишем payments (снимок team_id/team_title для monthly_fee, §2.2)
    • обновляем users_prices для (user_id + team_id + month): is_paid=1 (§2.1)

3.1.1 Витрина «Оплата» ученика: POST /payment → TransactionController@index

Маршрут: имя payment, метод POST (открывается из кабинета, не через прямой GET).
Доступ: middleware can:paying.classes (плюс общие auth и 2fa для кабинета). Без права — 403.
Шаблон: resources/views/payment/paymentUser.blade.php.

Назначение: показать период, итоговую сумму и кнопки Робокассы / T‑Bank (карта и СБП) для месячного абонемента или ввода суммы для сценариев без привязки к месяцу из календаря (если так передаётся форма).

  • Поля формы с дашборда (генерируются в resources/views/dashboard.blade.php):
    • paymentDate — подпись для человека, например Март 2026;
    • formatedPaymentDate — канонический первый день месяца: YYYY-MM-01 (рекомендуется всегда передавать с витрины, чтобы не зависеть от парсинга русской строки);
    • outSum — может дублировать цену для старых сценариев; для месячного периода на сервере не используется как источник истины.
  • Период на сервере: если пришёл валидный formatedPaymentDate (YYYY-MM-DD), он имеет приоритет; иначе строка paymentDate разбирается через TransactionController::formatedDate (русское «Месяц Год» → YYYY-MM-01). Для разбора используется DateTime::createFromFormat('!F Y', …): префикс ! обязателен, иначе PHP подставляет «сегодняшний» день в неполную дату и короткие месяцы (например февраль) могут сместиться в следующий календарный месяц.
  • Группа и сумма для месячного абонемента: hidden team_id (с дашборда или после resolver); сумма — только из users_prices для тройки (user_id, team_id, new_month): сервис App\Services\Payments\UserPriceMonthlyFeePaymentResolver (метод resolveOrAbort, §2.1). Нормализация строки суммы до формата копеек — App\Support\Payments\PaymentOutSumNormalizer. Если начисления за выбранный месяц/группу нет, группа не указана при нескольких начислениях или сумма некорректна — ответы 403 / 422 с понятным текстом (не подмена суммы из POST).
  • Отображение «Итого» на странице: целые рубли с разбивкой тысяч (копейки только во внутренних полях форм для провайдера).
  • Строка «Группа» на витрине (paymentUser.blade.php): при месячном абонементе и известной группе выводится $monthlyTeamTitle; в формах Робокассы и T‑Bank уходит hidden team_id ($monthlyTeamId).
  • «Поставщик услуг» — блок в summary при доступном T‑Bank; см. §3.1.2.
  • Согласованность с Init: те же правила суммы и периода применяются при старте оплаты — TransactionController::pay (Робокасса), TinkoffPaymentController::create / createSbp (T‑Bank): при наличии месячного периода сумма снова читается из users_prices, а не из пользовательского outSum; в payables.meta и meta intent сохраняется team_id.
Итог: для monthly_fee подделать сумму через форму нельзя: и экран, и редирект в Робокассу/T‑Bank используют цену из БД.

3.1.2 Блок «Поставщик услуг» на витрине оплаты (2026‑07)

Зачем: родитель видит, какому юр. лицу школы будут перечислены средства за услуги (net-выплата T‑Bank) и чьи реквизиты попадут в онлайн-чек CloudKassir — до нажатия «Оплатить».
Не путать с формой банка: на странице T‑Bank при Init может отображаться мерчант глобального терминала платформы; блок «Поставщик услуг» показывает юр. лицо из справочника по цепочке группы — то же, что LegalEntityResolver при Init и выплате (tbank §5.7, admin-legal-entities §5.1).
Robokassa: блок не показывается (нет привязки к ShopCode / юр. лицу справочника на витрине).

Текст при отсутствии юр. лица

Как определяется юр. лицо (источник правды)

Сервис App\Services\Payments\PaymentCheckoutContextResolver собирает DTO App\Support\Payments\PaymentCheckoutContext (витрина ученика и клубный взнос). На публичной СБП-странице абонемента — UserLessonPackagePublicPayService::resolvePublicPayCheckoutDisplay() (тот же PayableTeamResolver + PaymentCheckoutLegalEntityPresenter, §3.4.1).

  1. Группа платежа — PayableTeamResolver (тот же алгоритм, что при Init):
    • monthly_fee — team_id из resolver месячного абонемента (UserPriceMonthlyFeePaymentResolver);
    • custom_payment_fee — user_custom_payment.team_id;
    • lesson_package_fee — user_lesson_packages.team_id / календарь / primary group (см. §2.3);
    • club_fee — одна группа или выбор на странице клубного взноса.
  2. Юр. лицо — LegalEntityResolver::forTeamId(team_id, partner_id) → активная запись partner_legal_entities с ShopCode.
  3. Доступность T‑Bank на экране — PaymentService::isTbankAvailable($partner, $team) (глобальный терминал + ShopCode).
На витрине не используется fallback forPartner без группы для multi-entity: если из payable не удалось получить группу или привязку — юр. лицо не показывается, T‑Bank блокируется (согласовано с Init, §2.3.2).

3.2 Разовая оплата (club_fee/uniform/camp)

  1. На странице /payment/club-fee ученик указывает сумму и (при нескольких группах) выбирает группу. Без членства в группах оплата недоступна.
  2. Создаём payables:
    • type=club_fee (или uniform / camp)
    • meta.team_id — группа платежа
    • status=pending
  3. Создаём payment_intents с payable_id и уходим в Robokassa.
  4. Webhook ResultURL (успех) → в транзакции:
    • payment_intents.status=paid
    • payables.status=paid
    • пишем payments
    • в users_prices ничего не пишем
Правило, чтобы ничего не затиралось: каждая разовая оплата = новая строка в payables (и новый payment_intent).

3.3 Дополнительный платеж (кастомные даты) (type=custom_payment_fee)

Источник истины для суммы: user_custom_payment.amount. Сумма из POST (outSum) не используется как источник истины.
Группа: user_custom_payment.team_id — обязательна при создании в админке (POST /admin/setting-prices/custom-payments). См. setting-prices-custom-payments.
Связка: в payables.meta хранится user_period_price_id и team_id.
Состояние оплаты: при успешном webhook выставляем user_custom_payment.is_paid=1.
  1. Админ создаёт запись в user_custom_payment с учеником, группой и суммой.
  2. С дашборда в POST /payment передаются payment_kind=custom_payment и custom_payment_id (= user_custom_payment.id).
  3. На витрине оплаты (TransactionController@index) запись user_custom_payment проверяется на принадлежность текущему партнёру и пользователю; если дополнительный платеж уже оплачен — возвращаем 422.
  4. При старте оплаты (Robokassa / T‑Bank) создаём:
    • payables.type=custom_payment_fee, meta.user_period_price_id, meta.team_id
    • payment_intents с payable_id
  5. При успешном webhook:
    • payment_intents.status=paid
    • payables.status=paid
    • user_custom_payment.is_paid=1 по meta.user_period_price_id
    • создаём запись в payments для отчётов (location_id в снимке — только если явно передан в PaymentLedgerRecorder, см. блок «payments (финансовый журнал)» выше)

3.4 Назначенный абонемент (type=lesson_package_fee)

Источник истины для суммы: поле user_lesson_packages.fee_amount (рубли). Сумма из POST (outSum) не используется как источник истины.
Группа: при ≥2 активных юр. лицах партнёра — обязательный team_id при назначении абонемента; иначе определяется из календаря или primary group.
Связка: в payables.meta хранится user_lesson_package_id и team_id.
Состояние оплаты: при успешном webhook выставляем user_lesson_packages.is_paid = true. В users_prices ничего не пишем.
Чеки CloudKassir: позиция с наименованием «Абонемент». В журнале payments для T‑Bank в поле payment_month для этого типа записывается строка «Абонемент».
  1. Админ при назначении абонемента задаёт fee_amount и (при multi legal entity) группу.
  2. С кабинета в POST /payment передаются payment_kind=lesson_package и user_lesson_package_id. На шаблоне оплаты те же поля уходят в формы Робокассы и T‑Bank (включая СБП).
  3. На витрине (TransactionController@index) запись проверяется через App\Services\Payments\UserLessonPackageFeePaymentResolver: пользователь и партнёр, не оплачено, сумма > 0. Если уже оплачено или сумма некорректна — 422 / 404.
  4. При старте оплаты создаются payables.type=lesson_package_fee, payment_intents с payable_id.
  5. При успешном webhook (Robokassa ResultURL, T‑Bank уведомление) в транзакции: payables и intent переводятся в оплаченные, создаётся payments (без автоматического location_id из карточки ученика), для строки user_lesson_packages выставляется is_paid=true.
  6. При успешном возврате (джобы RobokassaProcessRefundJob, TinkoffProcessRefundJob для соответствующего payable) для этой же строки user_lesson_packages сбрасывается is_paid=false.
  7. Публичная ссылка СБП (без кабинета): админ выдаёт URL через public-pay-link; родитель открывает /pay/ulp/{token} — см. §3.4.1.

3.2.1 Страница оплаты клубного взноса (/payment/club-fee)

Маршрут: GET/POST /payment/club-fee (имя clubFee) → TransactionController@clubFee
Доступ: middleware can:payment.clubfee (+ auth, 2fa). Без права — 403; гость — редирект/401.
Шаблон: resources/views/payment/clubFee.blade.php
  • Назначение: отдельная страница для оплаты клубного взноса с произвольной суммой.
  • Группа (PayableTeamResolver):
    • 0 групп у ученика — страница с блокировкой (clubFeeBlocked), текст «Оплата недоступна: вы не состоите ни в одной группе»;

    • 2+ группы — селект «Группа» (clubFeeRequiresTeamChoice); без team_id в POST Init → 422 «Укажите группу для оплаты.»
  • Ввод суммы: пользователь вручную вводит сумму в единственное поле paymentAmount.
  • Доступные способы оплаты (флаги в контроллере clubFee, через PaymentCheckoutContextResolver):

    • T‑Bank (карта) — $tbankAvailable / $canTbankCard; при 2+ группах на первом рендере tbankAvailable=false, видимость по JS
    • T‑Bank СБП — $tbankSbpAvailable / $canTbankSbp; дополнительно сумма ≥ 10 ₽ (проверка в JS и на Init)
    • «Поставщик услуг» — $showTbankLegalEntityBlock, $serviceProviderLabel; см. §3.1.2
  • Динамический T‑Bank и «Поставщик услуг» при выборе группы (2+ группы):
    • в view передаётся teamTbankCheckout — map team_id → { card, sbp, serviceProviderLabel } (PaymentCheckoutContextResolver::clubFeeTeamCheckoutMap);
    • legacy teamTbankCardAvailable — map team_id → bool (поле card);
    • блоки #clubFeeTbankCardBlock, #clubFeeTbankSbpBlock, #clubFeeServiceProviderRow изначально display:none (пока группа не выбрана);
    • при смене группы и вводе суммы — updateClubFeeTbankVisibility(), updateClubFeeServiceProvider() (inline JS, проверка синтаксиса — BladeInlineJsSyntaxTest).
  • Ошибка Init при группе без юр. лица в multi-entity: «Для выбранной группы не настроено юр. лицо» (tbankAvailabilityError).
  • Формирование данных для оплаты:
    • paymentDate = "Клубный взнос" (строка, не дата)
    • formatedPaymentDate не передаётся (только для месячных оплат)
    • outSum берётся из поля ввода при отправке формы
    • JavaScript-валидация validateAndSetAmount() проверяет, что сумма > 0
  • Обработка оплаты:
    • При отправке формы вызывается тот же роут, что и для месячных оплат (POST /payment/pay для Робокассы, POST /payments/tinkoff/create и POST /payments/tinkoff/sbp для T‑Bank)
    • Создаётся payables с type=club_fee и meta.team_id
    • В users_prices запись не создаётся (только для monthly_fee)

3.3 Оплата через T‑Bank (мультирасчёты) (одностадийная)

Важно про “месяц”: источником истины является payables.month (и payables.type). В T‑Bank дополнительно передаём месяц в DATA.month только для трассировки.
Важно для надёжности webhooks: в DATA также передаём payment_intent_id/payable_id/user_id, а для всех типов — team_id (группа платежа для юр. лица и выплаты), чтобы обработчик мог связать webhook с нашей БД даже при гонке (webhook пришёл раньше сохранения tbank_payment_id/tbank_order_id в payment_intents).
  1. На странице выбора оплаты (POST /payment, см. п. 3.1.1) метод T‑Bank показываем только если:
    • глобальный терминал T‑Bank активен: payment_systems(partner_id=NULL, name=tbank) с заполненными ключами и is_enabled=true
    • есть ShopCode у активного юр. лица (partner_legal_entities.tinkoff_shop_code через LegalEntityResolver по группе платежа)
    • у пользователя есть право payment.method.tbankCard и/или payment.method.tbankSBP
    Блок «Поставщик услуг» — при тех же условиях, что T‑Bank доступен как инструмент (§3.1.2).
  2. При старте оплаты создаём payables (pending) и payment_intents (pending, provider=tbank).
  3. Вызываем T‑Bank /v2/Init (PayType=O), сохраняем PaymentId/OrderId в payment_intents.
  4. Webhook T‑Bank /webhooks/tinkoff/payments:
    • при Status=CONFIRMED в транзакции: payment_intents.status=paid, payables.status=paid, создаём payments (снимок группы для всех типов с meta.team_id, §2.2)
    • если payables.type=monthly_fee → UserPrice::markMonthlyPaid для (user_id, payables.meta.team_id, payables.month) (§2.1)
    • при финальном неуспехе (CANCELED/REJECTED/...) → payment_intents.status=failed

4) Настройки провайдеров и комиссии

Хранение ключей: ключи платёжных систем хранятся в таблице payment_systems в поле settings в зашифрованном виде.

4.1 T‑Bank ключи (глобальный терминал платформы)

Полное описание страницы настроек, endpoint'ов, прав и миграции — в отдельном разделе «Настройки → Платёжные системы».
  • Где вводим: /admin/settings/paymentSystem (одна запись partner_id = NULL, name = tbank)
  • Что нужно заполнить (для «подключено»):
    • terminal_key, token_password — приём платежей (eacq)
    • e2c_terminal_key, e2c_token_password — выплаты партнёрам (e2c)
    • is_enabled — глобальный вкл/выкл T‑Bank
  • Per-partner ключи T‑Bank не хранятся. Доступность метода у клуба — ShopCode активного юр. лица (LegalEntityResolver) и права пользователя. См. admin-legal-entities, account-partner-organization.
  • Автовыплата и задержка после оплаты — в tinkoff_commission_rules (не в payment_systems). См. tbank §5.

4.2 Комиссии и автовыплата (удерживаются с партнёра)

Комиссии настраиваются на странице /admin/settings/tbank-commissions и хранятся в таблице tinkoff_commission_rules. Подробно про интерфейс (таблица DataTables, модалки, фильтры, права) — в T‑Bank: раздел 4.3.
  • Комиссия банка (эквайринг): acquiring_percent / acquiring_min_fixed (по умолчанию 2.49% и мин 3.49 ₽)
  • Комиссия банка (выплата партнёру): payout_percent / payout_min_fixed (по умолчанию 0.10%)
  • Комиссия платформы: platform_percent / platform_min_fixed
  • Автовыплата партнёру (после CONFIRMED): auto_payout_enabled, auto_payout_delay_hours в tinkoff_commission_rules (партнёр + метод card/sbp/tpay/NULL)
  • Округление: математическое (round) до копейки
  • Формула выплаты партнёру: 
    net = gross - bank_acquiring_fee - bank_payout_fee - platform_fee

4.3 T‑Bank: оплата по СБП (QR)

На страницах выбора оплаты (/payment, /payment/club-fee) кнопка «Оплатить по СБП» отправляет форму на маршрут payment.tinkoff.sbp (TinkoffPaymentController::createSbp): создаются payables и payment_intents, вызывается банковский Init с полем DATA (связка с intent/payable), затем редирект на страницу QR в CRM (GET /tinkoff/qr/{PaymentId}). Подробности, отображение QR и доступы — в разделе T‑Bank: СБП (QR) в интерфейсе.
  • Факт оплаты: webhook /webhooks/tinkoff/payments (как у карточной оплаты T‑Bank); страница QR дополнительно опрашивает GetState для подсказки пользователю и редиректа после CONFIRMED.
  • Публичная оплата абонемента: отдельная страница GET /pay/ulp/{token} (§3.4.1) — только СБП, без редиректа через payment.tinkoff.sbp; Init при первом открытии страницы.
  • Отдельный маршрут POST …/payments/tinkoff/qr-init (TinkoffQrController::init) выполняет только Init и запись в tinkoff_payments без создания payables/payment_intents; в текущих шаблонах витрины оплаты используется сценарий через payment.tinkoff.sbp, а не qr-init.

5) Идемпотентность webhooks

Провайдеры могут вызывать webhooks несколько раз. Поэтому обработчик должен быть идемпотентным:
  • payment_intents(provider, provider_inv_id) — уникальная пара
  • повторный webhook не должен создавать дублей в payments/users_prices

6) Где что смотреть

  • История покупок/оплат: payables
  • Почему не прошёл webhook: логи + payment_intents
  • Факт денег: payments
  • Начисление / оплата месяца по группе: users_prices (team_id обязателен)
  • Оплаченная группа в журнале: payments.team_id / team_title (§2.2)
  • Автопроверки: UserClassPaymentFlowTest.php, MultiTeamMonthlyPaymentFlowTest.php, MultiTeamMonthlyPaymentAccessFeatureTest.php, PayableTeamPaymentFlowTest.php, PayableTeamPaymentInitFeatureTest.php, CustomPaymentsTeamAccessFeatureTest.php, LessonPackageAssignmentsTeamAccessFeatureTest.php, PaymentAssignmentTeamBackfillTest.php — витрина, multi-team, доступ, webhook, отчёт, маршрутизация по группе.

7) Онлайн‑чеки в отчётах по платежам

В отчётах платежей используются данные из таблицы fiscal_receipts для отображения колонки “Чек”. Права на дополнительные колонки комиссий, «К выплате», «Статус возврата» и кнопку «История» T‑Bank в админ-отчёте описаны в документе «Отчёт Платежи».
  • Админ-отчёт: /admin/reports/payments
  • Пользовательский отчёт: /reports/payments
  • Логика для T‑Bank: ссылка показывается только если URL начинается с https://receipts.ru/; иначе чек считается несформированным.
  • Чек возврата (T‑Bank cancel → CloudKassir): в админ-отчёте рядом с income‑чеком может появляться вторая иконка “Чек возврата”, которая основана на FiscalReceipt.type=income_return и поле return_receipt_url. Иконка отображается только когда return‑чек реально сформирован (есть валидный URL).
  • Логика для Robokassa: иконка чека не показывается, т.к. фискализация и доступ к чекам находятся на стороне партнёра.
