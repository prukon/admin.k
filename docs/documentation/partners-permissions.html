Партнёры: базовые роли и права по умолчанию

Файл: /docs/documentation/partners-permissions.html
Цель: при создании нового партнёра (школы) автоматически назначать базовые permissions системным ролям user и admin в рамках этого партнёра.

1) Как устроены роли/права (важно)

  • Системные роли: roles.name = user/admin существуют один раз в системе (общие для всех партнёров).
  • Права: permissions — единый справочник имён permissions.
  • Назначение прав ролям делается в разрезе партнёра через pivot-таблицу permission_role:
    • permission_role.partner_id
    • permission_role.role_id
    • permission_role.permission_id
Это не “классический” Spatie-permission по умолчанию (где role_has_permissions глобальный). В kidscrm права роли могут отличаться между партнёрами, поэтому ключевой контекст — partner_id.

2) Источник истины: конфиг базовых прав

Списки базовых permissions для ролей user и admin живут в конфиге:
/config/role_base_permissions.php
  • Почему конфиг: один источник правды для модели (автопроставление), сидеров и тестов.
  • Важно: если включён config:cache, после деплоя нужно обновить кеш конфигов.

3) Автопроставление при создании партнёра

Реализация сделана “прямо в модели” партнёра:
/app/Models/Partner.php (хуки creating и created в booted()).

3.1 Fail-fast в creating (без “полупартнёра”)

  • Перед вставкой партнёра проверяем, что все permissions из конфига реально существуют в permissions.
  • Если чего-то нет — бросаем RuntimeException и создание партнёра прерывается (строка в partners не появится).

3.2 Назначение в created

  • Для нового partner_id берём:
    • role_id для ролей user и admin
    • permission_id по спискам имён из конфига
  • Вставляем строки в permission_role пачкой (bulk insert) и идемпотентно через insertOrIgnore.

4) Ограничение на дубли: UNIQUE по тройке

В permission_role должен существовать уникальный индекс на: (partner_id, role_id, permission_id).
  • Это гарантирует отсутствие дублей и корректную идемпотентность.
  • Добавлена миграция-страховка: /database/migrations/2026_02_19_000001_ensure_permission_role_unique_triplet.php (для MySQL/MariaDB, с дедупликацией при необходимости).

5) Сидеры базовых прав (dev/test сценарии)

Сидеры используют тот же конфиг и заполняют permission_role для всех партнёров в БД:
  • /database/seeders/UserRoleBasePermissionsSeeder.php
  • /database/seeders/AdminRoleBasePermissionsSeeder.php
  • Оба сидера “fail-fast”: если permissions из конфига отсутствуют — сидер падает с ошибкой.

6) Автотесты

Фичтесты на это поведение:
/tests/Feature/Crm/Permissions/PartnerBasePermissionsTest.php
  • Проверяют автопроставление для user/admin по конфигу
  • Проверяют изоляцию между партнёрами
  • Проверяют fail-fast без “полупартнёра”
  • Проверяют идемпотентность и наличие UNIQUE по тройке (MySQL/MariaDB)
php artisan test --env=testing --filter PartnerBasePermissionsTest

7) Кастомные роли в админке

Отдельный сценарий: администратор создаёт новую (несистемную) роль на вкладке «Права и роли». Стартовый набор прав для такой роли берётся из того же конфига, ключ roles.admin, но логика и ограничения отличаются от сидеров и от автопроставления при создании партнёра — см. Настройки: кастомные роли и стартовый набор прав.
