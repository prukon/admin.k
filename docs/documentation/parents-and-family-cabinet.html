Родители и семейный кабинет

Файл: /docs/documentation/parents-and-family-cabinet.html · URL: /docs/documentation/parents-and-family-cabinet · оглавление: /doc
Назначение. Справочник родителей (parents) для ФИО подписанта договора и семейного кабинета (несколько детей с одним users.parent_id). Вход в CRM — только под учёткой ученика (users, роль user). Отдельной учётки «мама/папа» нет. Контакты для SMS и оплат — users.phone / users.email учётки ученика, под которой выполнен вход.

0) Как это работает (сценарий)

Пример. Брат 1 и брат 2 — две записи users с одним parent_id (одна строка в parents). Вход под любым из них даёт доступ к данным обоих через переключатель «Просмотр данных ученика» в левом сайдбаре (layout admin2).
  • Права (роли, permissions) остаются у залогиненного ученика; переключатель меняет только контекст просмотра (платежи, консоль и т.д.) — session active_student_user_id.
  • Если у ученика нет parent_id или нет второго активного ребёнка с тем же parent_id — переключателя нет.
  • При семейном режиме (переключатель виден) в сайдбаре над селектом «Имя» и «Почта» показываются parents.full_name и parents.email (см. FamilyStudentContextService::sidebarPanelIdentity).
  • Редактирование ФИО родителя в личном кабинете меняет общую запись parents — изменения видны всем детям с этим parent_id.

1.2) Связь с учеником

Миграция: database/migrations/2026_05_28_120001_add_parent_id_to_users_table.php

  • users.parent_id → FK parents.id (nullable) — «родитель этого ученика».
  • Один родитель (parents) → много учеников (users с одним parent_id).
  • Переключение детей в кабинете: у залогиненного ученика тот же parent_id, что и у брата/сестры (тот же партнёр, is_enabled).
Legacy удалён. Колонки users.parent_lastname, users.parent_firstname, users.parent_middlename удалены миграцией 2026_05_28_130000_drop_legacy_parent_name_columns_from_users_table.php; колонка parents.user_id (родительский логин) — миграцией 2026_05_28_140000_drop_user_id_from_parents_table.php. В HTTP-формах по-прежнему используются имена полей parent_lastname / parent_firstname / parent_middlename — это ключи запроса, значения пишутся в parents.

3) Админка: ученики (/admin/users)

Список и создание на этой странице — только роль user (ученик). Обзор вкладок раздела: admin-users-section.

Подробно: admin-users §2.1. Кратко:

МетодURLНазначение
GET/admin/users/parents/searchSelect2: поиск родителей текущего партнёра (q, id, limit)
POST / PATCH/admin/users, /admin/users/{user}parent_id и/или parent_* в теле → StudentParentSyncService::syncForStudent
  • В модалках create/edit: блок родителя только для роли user; переключатель «Из справочника» / «Новый родитель» (если в справочнике есть записи), иначе сразу поля ФИО — см. admin-users §2.1.2.
  • В списке DataTables колонка «Родитель» — склейка ФИО из parents (через parentProfile).
  • Поиск по name ищет также по ФИО в parents (whereHas parentProfile).
  • Сортировка col=3: parents.lastname, parents.firstname (left join).
  • Право: достаточно users.view + store/update пользователя; отдельного permission на родителя в админке нет.
  • Пол и комментарий ученика — §admin-users §2.4: пол — users.sex (CRM и кабинет), комментарий — только CRM (users.comment, §4.4).

4) Личный кабинет: учётная запись

Страница: GET /account-settings/user/edit (account.user.edit), middleware can:account.user.view.
Блок «Данные родителя» на вкладке «Личные данные» (resources/views/account/users.blade.php).

4.1) Права

PermissionGateПоведение
account.user.viewaccount.user.viewДоступ к странице и маршрутам группы account.user.*
account.user.parent.updateaccount.user.parent.updateРедактирование ФИО родителя; без права — поля disabled (readonly)
users.sexusers.sexПол ученика: в CRM (колонка, модалки) и в личном кабинете (/account-settings/user/edit); без права — поле не рендерится. Комментарий в кабинете недоступен.

Базовая роль user получает account.user.parent.update и users.sex из config/role_base_permissions.php (сидер PermissionSeeder).

4.2) Сохранение

  • PATCH /account-settings/user — AccountUpdateRequest + AccountController@update.
  • Поля родителя валидируются только при account.user.parent.update.
  • Пол (sex) — только при users.sex (§4.4).
  • Сервис: StudentParentSyncService::updateFromAccount — обновляет parentProfile ученика; при отсутствии создаёт запись в parents и выставляет users.parent_id.
  • Ошибки валидации — JSON 422, ключи parent_lastname, sex и т.д. (вывод под полями на фронте).

4.4) Пол ученика в кабинете

Одно право users.sex — и для CRM, и для кабинета (legacy account.user.sex.update удалено, см. миграцию 2026_06_20_130000_merge_account_user_sex_into_users_sex.php и §admin-users §2.4).
Поле «Пол» на resources/views/account/users.blade.php — <select name="sex"> с вариантами из App\Enums\UserSex.
Валидация: AccountUpdateRequest (nullable|in:male,female) при users.sex.
Комментарий (users.comment) в личном кабинете не отображается и не редактируется.
  • Изменения пола пишутся в my_logs (event = user.account_updated).
  • Тесты: UserCommentAndSexFeatureTest — раздел «Личный кабинет: пол через users.sex».

5) Семейный кабинет (переключение детей)

5.1) UI и условия показа

  • Partial: resources/views/includes/family_student_switcher.blade.php, подключается в layouts/admin2.blade.php (блок user-panel, под строкой «Роль»).
  • Переключатель виден, если FamilyStudentContextService::accessibleStudents() содержит больше одного ученика.
  • В селект попадают: активные (is_enabled) ученики с тем же parent_id и partner_id; залогиненный ученик всегда в списке, даже если отключён.
  • Отключённый брат/сестра в селект не попадает; прямой POST switch на него → 403.

6) Договоры

На карточке договора, в модалке «Создать договор» (read-only) и в предзаполнении подписанта ФИО берётся из User::parentFormFields() / parent_full_name → таблица parents. См. contracts §5 и §2–3.

  • ContractsController@show — User::with('parentProfile'), без legacy-колонок на users.
  • ContractPrefillResolver — плейсхолдеры PARENT_* из профиля родителя.

9) Что не поддерживается

  • Отдельная учётная запись родителя для входа (parents.user_id удалён).
  • Переключение между детьми при разных parent_id (разные записи в parents).
  • Редактирование parents.phone / parents.email в UI личного кабинета (поля есть в БД; в форме — только ФИО).
  • Семейный контекст на страницах договоров у роли user (договоры — право contracts.view, админка).
