Нагрузка тренеров

Файл: /docs/documentation/schedule-trainer-workload.html · страницы: GET /schedule/trainer-workload (schedule.trainer-workload), GET /schedule/trainer-workload/data (schedule.trainer-workload.data)
Не путать с «Расписание школы» (слоты абонементов, team_schedule_slots, lessonPackages.view).
Отчёт строится только из журнала посещаемости (schedule_users + статус «Посетил» + trainer_profile_id).
Вкладка «Нагрузка тренеров» — вторая вкладка на экране раздела «Журнал расписания» (общая оболочка resources/views/admin/schedule/index.blade.php, по аналогии с вкладками в отчётах админки).
Первую вкладку «Журнал расписания» см. в отдельном разделе.

1) Право доступа schedule.view

То же permission, что и для журнала /schedule. Отдельного права на отчёт нет.

СитуацияРезультат
ГостьGET /schedule/trainer-workload → редирект на логин; GET …/data → 401
Авторизован, нет schedule.viewСтраница и data → 403
Есть schedule.viewСтраница и data при валидных параметрах → 200 (в т.ч. для каждого из трёх пресетов месяца и режимов show_groups=0|1)

Право trainers.view для отчёта не требуется: список тренеров берётся из активных trainer_profiles партнёра (как в модалке «Посетил» журнала). Smoke по всем сценариям с правом — ScheduleTrainerWorkloadAccessFeatureTest::test_all_trainer_workload_endpoints_return_ok_with_schedule_view.

2) Эндпоинты

МетодURLИмяНазначение
GET /schedule/trainer-workload schedule.trainer-workload HTML: фильтры, матрица тренер × дни недели, первая отрисовка на сервере
GET /schedule/trainer-workload/data schedule.trainer-workload.data JSON + фрагмент HTML таблицы (AJAX при смене периода / переключателя)

Оба маршрута в routes/web.php в группе middleware('can:schedule.view') (внутри auth + 2fa).

2.1) Параметры запроса

ПараметрОписание
date_from, date_toОпционально, формат Y-m-d. Если оба пустые — период последние 30 календарных дней (сегодня − 29 … сегодня). Если указана только одна дата — вторая достраивается ещё на 29 дней в ту же сторону.
show_groupsОпционально, boolean. По умолчанию выключено (false): в ячейках только суммы. show_groups=1 — разбивка по группам (название + число дат).

Валидация: TrainerWorkloadReportRequest — date_to >= date_from, максимальная длина периода 366 дней (иначе 422, ошибки под полями дат в UI).

2.2) Ответ GET …/data

  • date_from, date_to — фактический период после нормализации
  • show_groups — bool
  • table_html — HTML фрагмент таблицы (_trainer_workload_table.blade.php)
  • report — массив агрегата (те же данные, что для первой отрисовки на сервере; UI обновляет в основном table_html)

Изоляция: агрегат только по users.partner_id = current_partner и trainer_profiles.partner_id = current_partner.

3) Источник данных и формула ячейки

3.1) Какие записи журнала попадают в отчёт

  • Таблица schedule_users, join users (is_enabled = 1, партнёр текущий).
  • status_id = системный «Посетил» (Status::globalVisitedId()).
  • trainer_profile_id IS NOT NULL.
  • date в выбранном периоде [date_from, date_to].

Записи «Посетил» без тренера, другие статусы и визиты чужого партнёра не учитываются.

3.2) Группировка

SQL-агрегат (сервис TrainerWorkloadReportService):

  • ось Y — все активные тренеры партнёра (trainer_profiles.is_enabled, user включён), сортировка sort_order, id (даже без визитов за период — пустая строка);
  • ось X — дни недели ISO 1–7 (Пн … Вс), из WEEKDAY(date)+1;
  • в ячейке — уникальные группы ученика (users.team_id → teams.title, без группы → «Без группы»);
  • число в ячейке — COUNT(DISTINCT DATE(schedule_users.date)) для пары (тренер, день недели, группа) за период (не число учеников и не число строк schedule_users).

3.3) Режим «Показывать группы»

ПереключательОтображение в ячейке
ВключёнСписок: Название группы + счётчик (чипы)
Выключен (по умолчанию)Одно число — сумма dates_count по всем группам в этой ячейке

3.4) Строка и столбец «Итого»

  • Столбец «Итого» (справа у каждого тренера) — сумма по всем дням недели за период (по группам или одной суммой — как в режиме переключателя).
  • Строка «Итого» (внизу) — сумма по всем тренерам для каждого дня недели.
  • Правый нижний угол — общий итог за матрицу.

То же значение столбца «Итого» (режим без групп) используется как trainings_count на вкладке «ЗП тренеров» за календарный месяц (TrainerWorkloadReportService::trainerRowTrainingsTotals()).

4) UI и AJAX

  • Фильтры: «С», «По» (тип date), переключатель «Показывать группы». Кнопок «Показать» / «Сбросить» нет — обновление на лету.
  • Под полями дат — быстрый выбор месяца (см. §4.1).
  • Смена дат — debounce ~300 мс, затем GET …/data.
  • Смена переключателя — запрос сразу.
  • URL в адресной строке синхронизируется через history.replaceState (период + show_groups=0|1).
  • Во время загрузки таблица слегка затемняется (класс is-loading на контейнере).
  • Дублирующий бейдж периода справа в шапке убран — период виден в полях «С» / «По».

4.1) Быстрый выбор месяца

Под полями «С» / «По» выводятся три текстовые ссылки с названиями календарных месяцев (именительный падеж, как в фильтре месяца журнала: «Март», «Апрель», «Май»). Состав ссылок всегда пересчитывается от текущей даты сервера при открытии страницы:

Порядок (слева → направо)МесяцЧто подставляется в «С» / «По»
1Позапрошлый календарный месяц1-е … последний день этого месяца
2Прошлый календарный месяцто же
3Текущий календарный месяцто же (date_to — последний день месяца, не «сегодня»)

Пример при дате сервера 15 мая 2026: ссылки «Март», «Апрель», «Май» → периоды 2026-03-01…2026-03-31, 2026-04-01…2026-04-30, 2026-05-01…2026-05-31. В январе — «Ноябрь», «Декабрь», «Январь» (с учётом смены года).

  • Клик по ссылке — в поля «С» и «По» записываются границы выбранного месяца, сразу выполняется тот же AJAX GET …/data, что при ручной смене дат (без отдельной кнопки).
  • Подсветка активной ссылки — класс is-active и aria-current="true", если в полях ровно полный календарный месяц (совпадение с одним из трёх пресетов). При периоде по умолчанию (последние 30 дней) или произвольном диапазоне активной ссылки нет.
  • Ручная правка дат в полях снимает или переносит подсветку (логика в trainer-workload.js, функция syncActiveMonthLinks).
  • Отдельных query-параметров и эндпоинтов для пресетов нет — только стандартные date_from / date_to. Любой полный месяц укладывается в лимит 366 дней.

Разметка: блок #trainer-workload-month-presets, ссылки .trainer-workload-month-link с атрибутами data-date-from, data-date-to (значения считаются в trainer_workload.blade.php через Carbon). Стили: .trainer-workload-month-presets* в schedule.css.

Шаблоны: resources/views/admin/schedule/trainer_workload.blade.php, _trainer_workload_table.blade.php, _trainer_workload_cell.blade.php.

Скрипт: resources/js/trainer-workload.js (подключается только на вкладке нагрузки в schedule/index.blade.php).

Стили: общий файл resources/css/schedule.css (блок .trainer-workload-*).

5) Связь с журналом и тренерами

  • Тренер в ячейке журнала задаётся при сохранении «Посетил» — см. Журнал: тренер при «Посетил».
  • Справочник тренеров и привязка к группам — Тренеры (админка).
  • Массовое «Личное расписание» (update-schedule-range) не заполняет trainer_profile_id — такие дни в отчёт не попадут, пока тренер не указан вручную в модалке ячейки.
