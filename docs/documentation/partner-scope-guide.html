Как правильно работать с партнёром в CRM

Файл: /docs/documentation/partner-scope-guide.html · руководство для рефакторинга страниц
Для кого. Эта страница — практическая инструкция: как получать «текущего партнёра» для обычных пользователей и как делать экраны, где superadmin видит данные всех школ (с фильтром «Все партнёры»). Эталон реализации: вкладка Настройки → Логи. Теория middleware и блокировок: partner-context.html.

1) Две задачи — не путать

ЗадачаКтоЧто в SQL
A. Обычная CRM-страница Админ школы, тренер с правами, любой не-superadmin Всегда WHERE partner_id = current_partner
B. Cross-partner страница Только superadmin Все партнёры, или один выбранный в фильтре
Permission viewing.all.logs и похожие названия не означают «все партнёры». Cross-partner — только явный режим SUPERADMIN_ALL_OR_FILTER + проверка роли на сервере.

2) Цепочка на каждый HTTP-запрос

Браузер
  → middleware web (auth, 2fa, …)
  → middleware SetPartner
       · не-superadmin: session('current_partner') = user.partner_id  (нельзя подменить)
       · superadmin: session('current_partner') = выбранная в UI школа (может быть пусто → редирект)
       · app()->instance('current_partner', Partner)
  → Controller extends AdminBaseController
       · методы делегируют в PartnerContext
  → SQL с partner-scope

3) Слои кода (куда смотреть)

4) Задача A — обычная страница (99% экранов)

4.1 Наследование контроллера

Админ-контроллер всегда extends AdminBaseController и принимает PartnerContext в конструкторе (через parent::__construct($partnerContext)). 
class TeamController extends AdminBaseController
{
    public function __construct(TeamService $service, PartnerContext $partnerContext)
    {
        parent::__construct($partnerContext);
        // …
    }
}

4.2 Какие методы вызывать

НужноМетодНе использовать в новом коде
ID текущего партнёра $this->requirePartnerId() или $this->partnerId() app('current_partner')->id напрямую
Модель Partner $this->requirePartner() Partner::find(session(...))
Ограничить Eloquent-запрос $this->scopeByPartner($query, 'teams.partner_id') ручной ->where('partner_id', …) с ID из request
Проверка superadmin для UI $this->isSuperAdmin() только Gate без partner-scope

4.3 Пример списка / DataTables

public function data(FilterRequest $request)
{
  $query = Team::query()->with([...]);

  // Режим по умолчанию STRICT_CURRENT — только current_partner
  $this->scopeByPartner($query, 'teams.partner_id');

  // дальше фильтры, сортировка, DataTables::of($query)…
}

4.4 Создание / изменение записи

  • При create всегда проставляй partner_id из $this->requirePartnerId(), не из формы.
  • При update / delete сначала найди модель через запрос с scopeByPartner или проверь (int) $model->partner_id === $this->requirePartnerId() — иначе 403/404.
  • В Form Request для валидации «принадлежит партнёру» можно брать ID из app('current_partner')?->id (как в существующих Request), но scope в контроллере обязателен.

4.5 Режим по умолчанию

scopeByPartner() = applyPartnerScope(..., PartnerScopeMode::STRICT_CURRENT). Если partnerId() в контексте есть — в запрос добавится where partner_id = …. Для не-superadmin partnerId() всегда совпадает с его школой.

5) Задача B — страница «все партнёры» (только superadmin)

5.1 Когда нужен отдельный режим

  • Superadmin должен видеть строки всех школ в одной таблице.
  • В фильтрах есть «Все партнёры» и выбор конкретной школы.
  • Обычный админ этой страницы либо не видит, либо видит только свою школу (как на Логах — через permission, но scope жёсткий).

5.2 Сервер: явно передать режим

На конкретном action (обычно JSON для DataTables), не глобально:

use App\Support\PartnerScopeMode;

public function data(Request $request)
{
    $query = MyModel::query();

    $this->applyPartnerScope(
        $query,
        'my_table.partner_id',
        PartnerScopeMode::SUPERADMIN_ALL_OR_FILTER,
        $request->input('filter_partner_id'),
    );

    return DataTables::of($query)->…;
}

Логика applyPartnerScope при SUPERADMIN_ALL_OR_FILTER:

  • superadmin + filter_partner_id пусто / all → без where partner_id;
  • superadmin + числовой filter_partner_id → один партнёр;
  • не superadmin → всегда как задача A (current_partner), даже если в AJAX прислали all.

5.3 Если контроллер не наследует AdminBaseController

В трейте/сервисе:

app(PartnerContext::class)->applyPartnerScope(
    $query,
    'my_logs.partner_id',
    PartnerScopeMode::SUPERADMIN_ALL_OR_FILTER,
    request()->input('filter_partner_id'),
);

Пример: трейт BuildsLogTable на вкладке логов.

5.4 UI (Blade + JS)

  1. В контроллере страницы: 'isSuperadmin' => $this->isSuperAdmin(), список партнёров для select (Partner::orderBy('title')->get()) — только если superadmin.
  2. Фильтр: <select name="filter_partner_id"> с опцией value="all" «Все партнёры».
  3. DataTables AJAX: всегда передавать filter_partner_id (для superadmin по умолчанию all).
  4. Колонка «Партнёр» / partner_title в таблице — только в разметке для superadmin.
  5. Для сортировки по партнёру: leftJoin + orderColumn('partner_title', …) (см. реализацию логов).

5.5 Эталон в проекте

  • Контроллер: SettingController::showLogs, logsData
  • View: resources/views/admin/setting/logs.blade.php
  • Документация раздела: settings-logs.html

Похожий паттерн без enum (ручной if ($isSuperadmin)): выплаты T‑Bank tbank-admin-payouts — при рефакторинге лучше свести к applyPartnerScope.

6) Чеклист рефакторинга страницы

6.1 Любая partner-scoped страница

  1. Контроллер extends AdminBaseController.
  2. Убрать прямой app('current_partner')->id → requirePartnerId() / scopeByPartner.
  3. Убрать partner_id из входящих request как источник истины.
  4. На write-операциях проверять принадлежность записи текущему партнёру.
  5. Feature-тест: пользователь партнёра A не видит данные партнёра B.

6.2 Дополнительно для cross-partner (superadmin)

  1. Только нужный action использует SUPERADMIN_ALL_OR_FILTER.
  2. Параметр фильтра: filter_partner_id, значение all = все партнёры.
  3. Тест: superadmin + all видит две школы; админ + all — только свою.
  4. Тест: superadmin + конкретный id — только эта школа.
  5. Тест: гость / без permission → 403.

7) Антипаттерны

НельзяПочемуВместо этого
where partner_id = $request->partner_id подмена ID в запросе scopeByPartner / applyPartnerScope
Убрать where partner_id «для superadmin» без режима утечка на всех action контроллера только один метод с SUPERADMIN_ALL_OR_FILTER
Доверять filter_partner_id=all без проверки роли админ увидит чужие данные логика в PartnerContext::applyPartnerScope
Глобально отключить partner filter для superadmin сломает изоляцию на других страницах режим per-endpoint
В тестах подменять app('current_partner') вручную маскирует SetPartner withSession(['current_partner' => $id])

8) Шпаргалка (копировать в голову)

// Обычная страница
$query = Model::query();
$this->scopeByPartner($query, 'models.partner_id');

// Cross-partner (только этот endpoint!)
$this->applyPartnerScope(
    $query,
    'models.partner_id',
    PartnerScopeMode::SUPERADMIN_ALL_OR_FILTER,
    $request->input('filter_partner_id'), // "all" | "123"
);

// Текущий партнёр
$pid = $this->requirePartnerId();
$partner = $this->requirePartner();

// Superadmin?
