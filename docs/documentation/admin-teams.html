Группы (админка): /admin/teams

Файл: /docs/documentation/admin-teams.html
Раздел «Группы» в боковом меню ведёт на GET /admin/teams. Доступ к странице и ко всем CRUD/JSON эндпоинтам групп задаётся одним правом: groups.view (группа маршрутов в routes/web.php с middleware can:groups.view).

1) Право schedule.view на странице групп

Одно и то же permission schedule.view (в справочнике permissions исторически описано как страница «Журнал расписания») дополнительно используется для элементов «Расписание» на /admin/teams: оно не открывает саму страницу групп и не заменяет groups.view.

Элемент UI При отсутствии schedule.view При наличии schedule.view
Таблица групп (DataTables) Колонка «Расписание» и пункт «Расписание» в выпадающем списке «Поля списка» не выводятся (Blade @can('schedule.view'), колонка на клиенте отключена через флаг JS). Колонка с подписью дней недели (weekdays_label) и настройка видимости колонки доступны.
Модалка «Создание группы» Блок выбора дней недели («Расписание») скрыт. Блок с чекбоксами дней недели показывается.
Модалка «Редактирование группы» Блок «Расписание» скрыт; скрипт не заполняет чекбоксы дней. Чекбоксы дней недели показываются и заполняются из GET /admin/team/{id}/edit.
Не путать с вкладкой «Расписание школы» в разделе абонементов: её экран и календарь завязаны на lessonPackages.view и связанные права — см. Расписание школы: календарь. Там другой контур доступа.

2) Где ещё используется schedule.view в проекте

  • Gate schedule.view в app/Providers/AuthServiceProvider.php (проверка hasPermission('schedule.view') для текущего партнёра).
  • Пункт меню «Журнал расписания» (/schedule) в resources/views/includes/sidebar.blade.php — директива @can('schedule.view').
  • Группы маршрутов журнала и статусов расписания в routes/web.php с middleware('can:schedule.view') (например /schedule, /schedule/statuses и др.).

3) Серверное поведение (группы)

  • Список для таблицы GET /admin/teams/data: в JSON по-прежнему приходит поле weekdays_label для каждой строки — скрытие колонки только клиентское; это сознательно не фильтруется на сервере для пользователей без schedule.view.
  • Обновление группы PATCH /admin/team/{id}: если у пользователя нет schedule.view, поле weekdays из запроса отбрасывается до вызова сервиса — назначение дней недели изменить нельзя (в т.ч. при попытке передать дни вручную). Запись в логах изменения «Дни недели» для такого сохранения не формируется.
  • Сервис App\Services\TeamService::update: синхронизация связи team_weekday выполняется только если ключ weekdays был передан в данных обновления (после валидации). Если ключа нет — связь дней недели не трогают.
  • Сортировка DataTables на GET /admin/teams/data: порядок сортировки определяется по имени колонки из массива запроса columns[*].name, а не только по индексу колонки — чтобы корректно работать при скрытой колонке «Расписание».

4) Базовые роли и конфиг

В config/role_base_permissions.php для роли admin строка schedule.view по умолчанию закомментирована: у нового партнёра у администратора это право может отсутствовать, пока его явно не включат в матрице прав партнёра. Общая модель прав — в Партнёры: базовые роли и права.

5) Представления и контроллер

  • resources/views/admin/team.blade.php — таблица, настройки колонок, инициализация DataTables.
  • resources/views/includes/modal/createTeam.blade.php, editTeam.blade.php — модалки создания/редактирования.
  • app/Http/Controllers/Admin/TeamController.php — методы index, data, store, edit, update, логи.

6) Автотесты

Фичтесты: tests/Feature/Crm/Teams/TeamScheduleViewAndPageAccessFeatureTest.php — видимость колонки/блоков, недоступность изменения дней без права, доступность эндпоинтов при groups.view, сортировка по имени колонки. Общие сценарии групп: tests/Feature/Crm/Teams/TeamControllerTest.php.
