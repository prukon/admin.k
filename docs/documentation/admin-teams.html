Группы (админка): /admin/teams

Файл: /docs/documentation/admin-teams.html · страница CRM: GET /admin/teams
Страница входит в раздел CRM «Справочники» — вкладка «Группы» (первая в порядке вкладок; /admin/teams, admin.team.index, $activeTab = 'groups'). Обзор: directories-hierarchy.
Сайдбар: при единственном праве groups.view пункт меню называется «Группы»; при двух и более правах справочников — «Справочники» (App\Support\DirectoriesMenu). Отдельного второго пункта «Группы» в сайдбаре нет.
Порядок вкладок на всех страницах справочников: Группы → Объекты → Районы → Виды спорта → Юр. лица (partial admin/directories/_section_tabs.blade.php).
Доступ к странице и ко всем эндпоинтам раздела — одно право groups.view (middleware can:groups.view в routes/web.php). Без него — 403; для гостя — редирект/401.
Дополнительные права (schedule.view, trainers.view, locations.view, sport_types.view, legal_entities.view) не открывают страницу групп и не заменяют groups.view — они только включают отдельные элементы UI на этой странице.
Поле «Тренировочная база» (teams.training_base) и право groups.training_base.view удалены миграцией 2026_06_13_150000_drop_training_base_from_teams.php.

1.1) Удаление группы

DELETE /admin/team/{team} (TeamController@destroy):
  • soft delete — teams.deleted_at;
  • все ученики отвязываются от группы в pivot: TeamUserSyncService::detachTeamFromAllStudents;
  • legacy users.team_id не изменяется (архивный снимок);
  • аудит team.deleted в my_logs.
Связь M:N и влияние на «Цены» / users_prices — student-team-membership §1.2.

2) Таблица групп и «Поля списка»

Таблица — DataTables (serverSide), фильтры: название группы, статус (активные/неактивные); при locations.view — также объект, район объекта и администратор объекта; при trainers.view — тренер; при sport_types.view — вид спорта; при ≥2 активных юр. лиц и legal_entities.view — колонка «Юр. лицо». Колонка «#» (нумерация) не настраивается.

Порядок колонок (при всех правах)

  1. # (rownum) — всегда видима
  2. Сортировка (order_by)
  3. Название (title) — ссылка (edit-team-link) на модалку редактирования; AJAX GET /admin/team/{id}/edit (JSON). Паттерн — reusable-ui-partials §3.2
  4. Тренер (trainer_label) — только при trainers.view
  5. Объект (locations_label) — только при locations.view и непустом справочнике объектов
  6. Вид спорта (sport_type_label) — только при sport_types.view и непустом справочнике видов спорта
  7. Юр. лицо (legal_entity_label) — только при legal_entities.view и ≥2 активных юр. лиц партнёра; см. §5.2
  8. Расписание (weekdays_label) — только при schedule.view
  9. Стоимость (month_price) — всегда (см. §2.1)
  10. Статус (status_label)
  11. Действия (actions)

Выпадающий список «Поля списка» (иконка колонок) сохраняет видимость в user_table_settings.columns для table_key = teams_index. Ключи: order_by, title, trainer_label, locations_label, sport_type_label, weekdays_label, month_price, status_label, actions (опциональные ключи появляются в UI только при соответствующем праве или наличии справочника).

2.1) Поле «Стоимость» (teams.month_price)

Необязательная месячная стоимость группы в целых рублях (без копеек). Доступна всем пользователям с правом groups.view; отдельного permission не требуется.

Где Поведение
БД Колонка teams.month_price — unsignedInteger, nullable. Пустое значение в форме → NULL. Допустимо 0 (бесплатная группа).
Таблица Колонка «Стоимость»: тип пресета money (KidsCrmDataTable, ключ month_price). Число — 1 046 руб (toLocaleString('ru-RU') + « руб»), выравнивание вправо через .dt-col-money-value. Пусто → «—» слева (как в колонках text). Сортировка по columns[*].name = month_price.
Модалки create / edit Поле «Стоимость в месяц» (month_price): type="number", step="1", min="0". Необязательное.
JSON GET /admin/teams/data В каждой строке ключ month_price — целое число или null (без форматирования).
JSON GET /admin/team/{id}/edit Ключ month_price — целое или null.
«Поля списка» Ключ month_price — видимость колонки «Стоимость».
Логи групп (my_logs) При создании/изменении группы — аудит team.created / team.updated (в т.ч. строка «Стоимость в месяц: …» в description).
Оплаты / счета Не используется в расчётах, счетах и интеграции с T‑Bank — только справочное поле в админке групп.

Валидация: Team\StoreRequest / UpdateRequest — month_price: nullable|integer|min:0; пустая строка в prepareForValidation → null. Ошибки возвращаются на фронт под полем.

Не путать с устаревшей таблицей team_prices (исторические цены по месяцам) — поле month_price хранится непосредственно в teams и не связано с team_prices.

3) Право schedule.view

Permission управляет страницей «Журнал расписания» (/schedule) и дополнительно — элементами «Расписание» на /admin/teams.

Элемент UI Без schedule.view С schedule.view
Таблица Нет колонки «Расписание» и пункта в «Поля списка» (@can('schedule.view') в Blade, флаг canViewSchedule в JS). Колонка weekdays_label (дни недели через запятую) и настройка видимости.
Модалки создания / редактирования Блок чекбоксов дней недели скрыт; weekdays в PATCH отбрасывается в TeamController. Чекбоксы дней; данные из GET /admin/team/{id}/edit.
JSON GET /admin/teams/data Поле weekdays_label всё равно приходит в каждой строке — скрытие только на клиенте. То же.
Не путать с «Расписание школы» (абонементы, календарь) — там lessonPackages.view и отдельные маршруты. См. Расписание школы: календарь.
Полное описание журнала /schedule (ячейки, статусы, тренер при «Посетил»): Журнал расписания.

4) Право trainers.view — колонка «Тренер»

Permission раздела Тренеры (админка) (/admin/trainers). На странице групп управляет колонкой и селектом тренера в модалках.

Элемент UI Без trainers.view С trainers.view
Таблица Нет колонки «Тренер» и пункта в «Поля списка» (@can('trainers.view'), canViewTrainers в JS). Колонка trainer_label — ФИО тренера (User::full_name первого профиля из team_trainer). Пусто → «—».
Модалки создания / редактирования Селект trainer_profile_id скрыт; ключ удаляется в TeamController до сохранения. Селект активных тренеров партнёра (trainerOptions в index).
JSON GET /admin/teams/data Поле trainer_label всё равно приходит — скрытие только на клиенте (как weekdays_label). То же.
Сортировка таблицы — По columns[*].name = trainer_label: join team_trainer → trainer_profiles → users, порядок по lastname, name.

5) Право locations.view — объект группы

Permission справочника объектов. На странице групп управляет полем teams.location_id (один объект на группу) и фильтром таблицы. Общая логика: Объекты ↔ группы.

Элемент UI Без locations.view С locations.view
Модалки create/edit Селект location_id скрыт; ключ удаляется в TeamController до сохранения. Select «Объект» — активные объекты партнёра; пустое значение — группа без объекта.
Фильтр таблицы Селекты «Объект», «Район», «Администратор» скрыты (#filter-location, #filter-district, #filter-admin). Объект (#filter-location): пусто — все; none — location_id IS NULL; id — группы объекта.
Район (#filter-district, если есть активные районы): пусто — все; none — группа без объекта или объект без района; id — группы, чей объект в этом районе (teams → location → district_id).
Администратор (#filter-admin, если есть admin-пользователи): пусто — все; none — группа без объекта или объект без администраторов в pivot location_admin_user; id — группы, чей объект привязан к этому администратору (M2M).
Колонка «Объект» Нет в таблице и в «Поля списка». locations_label — название объекта или «—»; ключ в columns-settings.
GET /admin/team/{id}/edit Без ключа location_id. Числовой location_id или null.

Валидация: Team\StoreRequest / UpdateRequest — location_id nullable, exists у locations текущего партнёра (rules только при locations.view); ошибки на русском под полем.

5.1) Право sport_types.view — вид спорта группы

Permission справочника видов спорта (/admin/sport-types). На странице групп управляет необязательной привязкой группы к виду спорта и фильтром таблицы.

Элемент UI Без sport_types.view С sport_types.view
Модалки create/edit Селект sport_type_id скрыт; ключ удаляется в TeamController до сохранения. Селект активных видов спорта партнёра (sportTypeOptions в index); пустое значение допустимо.
Фильтр таблицы Селект «Вид спорта» скрыт (#filter-sport-type). Пусто — все группы; none — только без sport_type_id; id — группы с этим видом спорта.
Колонка «Вид спорта» Нет в таблице и в «Поля списка». sport_type_label в JSON data; ключ sport_type_label в columns-settings (если справочник не пуст).
GET /admin/team/{id}/edit Без ключа sport_type_id. Числовой sport_type_id или null.

Валидация: Team\StoreRequest / UpdateRequest — sport_type_id через PartnerSportTypeId (только при sport_types.view в rules): id должен принадлежать партнёру и быть активным (is_enabled). Дополнительная проверка — TeamService::resolveSportTypeIdForPartner(). Права sport_types.view / sport_types.manage не входят в базовый набор для новых партнёров (config/role_base_permissions.php).

5.2) Право legal_entities.view — юр. лицо группы

Справочник и sm-register: admin-legal-entities. Резолвер для оплат: LegalEntityResolver. При ≥2 активных юр. лицах (PartnerLegalEntityMode::isMultiEntity) привязка группы обязательна для оплат T‑Bank — payments §2.3.5.
Элемент UIУсловие показаПоведение
Селект legal_entity_id в модалках create/edit legal_entities.view и ≥2 активных юр. лиц Обязательный (required, метка *); опция по умолчанию — «Выберите юр. лицо» (без «По умолчанию»). Подписи — displayTitle(). Подсказка: от привязки зависят оплаты T‑Bank и выплаты. Без права ключ удаляется в TeamController.
Колонка «Юр. лицо» Те же условия JSON: legal_entity_label (displayTitle() при явной активной привязке), legal_entity_fallback (bool). Без привязки в multi-entity — пустой label, в UI «—», legal_entity_fallback=true. Явная привязка — только название без иконки fallback.
JSON edit Те же условия legal_entity_id — число; при multi-entity обязателен в валидации.

Валидация: PartnerLegalEntityId, Team\StoreRequest / UpdateRequest — legal_entity_id required при PartnerLegalEntityMode::isMultiEntity; сообщение «Выберите юр. лицо для группы». TeamService::resolveLegalEntityIdForPartner().

Non-AJAX safety-net (store): POST /admin/teams без X-Requested-With → redirect 302 на index, группа создана; при ошибке валидации — redirect back + session errors (не пустой 200). Update: успешный PATCH без AJAX → JSON 200 { message } (текущий контракт контроллера); ошибка валидации — redirect 302 + session errors.

5.3) Удалённые поля (история)

Следующие возможности сняты и в текущей версии CRM отсутствуют:
  • teams.address, право groups.address.view — миграция 2026_06_13_140000_refactor_team_location_and_drop_address.php. Адрес только у объекта (locations.address); на лендинге в team-info строка «Адрес» берётся из выбранного объекта.
  • teams.training_base, право groups.training_base.view — миграция 2026_06_13_150000_drop_training_base_from_teams.php. Поле не выводится в таблице, модалках и на лендинге.
  • Pivot location_team (many-to-many) заменён на teams.location_id — см. Объекты ↔ группы.

6) Привязка тренера к группе (trainer_profile_id)

Связь many-to-many team_trainer; в UI на группу — не более одного тренера (TeamTrainerSyncService::syncTrainerForTeam).
Сохранение: POST /admin/teams, PATCH /admin/team/{id} — поле trainer_profile_id (id из trainer_profiles текущего партнёра). Пустое значение снимает тренера с группы.
GET /admin/team/{id}/edit (JSON) возвращает trainer_profile_id первого связанного профиля.
Обратная привязка из карточки тренера — team_ids[] на /admin/trainers. Подробнее: Тренеры: привязка к группам.

7) Серверное поведение GET /admin/teams/data

  • Фильтры запроса: title, status (active / inactive), trainer_profile_id (id или none), location_id (id или none, только при locations.view), district_id (id или none, только при locations.view — фильтр по location.district_id), admin_user_id (id или none, только при locations.view — фильтр по pivot location_admin_user), sport_type_id (id или none, только при sport_types.view), параметры DataTables.
  • Eager load: weekdays, trainerProfiles.user, location (при locations.view), sportType (при sport_types.view).
  • Поля строки JSON: id, order_by, title, trainer_label, locations_label, locations_label_full, locations_names (при locations.view), weekdays_label, weekdays_items, sport_type_label (при sport_types.view), legal_entity_label / legal_entity_fallback (при multi legal entity mode), month_price, status_label, is_enabled.
  • Сортировка по columns[order.0.column].name: order_by, title, trainer_label, sport_type_label (join sport_types), locations_label (join locations по teams.location_id), month_price, status_label.
  • Обновление weekdays / trainer_profile_id / location_id / sport_type_id / legal_entity_id без соответствующего права — ключи отбрасываются в контроллере до сервиса (см. разделы 3–5.2). Поле month_price доступно при любом groups.view.

8) Базовые роли и конфиг

В config/role_base_permissions.php для роли admin:

  • groups.view — есть (доступ к разделу групп);
  • sport_types.view, sport_types.manage — нет (включаются вручную в матрице прав партнёра);
  • trainers.view — есть (колонка «Тренер» и селект в модалках у типового админа);
  • schedule.view — закомментировано (колонка «Расписание» и дни в модалках нужно включать в матрице прав партнёра).

Общая модель прав: Партнёры: базовые роли и права.
