Тренеры (админка)

Файл: /docs/documentation/admin-trainers.html · URL: /docs/documentation/admin-trainers · страница CRM: /admin/trainers

1) Назначение и модель данных

Тренер — это обычная учётная запись users с системной ролью trainer и дополнительным профилем trainer_profiles (1:1 по user_id).
ФИО, email, телефон, пароль, аватар — в users; описание, флаг активности профиля и порядок сортировки — в trainer_profiles.
Отдельной таблицы «тренеры» с собственным ФИО нет (историческая trainers удалена миграцией рефакторинга).
Таблица / сущностьНазначение
roles.name = trainer Системная роль (сидер RolesSeeder, id=4). В UI списка ролей скрыта: is_visible = 0. Базовые права — config/role_base_permissions.php → ключ trainer.
trainer_profiles Профиль тренера: partner_id, user_id (unique), description, is_enabled, sort_order, soft delete.
team_trainer Связь группа ↔ тренер (many-to-many с ограничением в UI: на группу — один тренер; у тренера — несколько групп). Поля: partner_id, team_id, trainer_profile_id.

Миграции: 2026_05_20_140000_refactor_trainers_to_trainer_profiles.php, 2026_05_20_150000_create_team_trainer_table.php.

Не путать: поле users.team_id у ученика — его основная группа в CRM. Привязка тренера к группам идёт только через team_trainer, не через users.team_id у тренера (у тренера team_id при сохранении сбрасывается в null).

Все маршруты раздела в routes/web.php обёрнуты в middleware('can:trainers.view'). Без trainers.view страница и JSON-операции отвечают 403. Пункт меню «Тренеры» — resources/views/includes/sidebar.blade.php (@can('trainers.view')).

2.1) Права

ПравоЧто даёт
trainers.view Раздел «Тренеры», CRUD, выбор тренера в модалках групп, чекбоксы групп у пользователя с ролью trainer, фильтр «Тренер» в отчётах (UI).

У роли admin по умолчанию trainers.view включён в config/role_base_permissions.php — см. базовые права партнёра.

3) Синхронизация профиля при смене роли

App\Services\TrainerProfileSyncService::syncForUser вызывается из UserService после обновления пользователя:

  • роль trainer → создать или восстановить (restore) trainer_profiles;
  • любая другая роль → soft delete существующего профиля по user_id.

Создание тренера через POST /admin/trainers сразу создаёт user с role_id = trainer и профиль. При обновлении через страницу тренеров, если у user была другая роль, но профиль остался — роль принудительно возвращается на trainer.

4) Привязка к группам

App\Services\TeamTrainerSyncService:

  • Из карточки тренера — массив team_ids[] в store/update: тренер привязывается к нескольким группам; лишние строки в team_trainer для этого профиля удаляются.
  • Из группы — поле trainer_profile_id в POST/PATCH /admin/team/{id} (TeamController): на группу назначается один тренер; предыдущий тренер группы снимается.

UI: чекбоксы групп — resources/views/admin/trainers/_teams_checkboxes.blade.php; в модалках групп — селект тренера при @can('trainers.view') — см. Группы (админка).

5) Редактирование пользователя с ролью «Тренер»

В модалке editUser.blade.php при trainers.view и выбранной роли trainer: скрывается селект основной группы (team_id), показываются чекбоксы team_ids[]. UserService::update синхронизирует группы только если у актора есть trainers.view и в запросе передан ключ team_ids. JSON GET /admin/users/{user}/edit отдаёт trainer_team_ids для предзаполнения.

Подробнее о разделе пользователей: Пользователи в админке.

6) UI страницы /admin/trainers

  • Список: ФИО, email, телефон, статус, действия.
  • Создание — модалка без аватара; обязательны имя и фамилия; email и пароль опциональны.
  • Редактирование — как у пользователя: аватар с cropper (editAvatar), маска телефона, смена пароля отдельной кнопкой → POST /admin/user/{id}/update-password (право users.password.update).
  • Ошибки валидации возвращаются JSON (422) и выводятся под полями формы.

7) Фильтр «Тренер» в отчётах

Параметр: filter_trainer_profile_id (id из trainer_profiles). Логика: берутся team_id из team_trainer для выбранного профиля, затем отбираются ученики с users.team_id из этого списка. Если у тренера нет групп — отчёт пустой (whereRaw('1 = 0')).

Подробная таблица отчётов и отличие от фильтра локации: Отчёты: фильтр по тренеру.

  • Select2 поиск: GET /admin/reports/payments/trainers-search (reports.payments.trainers.search) — в группе reports.view; по ФИО user.
  • Отчёты: Платежи, Платежи по месяцам, LTV, Задолженности — тот же параметр в DataTables и …/total.

8) Изоляция по партнёру

  • Список и операции — только trainer_profiles.partner_id = current_partner.
  • Чужой профиль по id → 404 на show/update/destroy.
  • team_ids и trainer_profile_id валидируются: группа и профиль должны принадлежать текущему партнёру.
