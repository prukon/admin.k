Расписание школы (календарь абонементов)

Страница входит в раздел «Абонементы» как вкладка «Расписание школы». Доступ к экрану и JSON календаря задаётся правом lessonPackages.view (группа маршрутов в routes/web.php с middleware can:lessonPackages.view). Отдельно: управление слотами расписания групп (scheduleSlots.manage / просмотр) описано в коде прав и тестах календаря; здесь — клиентская часть календаря и модалки слота.
Не путать с журналом посещаемости /schedule (schedule.view, таблицы schedule_users / statuses) — см. Журнал расписания.

Данные недели для сетки

Фронт подгружает слоты через GET /admin/lesson-packages/school-schedule/week (параметры week, опционально location_id). Каждый элемент массива occurrences содержит, среди прочего, поле team_title — название группы (команды) для отображения в ячейке и в UI.

Параметр location_idЗначение в UIФильтрация слотов (TeamScheduleCalendarService)
не передан или пустая строка Все Все активные слоты партнёра на неделе
none Без объекта Только team_schedule_slots.location_id IS NULL
числовой id объекта Название объекта из справочника Только слоты с team_schedule_slots.location_id = id

Парсинг параметра — LessonPackageController::schoolScheduleWeek. Конвенция none совпадает с фильтрами групп на /admin/teams и отчётами (filter_location_id).

Загрузка страницы календаря

Контроллер LessonPackageController::schoolSchedule передаёт во view:

  • teams — все группы партнёра с полями id, title, location_id (FK на объект, не pivot);
  • locations — активные объекты партнёра (id, name) для фильтра при locations.view.

Связь many-to-many Team::locations() не используется — одна группа привязана к одному объекту. Подробнее: Объекты ↔ группы.

Фильтр «Объект» в тулбаре календаря

Блок с выбором объекта (id="schoolCalLocation", подпись «Объект») рендерится только при наличии права locations.view (директива Blade @can('locations.view')). У базовой роли admin в конфиге config/role_base_permissions.php это право включено; у кастомных ролей его можно отозвать — тогда фильтр не показывается, но календарь по-прежнему доступен при lessonPackages.view (все слоты партнёра без фильтра по объекту).

Опции селекта (partial admin/lessonPackages/partials/locationFilterOptions.blade.php):

  • Все — value="", выбрано по умолчанию; при загрузке недели параметр location_id в query не передаётся;
  • Без объекта — value="none"; запрос …/week?location_id=none;
  • далее — активные объекты партнёра (locations.is_enabled = true), сортировка по названию.

При смене фильтра вызывается loadWeek() (перезагрузка сетки). Значение тулбара также подставляется в модалку «Добавить слот» при клике по пустой ячейке календаря.

Привязка групп к объектам (teams.location_id) — Объекты ↔ группы. Группа без объекта на лендинг не попадает; в слотах такая группа допустима (см. модалку ниже).

Модалки создания/редактирования слота расписания

На вкладке «Расписание школы» (и в таблице слотов групп) подключается admin.teamScheduleSlots.partials.slotModals. Создание/изменение — права scheduleSlots.view + scheduleSlots.manage (маршруты POST/PUT admin/team-schedule-slots).

Порядок полей и селект «Объект»

При locations.view в модалках create/edit поля идут в порядке: Объект → Группа → день недели → время → период. Те же опции, что в тулбаре календаря (locationFilterOptions): «Все», «Без объекта», список объектов.

Клиентская фильтрация групп — filterTeamSelectForLocation (класс .js-slot-location-select / .js-slot-team-select, атрибут data-location-id у option группы):

Выбранный объект в модалкеГруппы в селекте team_idСохраняемый location_id слота
Все (value="") Все группы партнёра null (пустая строка в форме)
Без объекта (value="none") Только группы с teams.location_id IS NULL null (none и "" нормализуются в Request)
Конкретный объект (id) Группы с teams.location_id = id Числовой id объекта

Нормализация перед валидацией: StoreTeamScheduleSlotRequest / UpdateTeamScheduleSlotRequest — location_id равный '' или 'none' → null.

Серверная проверка пары группа + объект слота

  • TeamLocationAvailabilityService::assertTeamAllowedAtLocation — при заданном location_id слота группа должна иметь тот же teams.location_id; группа без объекта при ненулевом объекте слота → 422 на team_id;
  • при location_id = null у слота ограничений по объекту группы нет (но для «Без объекта» UI показывает только группы без объекта).

При открытии edit JSON слота с location_id = null в селекте выбирается «Без объекта» (none).

Привязка абонементов и лимит строк календаря

При успешной привязке гибкого, фиксированного или разового абонемента к слотам создаются строки user_team_schedule_slots с заполненным user_lesson_package_id. Поле user_lesson_packages.lessons_remaining при этом не изменяется — оно корректируется только при смене статуса занятия (см. ниже). Ограничение по объёму абонемента выражено в том, что число таких строк по назначению не может превысить lessons_total. Детали периода starts_at/ends_at и логики фиксированной цепочки — в разделе «Абонементы».

Статусы занятия и списание остатков

Для каждой записи ученика на слот и дату (абонемент или пробное) в календаре школы можно сохранять статус из справочника партнёра lesson_occurrence_statuses. У статуса есть флаг consumes_lesson (например, «Посетил» и «Не посетил» — списывают, «Запись», «Отмена», «Заморозка» — нет; точный набор создаётся сидером и может быть расширен в админке с учётом этого флага).

  • Абонемент: дельта к user_lesson_packages.lessons_remaining считается по переходу «предыдущий статус списывал / новый списывает» (UserLessonPackageConsumptionAdjuster). Повторное сохранение того же типа списания не меняет остаток дважды.
  • Пробное: для строки с is_trial_lesson корректируются trial_lessons_remaining / trial_lessons_total (SchoolScheduleTrialLessonConsumptionAdjuster).

POST сохранения: /admin/lesson-packages/school-schedule/occurrence-status (имя маршрута admin.lesson-packages.school-schedule.occurrence-status.store). В ответе при успехе могут быть объекты user_lesson_package { id, lessons_remaining, lessons_total } или trial_registration { user_team_schedule_slot_id, lessons_remaining, lessons_total } — их использует фронт для обновления счётчиков в модалке.

GET истории по паре слот + дата + ученик (+ опционально user_lesson_package_id для различения записей): /admin/lesson-packages/school-schedule/occurrence-status/history.

JSON недели: регистрации в ячейке

Каждый элемент occurrences[] содержит массив registrations[]. Для строк из БД формируются поля (среди прочего):

  • registration_kind — trial или package;
  • is_trial_lesson, is_single_lesson — флаги типа записи;
  • schedule_type — тип шаблона абонемента (no_schedule у разового); для пробного — null;
  • lessons_remaining, lessons_total — для абонемента берутся из связанного user_lesson_packages, для пробного — из trial_lessons_* строки календаря;
  • line, user_label, current_status, occurrence_status_history_count — для отображения в модалке «Записаны на занятие».

Сервис форматирования: TeamScheduleCalendarService::formatSlotRegistrationLine.

Разовые занятия в модалке слота

Разовое занятие — назначение абонемента с шаблоном schedule_type = no_schedule (user_lesson_packages + строка user_team_schedule_slots). В отличие от гибкого/фиксированного абонемента, запись доступна без предварительного назначения на странице «Назначения»: при отсутствии свободного назначения создаётся новое в той же операции.

Доступность (GET …/slot-user-bind-actions)

Параметры: user_id, team_schedule_slot_id, occurrence_date. В ответе объект single_lesson:

  • allowed, reason — можно ли записать; при занятой ячейке слота/даты у ученика — allowed=false;
  • mode — bind_existing (есть «свободное» назначение: число строк в user_team_schedule_slots по этому user_lesson_package_id < lessons_total) или create_new;
  • existing_assignments[] — { id, label } для select (если несколько); label формирует formatSingleLessonAssignmentLabel;
  • templates[] — активные шаблоны партнёра (is_active, lessons_count > 0) с { id, label, fee_amount_default } (price_cents / 100). При bind_existing массив пустой.

Если свободных назначений нет и активных шаблонов разового занятия тоже нет — allowed=false, reason про отсутствие шаблонов.

Два разовых занятия у одного ученика в один день в разных слотах разрешены (проверка только пересечения слот+дата, не «одно разовое в день»).

UI-поток

  1. Выбрать ученика в модалке слота — фронт запрашивает slot-user-bind-actions.
  2. bind_existing, одно назначение — клик «Добавить разовое занятие» сразу вызывает POST без формы.
  3. bind_existing, несколько назначений — раскрывается select «Назначение», затем «Записать в расписание».
  4. create_new — всегда форма: шаблон + стоимость (редактируемая), затем submit.

Ошибки валидации с бэкенда (422, объект errors) выводятся под соответствующими полями формы. Legacy-модалка #schoolCalSingleModal и POST …/assign-single-lesson по-прежнему привязывают уже созданное назначение; основной сценарий — модалка слота.

API: запись (POST …/single-lesson-registration)

Имя маршрута: admin.lesson-packages.school-schedule.single-lesson-registration.store. Request: StoreSchoolCalendarSingleLessonRegistrationRequest. Сервис: SchoolCalendarSingleLessonRegistrationService::register.

Обязательные поля: user_id, team_schedule_slot_id, occurrence_date (Y-m-d).

Взаимоисключающие варианты (ровно один):

  • Привязка существующего: user_lesson_package_id — назначение из singleLessonAssignmentsQuery для этого ученика;
  • Создание нового: lesson_package_id + fee_amount (число ≥ 0) — шаблон из singleLessonTemplatesQuery; создаётся user_lesson_packages (is_paid=false, lessons_total/lessons_remaining из шаблона) и строка календаря в одной транзакции.

При привязке: период назначения задаётся через UserLessonPackageCalendarPeriodService::applyFirstCalendarAnchor (если даты ещё NULL); lessons_remaining не уменьшается. Успех — 200 с { "message": "…" }; фронт перезагружает неделю календаря.

В JSON недели у регистрации разового: is_single_lesson=true, schedule_type=no_schedule, registration_kind=package.

API: отмена записи (DELETE …/single-lesson-registration/{userTeamScheduleSlot})

Имя маршрута: admin.lesson-packages.school-schedule.single-lesson-registration.destroy. Допустима только строка с user_lesson_package_id и шаблоном no_schedule (не пробное, не гибкое/фиксированное). Идентификатор — user_team_schedule_slot_id из JSON недели. Назначение user_lesson_packages сохраняется и снова доступно для привязки.

Пробные занятия: добавление, баланс и ограничения

Пробное — запись без абонемента: user_team_schedule_slots с is_trial_lesson = true, user_lesson_package_id = NULL. Пробное действует только в выбранную дату: ends_at равно starts_at. При создании через API задаются trial_lessons_remaining = 1, trial_lessons_total = 1. Списание единицы пробного «урока» происходит при выборе статуса с consumes_lesson = true (аналогично абонементу, но поля другие).

У пользователя в таблице users хранится флаг has_used_school_schedule_trial: после успешного создания пробной записи он выставляется в true. Пока существует строка пробного у ученика (is_trial_lesson = true и user_lesson_package_id IS NULL), вторую создать нельзя; те же условия проверяются в SchoolCalendarTrialEligibilityService и при создании (блокировка строки в транзакции). Ячейка слота/даты не должна быть занята абонементом или посторонней записью.

GET …/trial-registration-eligibility — заранее проверяет возможность добавления (ответ allowed, reason).

UI: «(Отменить)» в списке записей

В модалке слота в блоке «Записаны на занятие» ссылка (Отменить) выводится для:

  • пробного — рядом с подписью «Пробное занятие»;
  • разового (is_single_lesson или schedule_type=no_schedule) — рядом с названием абонемента в строке записи.

Доступна при праве lessonPackages.view. Пробное снимается через DELETE …/trial-registration/{id}; разовое — через DELETE …/single-lesson-registration/{id}. Гибкие и фиксированные абонементы в этом списке ссылку «(Отменить)» не получают.

API: удаление (отмена) пробного занятия

Для отмены используется эндпоинт:

  • DELETE /admin/lesson-packages/school-schedule/trial-registration/{userTeamScheduleSlot}

Идентификатор берётся из JSON недели (каждая регистрация содержит поле user_team_schedule_slot_id).

Побочные эффекты отмены

Пробное (DELETE …/trial-registration/{id})

  • Очистка статусов: удаляются записи в user_lesson_occurrence_status_events для этой даты/слота/ученика с user_lesson_package_id IS NULL.
  • Аудит: my_logs, event = schedule.trial_cancelled (см. аудит my_logs).

Разовое (DELETE …/single-lesson-registration/{id})

  • Строка календаря удаляется; назначение user_lesson_packages остаётся в системе.
  • Восстановление остатка: если последний статус по этой записи имел consumes_lesson = true, к lessons_remaining применяется дельта +1 (UserLessonPackageConsumptionAdjuster).
  • Очистка статусов: удаляются события в user_lesson_occurrence_status_events для этой даты/слота/ученика с тем же user_lesson_package_id.
  • Аудит: my_logs, event = schedule.single_lesson_registration_cancelled.

Связь с логикой абонементов

Период назначения starts_at/ends_at, лимит строк календаря по lessons_total, разовое занятие (no_schedule) и отсутствие списания lessons_remaining при самой привязке описаны в «Абонементы: назначение и период с календаря школы». Текущая страница дополняет это правилами статусов, пробного баланса, разовой записи из модалки слота и полей JSON недели; разметка модалки и право locations.view — только отображение.
