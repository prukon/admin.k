Объекты ↔ группы (teams.location_id)

Файл: /docs/documentation/location-team-bindings.html · миграции: 2026_06_13_140000_refactor_team_location_and_drop_address.php, 2026_06_13_150000_drop_training_base_from_teams.php
Суть: у каждой группы (teams) не более одного объекта — nullable FK teams.location_id → locations. Раньше связь была many-to-many через pivot location_team (удалён); «универсальные» группы без привязки трактовались как доступные во всех объектах — эта модель снята.
Группу можно создать без объекта (location_id = null), но на лендинг заявки такая группа не попадает.
Адрес на лендинге и в справочниках — только у объекта (locations.address); колонки teams.address, teams.training_base и права groups.address.view, groups.training_base.view удалены.
У ученика нет users.location_id — «география» в отчётах идёт через группы ученика в pivot team_user → teams.location_id (см. student-team-membership). Администраторы объекта (location_admin_user, M2M) — отдельная связь; на привязку групп к объекту не влияют (см. admin-locations §3). Локация занятия в календаре — отдельно: team_schedule_slots.location_id.

1) Правила доступности группы в объекте

Состояние группыДоступна в объекте LНа лендинге (GET …/teams?location_id=)
teams.location_id IS NULL Нет (ни в одном объекте для слотов и фильтров «по объекту») Не показывается
teams.location_id = L Только в объекте L Показывается при запросе с location_id = L
teams.location_id = L2 (другой объект) Только в L2, не в L Не показывается для L
Ученик без групп в pivot В отчёте «Задолженности» и на вкладке «Назначение абонементов» при фильтре по конкретному объекту не попадает (кроме режима «Без объекта» / none)

Реализация: App\Services\TeamLocationAvailabilityService (isTeamAllowedAtLocation, scopeAvailableForLocation, applyDebtUserTeamLocationFilter).

Перенос группы с объекта A на объект B меняет teams.location_id (одна группа не может быть одновременно в двух объектах). Sync с формы объекта снимает привязку у групп, убранных из multiselect team_ids[].

2) Схема БД

locations (Объект)
    └── teams.location_id (nullable FK, nullOnDelete)
            └── team_user (ученик ↔ группы, M:N)

school_leads.district_id + school_leads.location_id — снимок заявки
team_schedule_slots.location_id — объект слота расписания
  • teams.location_id — nullable, FK → locations, nullOnDelete (удаление объекта обнуляет привязку групп).
  • Таблица location_team — удалена (данные перенесены: для каждой группы взята минимальная location_id из pivot).
  • Колонка teams.address — удалена; адрес хранится в locations.address.
  • Колонка teams.training_base и permission groups.training_base.view — удалены (строка «Тренировочная база» больше не выводится в team-info на лендинге).
  • Модели: Team::location() (BelongsTo), Location::teams() (HasMany).
  • Синхронизация с формы объекта: App\Services\TeamLocationSyncService::syncTeamsForLocation.

3) UI и API: группы (/admin/teams)

Доступ к разделу — groups.view. Поле «Объект» — дополнительно locations.view.

  • Модалки create/edit: select location_id (один объект; пустое значение — без объекта).
  • POST /admin/teams, PATCH /admin/team/{id} — при locations.view ключ location_id сохраняется; без права ключ отбрасывается в TeamController.
  • GET /admin/team/{id}/edit — JSON location_id (число или null, только при locations.view).
  • Фильтр таблицы: параметр location_id — id объекта или none (группы с location_id IS NULL). Без locations.view параметр игнорируется.
  • Колонка «Объект» (locations_label) — название объекта или «—»; сортировка по join locations.
  • Валидация: Team\StoreRequest / UpdateRequest — location_id nullable, exists у locations текущего партнёра (rules только при locations.view); ошибки 422 на русском под полем.

Подробнее: admin-teams §5.

4) UI и API: объекты (/admin/locations)

Доступ — locations.view; создание/изменение/удаление — locations.manage.

  • Модалки create/edit: Select2 multiselect team_ids[] — список групп, привязанных к этому объекту (partial partials.select2.teams-multiselect).
  • POST /admin/locations, PUT /admin/locations/{location} — team_ids → TeamLocationSyncService::syncTeamsForLocation: выбранным группам ставится location_id текущего объекта; снятым с объекта — null; группа, перенесённая на другой объект через форму того объекта, получает новый location_id.
  • GET /admin/locations/{location} — JSON с team_ids (id групп, у которых location_id = этот объект).
  • DELETE /admin/locations/{location} — у связанных групп location_id → null (не cascade delete групп).

Обратная сторона — select location_id на /admin/teams. Подробнее: admin-locations.

5) Лендинг заявки

  • GET /lead/{slug}/teams?location_id= — только активные группы партнёра с teams.location_id = location_id (SchoolLeadLandingService::teamsForLocation).
  • GET /lead/{slug}/team-info?location_id=&team_id= — строка «Адрес» берётся из locations.address выбранного объекта, не из группы.
  • Группа без объекта или с другим location_id → 404 на team-info.

См. school-leads-landing.

6) Расписание школы и слоты

  • Страница GET /admin/lesson-packages/school-schedule загружает группы как Team::…->get(['id', 'title', 'location_id']) — без eager-load связи locations (одна группа привязана к одному объекту через FK).
  • Календарь: фильтр «Объект» в тулбаре и модалке слота — опции Все (без query-параметра), Без объекта (location_id=none), id объекта. API недели: GET …/school-schedule/week — см. Расписание школы § фильтр.
  • Модалки слота (slotModals.blade.php): порядок полей — объект, затем группа; partial locationFilterOptions; у option группы атрибут data-location-id; клиент — filterTeamSelectForLocation.
  • Сервер: POST/PUT admin/team-schedule-slots — assertTeamAllowedAtLocation → 422 на team_id, если у слота задан объект, а у группы другой или пустой teams.location_id. Значения none / пустая строка в location_id формы → null в БД.

7) Пользователи — без локации в карточке

В разделе /admin/users нет поля, фильтра и колонки «Локация ученика». Входящий location_id в POST/PATCH /admin/users не сохраняется. При создании клиента из заявки в ученика не копируется school_leads.location_id.

См. Пользователи §3.

8) Отчёты и платежи

ОтчётИсточник объекта для фильтра
Платежи, LTV, платежи по месяцам payments.location_id — снимок при первом создании строки платежа
Задолженности pivot team_user → teams.location_id. filter_location_id=none — нет групп в pivot или группа с location_id IS NULL
Назначение абонементов Тот же pivot и тот же filter_location_id, что у «Задолженностей» (applyDebtUserTeamLocationFilter). См. lesson-packages §assignments-tab

Единый параметр UI: filter_location_id (пусто / none / id). Право: locations.view. См. Отчёты §2.1 и lesson-packages §assignments-tab.
