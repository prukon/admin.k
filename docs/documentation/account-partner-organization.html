Организация партнёра: кабинет и админка

Файл: /docs/documentation/account-partner-organization.html · Юр. лица (справочник): admin-legal-entities · Оплаты / resolver: admin-legal-entities §1.1 · Права партнёра: partners-permissions
С 2026‑06‑30 реквизиты и данные юридического лица (ИНН, ОГРН/ОГРНИП, банк, тип бизнеса, НДС для чека и т.д.) хранятся и редактируются только в справочнике «Юр. лица» (partner_legal_entities).

На вкладке «Организация» в личном кабинете и в модалках CRUD супер-админа (/admin/partners) остаются только операционные поля партнёра (бренд клуба, контакты, SMS-имя, сортировка, активность). Реквизиты, город, индекс и данные руководителя на partners в UI отсутствуют. Legacy-колонки в таблице partners в БД не удалены; приложение их не читает из форм и не пишет (PartnerLegacyLegalFields::strip). При update уже сохранённые в БД city/zip/ceo не затираются, но из интерфейса недоступны.

URL: GET /account-settings/partner/edit (admin.cur.company.edit). Шаблон: resources/views/account/organizations.blade.php (вкладка partner в account.index).

3) Админка супер-пользователя: раздел «Партнёры»

URL: /admin/partners. Модалки create/edit: resources/views/includes/modal/editPartner.blade.php.

3.1) Поля create/edit (актуальные)

  • Основное: title (подпись «Название школы/секции»), sms_name, phone, email, website
  • Служебное: order_by, is_enabled

Legacy-поля (тип бизнеса, ИНН, банк, НДС, город, индекс, данные руководителя и т.д.) из модалок и JSON GET /admin/partner/{id}/edit убраны. Колонки partners.city, partners.zip, partners.ceo в БД сохранены, но приложение их не читает и не пишет (PartnerLegacyLegalFields::strip). В таблице партнёров удалены колонки «Организация» и «ИНН».

3.5) AJAX-submit модалок

  • Create — POST /admin/partners, заголовок X-Requested-With: XMLHttpRequest; успех → модалка успеха + reloadPartnersTable().
  • Edit — клик .edit-partner-link → GET …/edit → заполнение полей → PATCH …/partner/{id} по кнопке «Обновить».
  • Ошибки 422 — под полями (invalid-feedback / #edit-{field}-error).

4) Оплаты и фискализация без legacy fallback

Важно: если у партнёра нет ни одного активного юр. лица с зарегистрированным ShopCode (tinkoff_shop_code), приём платежей T‑Bank недоступен (PaymentService::isTbankAvailable() → false). Legacy partners.tinkoff_partner_id и partners.tax_id не используются. 
LegalEntityResolver (актуально):
  ShopCode     = entity.tinkoff_shop_code (только из справочника)
  fiscalTaxId  = entity.tax_id
  fiscalVat    = entity.vat
  fiscalOrgName = entity.organization_name → entity.title

Без активного юр. лица → null / пусто → T‑Bank и CloudKassir (ИНН принципала) не работают

Подробнее о выборе юр. лица по группе — payments §2.3.2, жёсткий guard при ≥2 юр. лицах — payments §2.3.5, справочник — admin-legal-entities.

Исключение (операционные поля партнёра): для CloudKassir по-прежнему читаются partners.email (обязателен), phone, website — не путать с legacy tax_id / organization_name на партнёре. См. admin-legal-entities §5.1.

5) Legacy sm-register на уровне партнёра

Маршруты POST /admin/tinkoff/partners/{id}/sm-register (и sm-patch / sm-refresh / sm-pull) возвращают redirect на справочник «Юр. лица» или 410 JSON. Регистрация T‑Bank — только через карточку юр. лица.
