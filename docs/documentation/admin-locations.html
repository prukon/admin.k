Объекты (админка): /admin/locations

Файл: /docs/documentation/admin-locations.html · обзор: directories-hierarchy · районы: admin-districts · привязка к группам: teams.location_id
UI «Объект» = модель Location (таблица locations). Раздел в меню: вкладка «Объекты» (вторая; общий partial, $activeTab = 'objects'). Заголовок страницы — «Справочники».
Сайдбар: при единственном locations.view пункт меню — «Объекты»; при 2+ правах справочников — «Справочники» (DirectoriesMenu). Отдельного пункта «Объекты» в сайдбаре нет.
Порядок вкладок: Группы → Объекты → Районы → Виды спорта.
Маршруты — под middleware('can:locations.view'). Создание/изменение/удаление — дополнительно locations.manage.

2) Поле «Район» (district_id)

  • district_id — nullable; select «Район» в create/edit (список активных районов партнёра, сортировка sort_order → name).
  • Колонка «Район» и фильтр в таблице объектов.
  • Уникальность имени: unique(partner_id, district_id_unique_bucket, name), где district_id_unique_bucket = COALESCE(district_id, 0).
  • Объекты без района не попадают на лендинг заявки (см. school-leads-landing).

3) Поле «Администратор» (admin_user_id)

  • Колонка locations.admin_user_id — nullable FK → users.id, nullOnDelete (миграция 2026_06_13_160000_add_admin_user_id_to_locations_table.php).
  • Допустимы только активные пользователи текущего партнёра с системной ролью admin (App\Support\PartnerAdminUserOptions).
  • Один администратор может быть назначен на несколько объектов; у объекта — не более одного администратора. Объект может быть без администратора.
  • Select «Администратор» в модалках create/edit — только при locations.manage; пустое значение → null.
  • Колонка «Администратор» (admin_user_label) и фильтр таблицы — при locations.view (значения: все / «Без администратора» (none) / id). Сортировка — join к users по ФИО.
  • JSON GET /admin/locations/data: ключи admin_user_id, admin_user_label (ФИО или пустая строка).
  • Модель: Location::adminUser(), User::managedLocations().
  • Валидация: LocationFieldRules — admin_user_id nullable, exists у users партнёра с ролью admin; ошибки под полем «Администратор».

4) Интерфейс страницы

View: resources/views/admin/locations/index.blade.php, вкладки справочников — admin/directories/_section_tabs.blade.php.

  • Toolbar — «Добавить», «Фильтры», «Колонки».
  • Фильтры: название/адрес, статус, район (все / «Без района» / id), администратор (все / «Без администратора» / id).
  • Колонка «Группы» — при наличии групп у партнёра; tooltip со списком (тип list, сокращение «X, еще N шт.» при 3+ объектах в списке).
  • Модалки — multiselect team_ids[] и select admin_user_id при locations.manage.
  • Колонка «Название» — при locations.manage тип link (js-location-edit, модалка edit); без manage — обычный text. См. reusable-ui-partials §3.2.

5) Привязка групп (team_ids)

Sync через TeamLocationSyncService::syncTeamsForLocation: выбранным группам проставляется teams.location_id текущего объекта; снятым — null. Обратная сторона — select location_id на /admin/teams. Подробнее: Объекты ↔ группы.

6) Связанные сущности

  • school_leads.location_id и school_leads.district_id — заявка (UI «Объект» / «Район»).
  • team_schedule_slots.location_id — слот расписания.
  • payments.location_id — снимок в журнале оплат.
