Объекты (админка): /admin/locations

Файл: /docs/documentation/admin-locations.html · обзор: directories-hierarchy · районы: admin-districts · привязка к группам: teams.location_id
UI «Объект» = модель Location (таблица locations). Раздел в меню: вкладка «Объекты» (вторая; общий partial, $activeTab = 'objects'). Заголовок страницы — «Справочники».
Сайдбар: при единственном locations.view пункт меню — «Объекты»; при 2+ правах справочников — «Справочники» (DirectoriesMenu). Отдельного пункта «Объекты» в сайдбаре нет.
Порядок вкладок: Группы → Объекты → Районы → Виды спорта.
Маршруты — под middleware('can:locations.view'). Создание/изменение/удаление — дополнительно locations.manage.

2) Поле «Район» (district_id)

  • district_id — nullable; select «Район» в create/edit (список активных районов партнёра, сортировка sort_order → name).
  • Колонка «Район» и фильтр в таблице объектов.
  • Уникальность имени: unique(partner_id, district_id_unique_bucket, name), где district_id_unique_bucket = COALESCE(district_id, 0).
  • Объекты без района не попадают на лендинг заявки (см. school-leads-landing).

3) Поле «Администратор» (admin_user_ids[])

Модель связи: many-to-many через pivot location_admin_user. Раньше использовалась колонка locations.admin_user_id (0..1 админ на объект) — она удалена миграцией 2026_06_13_170000_create_location_admin_user_table.php; существующие значения перенесены в pivot одной строкой на объект. Обратной совместимости API по полю admin_user_id нет.
  • Pivot location_admin_user — M2M между locations и users. Колонки: id, partner_id, location_id, user_id, timestamps; unique(location_id, user_id); cascadeOnDelete на FK к partners, locations, users.
  • Допустимы только активные (is_enabled = 1) пользователи текущего партнёра с системной ролью admin (App\Support\PartnerAdminUserOptions). Не более 5 администраторов на объект.
  • Один администратор может быть назначен на несколько объектов; у объекта — от 0 до 5 администраторов.
  • Multiselect «Администратор» (admin_user_ids[]) в модалках create/edit — только при locations.manage; пустой выбор → объект без администраторов (при update отправляется admin_user_ids: [] — полная замена списка).
  • Колонка «Администратор» (admin_user_label, тип DataTables list, itemsKey: admin_user_names) и фильтр таблицы — при locations.view. Фильтр: все / «Без администратора» (none) / id пользователя. Сокращение «X, еще N шт.» при 3+ ФИО; tooltip — полный список (admin_user_label_full). Сортировка — по минимальному ФИО среди администраторов объекта (подзапрос к pivot + users).
  • Модель: Location::adminUsers() (belongsToMany), User::managedLocations() (belongsToMany). Sync: LocationAdminUsersSyncService::syncAdminsForLocation — полная замена списка; невалидные id (чужой партнёр, не admin, выключен) отбрасываются на уровне сервиса после валидации Request.
  • Валидация: LocationFieldRules — admin_user_ids nullable array, max 5; admin_user_ids.* — integer, exists у users партнёра с ролью admin и is_enabled = 1. Сообщения: «Можно назначить не более 5 администраторов», «Выберите администратора из списка текущего партнёра» — под полем «Администратор».
  • При удалении объекта или пользователя строки pivot удаляются каскадом. Назначение администраторов не влияет на права доступа к CRM и не ограничивает видимость объектов для текущего пользователя — только справочник и фильтры (см. также фильтр «Администратор» на /admin/teams).

3.1) API: тела запросов и ответы

МетодПоля запроса (админы)Ответ / data
POST /admin/locations admin_user_ids[] — необязательный массив id (0..5). Отсутствие или [] → объект без админов. { message, location } — в location нет вложенного списка админов; после create pivot заполняется sync-сервисом.
PUT /admin/locations/{id} admin_user_ids[] — полная замена: только переданные id остаются в pivot. [] снимает всех админов. { message }
GET /admin/locations/{id} — admin_user_ids: number[] — отсортированы по загрузке связи; также district_id, team_ids (при locations.view).
GET /admin/locations/data Фильтр: admin_user_id — id админа или none (объекты без ни одной строки в pivot). Имя параметра в фильтре — единственное число; в формах create/edit — admin_user_ids[]. На каждую строку: admin_user_ids, admin_user_label, admin_user_label_full, admin_user_names (массив ФИО по возрастанию lastname, name).

4) Интерфейс страницы

View: resources/views/admin/locations/index.blade.php, вкладки справочников — admin/directories/_section_tabs.blade.php.

  • Toolbar — «Добавить» (при locations.manage), «История», «Фильтры», «Колонки».
  • Фильтры: название/адрес, статус, район (все / «Без района» / id), администратор (все / «Без администратора» (none) / id пользователя — query-параметр admin_user_id в /data).
  • Колонка «Группы» — при наличии групп у партнёра; tooltip со списком (тип list, сокращение «X, еще N шт.» при 3+ объектах в списке).
  • Колонка «Администратор» — тип list, тот же паттерн сокращения, что у «Группы».
  • Модалки — multiselect team_ids[] и admin_user_ids[] при locations.manage.
  • Колонка «Название» — при locations.manage тип link (js-location-edit, модалка edit); без manage — обычный text. См. reusable-ui-partials §3.2.

5) Аудит (my_logs)

Запись — через AuditLogger (audit-my-logs). События: location.created, location.updated, location.deleted (категория location).

  • create — снимок всех полей: название, район, адрес, описание, активность, администраторы, группы.
  • update — diff «старое → новое» по тем же полям; лог только при наличии изменений; группы — только если выполнялся sync team_ids.
  • delete — «Объект удалён: {name}. ID: {id}.» (level = security).
  • UI — кнопка «История» (иконка fa-clock-rotate-left) в toolbar → модалка includes.logModal, DataTables GET /admin/locations/logs-data (route('logs.data.location')).

6) Привязка групп (team_ids)

Sync через TeamLocationSyncService::syncTeamsForLocation: выбранным группам проставляется teams.location_id текущего объекта; снятым — null. Обратная сторона — select location_id на /admin/teams. Подробнее: Объекты ↔ группы.

7) Связанные сущности

  • school_leads.location_id и school_leads.district_id — заявка (UI «Объект» / «Район»).
  • team_schedule_slots.location_id — слот расписания.
  • payments.location_id — снимок в журнале оплат.
