Консоль ученика (/cabinet)

Файл: /docs/documentation/dashboard-cabinet.html · маршрут: dashboard · view: resources/views/dashboard.blade.php
Кратко. Страница «Консоль» — личный кабинет ученика (или семейный просмотр данных ребёнка): профиль, календарь посещений, дополнительные платежи, сезоны месячных абонементов с кнопками «Оплатить». Для роли с правом users.view (админ CRM) на той же странице доступны селекты «Группа» / «ФИО» и AJAX-подгрузка данных другого ученика. При 2+ группах у активного ученика в пользовательском режиме показывается селект «Выбор группы» — фильтрация месячных начислений по users_prices.team_id (только на клиенте).

1) Маршруты и права

МетодURLИмяPermissionНазначение

Middleware группы: auth, 2fa, can:dashboard.view. Без права — 403; гость — redirect на login / не 200 на JSON.

Оплата с консоли: POST /payment (paying.classes). См. payments §3.1.1, блок «Поставщик услуг» при T‑Bank — payments §3.1.2.

2) Контекст ученика на странице

DashboardController::index берёт ученика через FamilyStudentContextService::activeStudent(auth()->user()) — тот же механизм, что «Мои платежи» и семейный переключатель в сайдбаре.

  • Имя, дата рождения, почта, кастомные поля, расписание, users_prices, доп. платежи — из activeStudent.
  • Аватар в шаблоне — из auth()->user() (учётка входа), не из активного ребёнка. См. parents-and-family-cabinet §5.4.
  • Группы в pivot загружаются с фильтром teams.partner_id = current_partner.

3) Два режима UI на одной странице

УсловиеЧто видит пользователь
Есть users.view (админ CRM)Блок «Выбор ученика»: Select2 «Группа» + «ФИО», AJAX getUserDetails / getTeamDetails. Селект «Выбор группы» для ученика не показывается.
Нет users.view (роль user)Только свои (или семейные) данные активного ученика; при 2+ группах — селект «Выбор группы» (§4).

4) Селект «Выбор группы» (2+ группы)

Отдельного backend-маршрута для переключения группы на консоли нет: выбор хранится в sessionStorage браузера и применяется в JavaScript при отрисовке сезонов.

4.1) Условия показа

  • Partial: resources/views/includes/dashboard_team_switcher.blade.php.
  • Пользователь без права users.view (@cannot('users.view')).
  • У activeStudent в pivot 2 и более групп партнёра ($curUser->teams->count() >= 2).
  • При одной группе селект скрыт; жирное выделение названия группы в шапке не используется.

4.2) Размещение и подпись

  • Блок расположен сразу над секцией сезонов (.row.seasons), после календаря и блока дополнительных платежей.
  • Label селекта: «Выбор группы», элемент: #dashboard-active-team.

4.3) Поле «Группа:» в шапке профиля

  • При 2+ группах выводятся все названия через запятую (span.dashboard-group-name, атрибут data-multi-team="1" на .group-value).
  • Название группы, выбранной в селекте, выделяется жирным (<strong>) через JS (updateDashboardGroupLabel).
  • При одной группе — обычный текст без data-multi-team и без жирного.

4.4) Поведение при смене группы (клиент)

  1. На загрузке страницы в JS передаются все строки users_prices (userPriceAll) и список групп (dashboardTeams).
  2. Ключ sessionStorage: dashboard_active_team_id_{userId}, где userId — id активного ученика (важно при семейном переключении детей).
  3. Если в storage нет валидного id — выбирается первая группа из pivot (как дефолт на «Моя группа»).
  4. После выбора: фильтр userPrice по team_id → пересборка ячеек сезонов (apendPrice), скрытие пустых сезонов, пересчёт задолженностей.
  5. В форме «Оплатить» за месяц в hidden team_id попадает группа отфильтрованной строки users_prices (согласовано с payments §2.1).

4.5) Что селект группы не меняет

БлокПричина
Календарь посещенийДанные schedule_users не привязаны к группе на консоли; фильтрация не реализована.
Дополнительные платежи (user_custom_payment)Список на консоли не фильтруется селектом «Выбор группы»; team_id задаётся в админке при создании записи и используется при оплате (setting-prices-custom-payments).
AJAX getUserDetails (админ)По-прежнему отдаёт все userPrice; фильтр только в пользовательском режиме на клиенте.

4.6) Связь с несколькими начислениями за один месяц

До появления селекта при 2+ группах в одной ячейке месяца могли отображаться несколько кнопок «Оплатить (Группа …)». В пользовательском режиме с активным селектом за месяц показывается одна кнопка для выбранной группы. Логика нескольких кнопок в apendPrice сохранена для админского AJAX (без селекта «Выбор группы»).

5) Сезоны и месячные оплаты

  • Ячейки месяцев генерируются в JS (createSeasons), цены подставляются из массива userPrice (или отфильтрованного под группу).
  • Каждая оплачиваемая ячейка содержит paymentDate, formatedPaymentDate (YYYY-MM-01), hidden team_id, outSum (на сервере для monthly_fee не источник истины).
  • Resolver оплаты: UserPriceMonthlyFeePaymentResolver — при нескольких начислениях за месяц без team_id в POST → 422 «Укажите группу…»; селект на консоли как раз передаёт нужный team_id.
