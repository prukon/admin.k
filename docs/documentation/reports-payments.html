Отчёт “Платежи” (админка)

Файл: /docs/documentation/reports-payments.html · список разделов (/doc)

1) Где находится

  • Страница: /admin/reports/payments

2) Поля списка (управление видимостью колонок)

На странице отчёта доступна кнопка “Поля списка” (иконка колонок). Можно включать/выключать отображение колонок таблицы (включая “Действия”). Настройка сохраняется для текущего пользователя.
  • Где хранится: таблица user_table_settings
  • Ключ таблицы: table_key = reports_payments
  • API:
    • GET /admin/reports/payments/columns-settings — получить настройки
    • POST /admin/reports/payments/columns-settings — сохранить настройки (columns: {key: true/false})
  • Все эти маршруты (включая страницу и DataTables) находятся в группе с middleware can:reports.view — без права reports.view ответ 403.

3) Колонки: выплаты и комиссии (T‑Bank)

Эти колонки заполняются только для платежей через T‑Bank. Для других провайдеров значения будут пустыми.
Источник расчёта комиссий: правила из таблицы tinkoff_commission_rules. Комиссии удерживаются с партнёра. Округление: математическое (round) до копейки. Формула: 
bank_acquiring_fee = max(round(gross * acquiring_percent/100), acquiring_min_fixed)
bank_payout_fee    = max(round(gross * payout_percent/100),    payout_min_fixed)
platform_fee       = max(round(gross * platform_percent/100),  platform_min_fixed)

commission_total = bank_acquiring_fee + bank_payout_fee + platform_fee
net_to_partner   = gross - commission_total
  • Выплата (payout_amount): фактическая сумма выплаты партнёру. Заполняется только после успешного зачисления (статус выплаты COMPLETED в tinkoff_payouts). Если выплаты не было — поле пустое.
  • К выплате (net_to_partner): расчётная сумма к выплате партнёру (нетто). Заполняется сразу после оплаты по формуле: сумма − (комиссии банка + комиссия платформы).
  • Комиссия (commission_total): суммарные удержания по тарифу и комиссии банка (без детализации), заполняется всегда.
    В hover: “Суммарные удержания по тарифу и комиссии банка”.
  • Комиссия банка (bank_commission_total): сумма комиссии за эквайринг + комиссии за выплату. Заполняется всегда (даже если выплаты ещё не было), рассчитывается по правилам tinkoff_commission_rules.
    В hover показывается расшифровка: “Эквайринг” и “Выплата”.
  • Комиссия платформы (platform_commission): комиссия платформы, заполняется всегда, рассчитывается по правилам tinkoff_commission_rules.
Права доступа (интерфейс «Поля списка» и видимость колонок): без права reports.additional.value.view в таблице по умолчанию скрыты и нельзя включить через чекбоксы колонки с ключами bank_commission_total, platform_commission, commission_total, net_to_partner, refund_status (то же правило, что раньше действовало только для «Комиссия банка» и «Комиссия платформы»). Колонка «Выплата» (payout_amount), «Действия», базовые поля (ФИО, сумма, чек и т.д.) от этого права не зависят. Право нужно явно назначить роли в разрезе партнёра (permission_role); в базовом конфиге роли admin (config/role_base_permissions.php) оно не входит.
Важно (API DataTables): эндпоинт GET /admin/reports/getPayments по-прежнему отдаёт в JSON значения комиссий, нетто и refund_status для всех, у кого есть reports.view — ограничение касается отображения в UI и переключателей «Поля списка». Отдельно сервером защищён только просмотр таймлайна T‑Bank: см. раздел «История T‑Bank» ниже.
Порядок колонок (после “Провайдер”): “Комиссия банка” → “Комиссия платформы” → “Комиссия” → “К выплате” → “Выплата”.

4) Колонка “Чек” (онлайн‑чек)

Колонка “Чек” в отчёте “Платежи” показывает онлайн‑чеки из fiscal_receipts и быстрый переход к ним.
  • Для T‑Bank:
    • Оплата (income): берётся последняя запись чека по FiscalReceipt.type=income (в UI используется поле receipt_url). Если receipt_url валиден и начинается с https://receipts.ru/ — показываем активную иконку (ссылка открывается в новой вкладке).
    • Возврат (income_return): после успешного T‑Bank Cancel система создаёт чек возврата в CloudKassir и сохраняет его в fiscal_receipts как FiscalReceipt.type=income_return. Если URL возврат‑чека (return_receipt_url) валиден — показывается вторая иконка “Чек возврата” (также открывается в новой вкладке).
    • Если возврат уже создан, но чек возврата ещё не сформирован: иконка income будет иметь расширенную подсказку (tooltip) вида “Чек возврата ожидается (CloudKassir)” или “Чек возврата: ошибка”. Вторая иконка при этом не показывается до появления валидного return_receipt_url.
    • Если income‑чек не сформирован: показывается серая иконка с подсказкой “Чек не сформирован”.
  • Для Robokassa: иконка не показывается (ячейка пустая), подсказка: “Чек формируется у партнера в его онлайн-кассе”.
  • Почему так: при Robokassa чек формируется в инфраструктуре партнёра (его касса/ОФД), и платформа не имеет гарантированного доступа к ссылке на чек.
  • Источник записи: для income и income_return используются отдельные выборки “последнего” чека по payment_id (через MAX(id) внутри каждого type).
Важно: в JSON выдаче DataTables присутствуют технические поля receipt_url/has_receipt и return_receipt_url/has_return_receipt. В UI финальное отображение всё равно зависит от провайдера платежа (tbank/robokassa) по правилам выше.

5) Колонки: возвраты

В таблице есть 2 колонки, связанные с возвратами:
  • Действия (refund_action) — кнопка “Возврат” (может быть недоступна, если возврат запрещён правилами). Для платежей T‑Bank в той же ячейке может отображаться кнопка «История» — см. раздел 7.
  • Статус возврата (refund_status) — текущий статус возврата (например, pending/succeeded/failed). Видимость колонки в UI и доступ к переключателю в «Поля списка» — только при reports.additional.value.view (раздел 3).
Порядок этих колонок в таблице: “Действия”, затем “Статус возврата”.

5.1) T‑Bank: возврат и отмена отложенной выплаты партнёру

API: POST /admin/reports/payments/{payment}/refund (тело: опционально comment), в группе can:reports.view. Платёж должен относиться к current_partner.
  • Когда возврат запрещён (ответ 422): если по связке через таблицу tinkoff_payments найдена выплата tinkoff_payouts со статусом не REJECTED и с непустым tinkoff_payout_payment_id (выплата уже отправлена в банк на этапе e2c Init).
  • Когда возврат разрешён при «висящей» выплате: статус выплаты INITIATED (или иной не REJECTED), но банковский PaymentId выплаты ещё пустой — перед созданием возврата такие строки переводятся в REJECTED, сбрасывается when_to_run, в payload_state фиксируется cancelled_by_refund.
  • Кнопка в таблице: отключается по той же логике, что и сервер (поиск выплат только через tinkoff_payments, без опоры на deal_id отчёта).
Подробно (модель данных, поиск строк, джоба, идемпотентность, тесты): tbank-refunds-payout-cancel.html.

6) Права доступа: сводка по отчёту

Ниже — соответствие прав и функционала. Подробнее о модели прав: партнёры и permission_role.
  • reports.view — обязательно для страницы /admin/reports/payments, выборки /admin/reports/getPayments (AJAX), GET/POST …/columns-settings, создания возврата POST /admin/reports/payments/{payment}/refund.
  • reports.additional.value.view — управление видимостью «чувствительных» колонок и их чекбоксов в «Поля списка» (комиссии, «К выплате», «Статус возврата»; см. раздел 3). На тело ответа getPayments не влияет.
  • viewing.all.logs (в справочнике: «Просмотр всех логов») — кнопка «История» для T‑Bank, модальное окно на странице отчёта и доступ к GET /admin/reports/payments/{payment}/tbank-history (middleware can:viewing.all.logs). Без права кнопка не рендерится, эндпоинт отвечает 403. Требуется также reports.view (маршрут внутри той же группы).

7) История платежа T‑Bank (кнопка «История»)

Для строк с провайдером T‑Bank (наличие deal_id / банковского payment_id / payment_status в смысле отчёта) в колонке «Действия» может выводиться вторая кнопка — «История». По нажатию открывается модальное окно и выполняется запрос GET /admin/reports/payments/{id}/tbank-history (ожидается AJAX/JSON).
  • Доступ: только при viewing.all.logs; иначе кнопка и модалка не показываются, запрос к API — 403.
  • Содержимое ответа: объединённый таймлайн событий по платежу — логи статусов оплаты (tinkoff_payment_status_logs, webhook и др.) и логи выплат по связанным tinkoff_payouts (tinkoff_payout_status_logs). Платёж должен принадлежать текущему партнёру (tenant).
  • Связанная документация: общий обзор T‑Bank — tbank.html.
