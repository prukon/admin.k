Отчёты (админка): фильтры/AJAX/колонки/суммы

Файл: /docs/documentation/reports-admin.html · список разделов (/doc)

1) Общие принципы UI/UX

Начиная с 2026‑03, отчёты админки приводятся к единому каркасу (как в “Платежи”):
  • верхняя панель: заголовок + “Общая сумма” + кнопки “Фильтры”/“Колонки”
  • применение фильтров без перезагрузки страницы: preventDefault() → table.ajax.reload()
  • “Общая сумма” обновляется через отдельный endpoint …/total и меняется с анимацией
  • настройки колонок сохраняются для текущего пользователя в user_table_settings (если отчёт поддерживает “Колонки”)

2) Права доступа (middleware can:…)

Важно: разные отчёты находятся в разных группах прав. Без соответствующего permission ответ будет 403 (страница и все AJAX endpoints).
  • reports.view — отчёты в группе /admin/reports/* (Платежи, Задолженности, LTV, Платежи по месяцам).
  • reports.payment.intents.view — отчёт “Платежные запросы”.
  • reports.fiscal.receipts.view — отчёт “Чеки”.
Модель прав в разрезе партнёра: partners-permissions.
  • locations.view — не открывает отчёты само по себе, но даёт фильтр «Локация» в отчётах платежей и задолженностей (см. §2.1).

2.1) Фильтр по локации в отчётах

Единый параметр запроса: filter_location_id. Пустое значение — все записи; none — только без локации; числовой id — только активная локация текущего партнёра (locations.is_enabled = true). Фильтр в UI и на сервере применяется только при праве locations.view: без права блок не рендерится, а переданный в URL/AJAX filter_location_id игнорируется (защита от обхода через query string). Параметр уходит во все связанные запросы: DataTables, …/total, детализацию по месяцу / платежам ученика.
Отчёт Страница Поле фильтрации Смысл Элемент UI (id)
Платежи /admin/reports/payments payments.location_id Опциональный снимок payments.location_id (если был передан при первом create); колонка «Локация» pay-filter-location
Платежи по месяцам /admin/reports/payments/monthly payments.location_id Тот же снимок, что в журнале payments pay-monthly-filter-location
LTV /admin/reports/ltv payments.location_id Агрегация и детализация только по платежам с подходящим снимком pay-ltv-filter-location
Задолженности /admin/reports/debts Группа ученика + location_team Локация через users.team_id: группа привязана к локации или без привязок (доступна везде) pay-debt-filter-location
  • Запись payments.location_id при первой фиксации оплаты — payments.html, колонка и фильтр — reports-payments.html.
  • Старые строки payments без локации: location_id IS NULL попадают в «Без локации».
  • Снимок payments.location_id задаётся только при первом создании платежа (если передан явно), см. payments.html.
  • Привязка группы к локациям: /admin/teams, /admin/locations (pivot location_team). Обзор: Локации ↔ группы.
  • Задолженности — различие none и платежных отчётов: для долгов filter_location_id=none — ученики с users.team_id IS NULL или с «универсальной» группой (нет строк в location_team для этой группы). Для платежей/LTV/monthly none — только строки с payments.location_id IS NULL.
  • Задолженности — конкретная локация L: ученик попадает, если его группа привязана к L в location_team или универсальна; ученик без группы в выборку по L не входит.

2.2) Фильтр по тренеру в отчётах

Единый параметр запроса: filter_trainer_profile_id (id из trainer_profiles текущего партнёра). Логика: из team_trainer берутся team_id выбранного тренера, затем в выборку попадают ученики с users.team_id из этого списка. Если у тренера нет групп — пустой результат. Select2: GET /admin/reports/payments/trainers-search (reports.payments.trainers.search, middleware reports.view). Подробно о модели тренеров и привязках: Тренеры (админка).
Отчёт Страница Элемент UI (label)
Платежи /admin/reports/payments for="pay-filter-trainer">Тренер
Платежи по месяцам /admin/reports/payments/monthly for="pay-monthly-filter-trainer">Тренер
LTV /admin/reports/ltv for="pay-ltv-filter-trainer">Тренер
Задолженности /admin/reports/debts for="pay-debt-filter-trainer">Тренер
  • Блок фильтра в Blade рендерится только при trainers.view ($canViewTrainers в контроллерах отчётов). Без права страница отчёта остаётся доступной при reports.view, но селект «Тренер» не выводится.
  • Параметр filter_trainer_profile_id в AJAX/DataTables и …/total передаётся вместе с остальными фильтрами при применении формы (как у локации).
  • Отличие от фильтра локации (§2.1): на сервере фильтр по тренеру не привязан к проверке trainers.view в query — UI скрыт без права; при ручной подстановке параметра в URL фильтрация может сработать (см. тесты в PaymentReportTrainerFilterFeatureTest).

3) Отчёт “Задолженности”

  • Страница: /admin/reports/debts (middleware can:reports.view)
  • DataTables: GET /admin/reports/getDebts (AJAX)
  • Общая сумма: GET /admin/reports/debts/total (JSON: { total_formatted, total_raw })
Фильтр локации — §2.1: через группу ученика (users.team_id и location_team). Без reports.view страница и все AJAX-эндпоинты отвечают 403.

4) Отчёт “Платежи по ученикам (LTV)”

  • Страница: /admin/reports/ltv (middleware can:reports.view)
  • DataTables: GET /admin/reports/ltv/data (AJAX)
  • Детализация: GET /admin/reports/ltv/{user}/payments (AJAX)
  • Общая сумма: GET /admin/reports/ltv/total (JSON: { total_formatted, total_raw })
“Общая сумма” в LTV — это sum(payments.summ) по тем же фильтрам (и по текущему партнёру), что и таблица. Фильтр локации — §2.1: payments.location_id (опциональный снимок при create); тот же параметр в ltv/data, ltv/total и ltv/{user}/payments.

5) Отчёт “Платежи по месяцам”

  • Страница: /admin/reports/payments/monthly (middleware can:reports.view)
  • DataTables (месяцы): GET /admin/reports/payments/monthly/data (AJAX)
  • Детализация месяца: GET /admin/reports/payments/monthly/{yearMonth}/payments (AJAX)
  • Общая сумма: GET /admin/reports/payments/monthly/total (JSON: { total_formatted, total_raw })
Режим группировки (по дате операции / по месяцу абонемента) в UI хранится в форме/JS и не обязан отражаться в URL. В запросы DataTables/детализации/total передаётся параметр mode (subscription по умолчанию). Фильтр локации — §2.1: payments.location_id; параметр filter_location_id уходит в monthly/data, monthly/total и monthly/{yearMonth}/payments.

6) Отчёт “Платежные запросы” (payment_intents)

  • Страница: /admin/reports/payment-intents (middleware can:reports.payment.intents.view)
  • DataTables: GET /admin/reports/getPaymentIntents (AJAX)
  • Общая сумма: GET /admin/reports/payment-intents/total (JSON: { total_formatted, total_raw })
  • Колонки (настройки):
    • GET /admin/reports/payment-intents/columns-settings
    • POST /admin/reports/payment-intents/columns-settings (columns: {key: true/false})
    • Хранение: user_table_settings.table_key = reports_payment_intents
  • Select2:
    • GET /admin/reports/payment-intents/partners-search → { results: [{id,text}] }
    • GET /admin/reports/payment-intents/users-search → { results: [{id,text}] } (поддерживает partner_id)

7) Отчёт “Чеки” (fiscal_receipts)

  • Страница: /admin/reports/fiscal-receipts (middleware can:reports.fiscal.receipts.view)
  • DataTables: GET /admin/reports/fiscal-receipts/data (AJAX)
  • Общая сумма: GET /admin/reports/fiscal-receipts/total (JSON: { total_formatted, total_raw }, считает sum(fiscal_receipts.amount))
  • Колонки (настройки):
    • GET /admin/reports/fiscal-receipts/columns-settings
    • POST /admin/reports/fiscal-receipts/columns-settings (columns: {key: true/false})
    • Хранение: user_table_settings.table_key = reports_fiscal_receipts
  • Select2: GET /admin/reports/fiscal-receipts/partners-search → { results: [{id,text}] }
