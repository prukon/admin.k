Отчёты (админка): фильтры/AJAX/колонки/суммы

Файл: /docs/documentation/reports-admin.html · список разделов (/doc)

1) Общие принципы UI/UX

Начиная с 2026‑03, отчёты админки приводятся к единому каркасу (как в “Платежи”):
  • верхняя панель: заголовок + “Общая сумма” + кнопки “Фильтры”/“Колонки”
  • применение фильтров без перезагрузки страницы: preventDefault() → dtApi.reload({ keepPage: true }) (пресет KidsCrmDataTable)
  • “Общая сумма” обновляется через отдельный endpoint …/total и меняется с анимацией
  • настройки колонок сохраняются для текущего пользователя в user_table_settings (если отчёт поддерживает “Колонки”)
С 2026‑06 основные таблицы отчётов переведены на пресет KidsCrmDataTable.create (reusable-ui-partials §3): без scrollX, класс dt-columns-managed, dropdown «Колонки» с data-column-key, tooltip'ы через KidsCrmTooltip (глобально в layouts/admin2.blade.php). Модалка «История изменений» (includes/logModal.blade.php) на вкладках отчётов тоже использует KidsCrmDataTable.create('#logsTable', …) (без columns-settings, ленивая инициализация при открытии модалки).

2) Права доступа (middleware can:…)

Важно: разные отчёты находятся в разных группах прав. Без соответствующего permission ответ будет 403 (страница и все AJAX endpoints).
  • reports.view — отчёты в группе /admin/reports/* (Платежи, Задолженности, LTV, Платежи по месяцам).
  • reports.payment.intents.view — отчёт “Платежные запросы”.
  • reports.fiscal.receipts.view — отчёт “Чеки”.
  • reports.emails.view — отчёт “Исходящие письма”.
Модель прав в разрезе партнёра: partners-permissions.
  • locations.view — не открывает отчёты само по себе, но даёт фильтр «Локация» в отчётах платежей и задолженностей (см. §2.1).

2.0) Фильтр «Группа» в отчёте «Платежи»

Параметр filter_team_id — id группы партнёра (Select2, GET /admin/reports/payments/teams-search). Логика отличается от LTV и «Платежи по месяцам»: там фильтр только по членству ученика в pivot (applyReportTeamFilters). В основном отчёте «Платежи» используется UserTeamQuery::applyPaymentLedgerTeamFilters — учитывается снимок payments.team_id с fallback на pivot для legacy-строк (payments.team_id IS NULL). Колонка «Группа» для новых месячных оплат показывает оплаченную группу, не все группы ученика. Подробно: reports-payments §1.1, payments §2.2.

2.1) Фильтр по локации в отчётах

Единый параметр запроса: filter_location_id. Пустое значение — все записи; none — только без локации; числовой id — только активная локация текущего партнёра (locations.is_enabled = true). Фильтр в UI и на сервере применяется только при праве locations.view: без права блок не рендерится, а переданный в URL/AJAX filter_location_id игнорируется (защита от обхода через query string). Параметр уходит во все связанные запросы: DataTables, …/total, детализацию по месяцу / платежам ученика.

2.2) Фильтр по тренеру в отчётах

Единый параметр запроса: filter_trainer_profile_id (id из trainer_profiles текущего партнёра). Логика: из team_trainer берутся team_id выбранного тренера, затем в выборку попадают ученики, состоящие хотя бы в одной из этих групп (pivot team_user, UserTeamQuery::applyReportTrainerTeamFilter). Если у тренера нет групп — пустой результат. Select2: GET /admin/reports/payments/trainers-search (reports.payments.trainers.search, middleware reports.view). Подробно о модели тренеров и привязках: Тренеры (админка).

2.3) Фильтр «Активность ученика» в отчётах

Единый параметр запроса: status — фильтрация по полю users.is_enabled ученика (роль «user» в контексте платежей/долгов/назначений абонементов). Параметр уходит во все связанные запросы: DataTables, …/total, детализацию (LTV — платежи ученика; «Платежи по месяцам» — платежи месяца); на вкладке «Назначение абонементов» — только GET …/assignments/data. Отдельного permission для блока нет: при reports.view фильтр всегда виден (в отличие от локации §2.1 и тренера §2.2). В панели фильтров — последним перед кнопками «Применить» / «Сброс».

inactive «Только неактивные» users.is_enabled = 0

  • JS: параметр status добавляется в объект фильтров при AJAX (…FilterParams()); кнопка «Сброс» возвращает значение active (не «Все ученики»). Значение active не считается «нестандартным» фильтром — панель фильтров по умолчанию свёрнута.
  • LTV — не путать с колонкой «Статус»: колонка is_enabled (badge «Активен» / «Отключен») показывает статус ученика в строке таблицы; фильтр «Активность ученика» задаёт, каких учеников включать в выборку.
  • Не путать с отчётом «Payment intents»: там колонка/ключ status — статус платёжного намерения провайдера; параметр фильтра активности ученика используется только в отчётах Платежи / monthly / LTV / Задолженности.
  • Задолженности: фильтр применяется и к месячным (users_prices), и к периодным (user_custom_payment) источникам через общий join users.
  • Доступ: без reports.view страница и все endpoints (в т.ч. с status в query) — 403; гость — редирект / 401 / 403. На вкладке «Назначение абонементов» — право lessonPackages.view (см. lesson-packages §assignments-tab).
  • Подробнее про отчёт «Платежи»: reports-payments §2.2.

3) Отчёт “Задолженности”

  • Страница: /admin/reports/debts (middleware can:reports.view)
  • DataTables: GET /admin/reports/getDebts (AJAX, server-side; UI — KidsCrmDataTable.create('#debts-table'))
  • Общая сумма: GET /admin/reports/debts/total (JSON: { total_formatted, total_raw })
  • Источники строк:
    • Месячные — неоплаченные записи users_prices (is_paid=0, price>0, месяц < текущего). Каждая строка привязана к users_prices.team_id — у ученика в двух группах два долга за один календарный месяц возможны как две строки.
    • Периодные — неоплаченные user_custom_payment (доп. платежи по датам).
  • Колонки (настройки):
    • GET /admin/reports/debts/columns-settings (reports.debts.columns-settings.get)
    • POST /admin/reports/debts/columns-settings (reports.debts.columns-settings.save, тело: columns: {key: true/false})
    • Хранение: user_table_settings.table_key = reports_debts
    • Ключи колонок по умолчанию: user_name, month, price
Фильтр «Группа» (filter_team_id): для месячных долгов — users_prices.team_id = …; для периодных доп. платежей — ученик состоит в группе (pivot). Текстовый team_title — по любой группе ученика в pivot.
Фильтр локации — §2.1: через группы ученика в pivot (team_user → teams.location_id), не по users_prices.team_id. Платёжные отчёты — payments.location_id.
Фильтр «Активность ученика» — §2.3: параметр status; по умолчанию только активные (users.is_enabled = 1).
Без reports.view страница и все AJAX-эндпоинты отвечают 403.

4) Отчёт “Платежи по ученикам (LTV)”

  • Страница: /admin/reports/ltv (middleware can:reports.view)
  • DataTables: GET /admin/reports/ltv/data (AJAX; UI — KidsCrmDataTable.create('#ltv-table'))
  • Детализация: GET /admin/reports/ltv/{user}/payments (AJAX, только с заголовком X-Requested-With: XMLHttpRequest)
  • Общая сумма: GET /admin/reports/ltv/total (JSON: { total_formatted, total_raw })
  • Колонки (настройки):
    • GET /admin/reports/ltv/columns-settings (reports.ltv.columns-settings.get)
    • POST /admin/reports/ltv/columns-settings (reports.ltv.columns-settings.save)
    • Хранение: user_table_settings.table_key = reports_ltv
    • Ключи: user_name, team_title, total_price, payment_count, first_payment_date, last_payment_date, is_enabled
Детализация LTV — два формата ответа (один endpoint, различие по наличию параметра draw):
  • Без draw (legacy, раскрытие строки): JSON { user_id, payments: […] } — массив всех платежей ученика.
  • С draw (server-side DataTables во вложенной таблице): стандартный ответ DataTables (draw, recordsTotal, recordsFiltered, data[]) плюс мета-поля meta_payments_count и meta_sum_total (счётчик и сумма по тем же фильтрам, что и детализация).
Вложенная таблица платежей инициализируется через обычный $('#…').DataTable({ serverSide: true, … }), не через KidsCrmDataTable.create (намеренно: нет dropdown «Колонки» у дочерней таблицы).
“Общая сумма” в LTV — это sum(payments.summ) по тем же фильтрам (и по текущему партнёру), что и таблица. Фильтр локации — §2.1: payments.location_id (опциональный снимок при create); тот же параметр в ltv/data, ltv/total и ltv/{user}/payments. Фильтр «Активность ученика» — §2.3: status в тех же endpoints; при status=active детализация неактивного ученика возвращает пустой набор / нулевую сумму.
Колонка chevron (раскрытие детализации): единственная колонка с type: 'custom' — details-control (кнопка «+» / «−»). Остальные колонки — пресет (text, money, count, datetime, badge). См. reusable-ui-partials §3.1.

5) Отчёт “Платежи по месяцам”

  • Страница: /admin/reports/payments/monthly (middleware can:reports.view)
  • DataTables (месяцы): GET /admin/reports/payments/monthly/data (AJAX; UI — KidsCrmDataTable.create('#payments-monthly-table'))
  • Детализация месяца: GET /admin/reports/payments/monthly/{yearMonth}/payments (AJAX, yearMonth = YYYY-MM)
  • Общая сумма: GET /admin/reports/payments/monthly/total (JSON: { total_formatted, total_raw })
  • Колонки (настройки):
    • GET /admin/reports/payments/monthly/columns-settings (reports.payments.monthly.columns-settings.get)
    • POST /admin/reports/payments/monthly/columns-settings (reports.payments.monthly.columns-settings.save)
    • Хранение: user_table_settings.table_key = reports_payments_monthly
    • Ключи: month_title, payments_count, total_sum
Детализация месяца — аналогично LTV (§4): без draw → { month_key, mode, payments: […] }; с draw → server-side DataTables + meta_payments_count, meta_sum_total. Вложенная таблица — обычный DataTable(), не KidsCrmDataTable.
Режим группировки (по дате операции / по месяцу абонемента) в UI хранится в форме/JS и не обязан отражаться в URL. В запросы DataTables/детализации/total передаётся параметр mode (subscription по умолчанию). Фильтр локации — §2.1: payments.location_id; параметр filter_location_id уходит в monthly/data, monthly/total и monthly/{yearMonth}/payments. Фильтр «Активность ученика» — §2.3: status в тех же endpoints.
Как в LTV (§4): первая колонка details-control — пока type: 'custom'; данные месяца — text, count, money.
Отчёт «Платежи» (/admin/reports/payments) — полностью без custom: чек — icon, возврат — actions + поля refund_* в JSON. Подробно: reports-payments.

  • Колонки (настройки):

Поле payment_date в БД и в UI
  • В таблице payment_intents.payment_date — строка (VARCHAR), не datetime. Исторически туда пишется метка для Robokassa Shp_paymentDate: YYYY-MM-01 для месячного абонемента, Клубный взнос, Дополнительный платеж и т.д. (см. payments.html).
  • Фактическая дата/время успешной оплаты — колонка «Дата оплаты» (paid_at, тип datetime в UI).
  • Для отчёта PaymentIntentReportController::formatPaymentDateForReport в JSON отдаёт:
    • 2026-06-01 или 2026-06-06 13:17:07 → июнь 2026 (месяц+год по дате);
    • текстовые метки без изменений;
    • пусто → пустая строка (в UI — прочерк через KidsCrmTooltip).
  • Текстовые колонки (partner_title, user_name, …) в rawColumns — Yajra не экранирует их в JSON; на фронте KidsCrmTooltip.normalizeDisplayText снимает остаточные сущности перед отображением.
Клиентский контекст (поля client_* в строке DataTables): заполняются при создании intent через PaymentIntentClientContext::fromRequest. В JSON дополнительно отдаются вычисляемые client_os и client_browser (family + version). Модалка просмотра длинного текста: #paymentIntentMetaModal (общая для User-Agent и meta).

8) Отчёт “Исходящие письма” (outgoing_emails)

Журнал исходящих писем CRM по текущему партнёру. С 2026‑07 — модалка просмотра без ухода со списка, колонки «Создано»/«Отправлено» в формате datetime, кликабельная колонка «Кому», HTML-тело в iframe (не экранированный исходник). Welcome-письма ученикам (ClientWelcomeCredentialsMail) попадают сюда после конвертации лида и resend — см. admin-users §5.1.
  • Страница: /admin/reports/emails (вкладка activeTab=emails в admin/report/index.blade.php; middleware can:reports.emails.view)
  • DataTables: GET /admin/reports/emails/data — только AJAX (X-Requested-With или $request->ajax()); без AJAX → 404. UI: KidsCrmDataTable.create('#emails-table')
  • Счётчики: GET /admin/reports/emails/total (JSON: total_*, sent_*, failed_* по current_partner и тем же фильтрам, что таблица)
  • Источник записей: листенер App\Listeners\LogOutgoingEmail на MessageSending / MessageSent; partner_id — из заголовка X-Partner-Id на Mailable (приоритет) или из PartnerContext в web-запросе. Таблица БД: outgoing_email_logs.
  • Select2: GET /admin/reports/emails/mailable-classes-search → { results: [{id, text}] } (уникальные классы Mailable/Notification партнёра)
  • Карточка письма: GET /admin/reports/emails/{log} — см. §8.3

8.1) Фильтры и тулбар

  • Форма #emails-report-filters (GET /admin/reports/emails): при submit — preventDefault(), dtApi.reload() + обновление счётчиков через …/total (без полной перезагрузки страницы).
  • Поля фильтра (query string, передаются и в …/data, и в …/total):
    • created_at_from, created_at_to — дата создания записи лога
    • sent_at_from, sent_at_to — дата успешной отправки
    • status[] — multiselect Select2: sending, sent, failed
    • mailable_class — Select2 AJAX по mailable-classes-search
    • q — поиск по теме, to_summary, from_address, error_message

8.3) Просмотр письма: модалка и отдельная страница

По умолчанию просмотр — в модалке на странице списка (колонка «Кому», кнопка «Открыть»). Прямой URL и ссылка «Открыть на отдельной странице» в footer модалки — полная HTML-страница с layout.
  • Модалка: #outgoingEmailShowModal в outgoing_emails.blade.php; JS openOutgoingEmailShowModal(showUrl):
    • GET {showUrl}?modal=1
    • заголовки: X-Requested-With: XMLHttpRequest, Accept: text/html
    • ответ — HTML-фрагмент partial admin/report/partials/outgoing_email_show_content.blade.php (без layout)
  • Отдельная страница: GET /admin/reports/emails/{log} без modal=1 и без AJAX → view admin.report.outgoing_email_show (layout + partial).
  • Контент карточки: метаданные (даты, from/to/cc/bcc, тема, класс Mailable, очередь, попытки, ошибка), вкладки «HTML» / «Текст».
  • HTML-тело: iframe с src="data:text/html;charset=utf-8;base64,…" (base64_encode($log->html_body)) — рендер письма в изолированном фрейме; не srcdoc="{{ e(...) }}" (иначе виден экранированный исходник).
  • «Кому» в карточке: массив to_addresses (структура [{address, name}]); to_summary используется в таблице и поиске.
  • Footer модалки: ссылка «Открыть на отдельной странице» (#outgoingEmailShowModalOpenPage, target=_blank).

8.5) columns-settings

  • GET /admin/reports/emails/columns-settings → JSON-объект { column_key: bool, … }
  • POST /admin/reports/emails/columns-settings — body { columns: { key: true/false, … } }; валидация: columns обязателен и должен быть массивом
  • Хранение: user_table_settings.table_key = reports_outgoing_emails
  • Успех: { "success": true } (и для AJAX, и для non-AJAX POST — endpoint API-подобный, redirect на index не делается)
  • Ошибка валидации: AJAX / postJson → 422 JSON; non-AJAX POST без X-Requested-With → 302 redirect back + errors.columns (не пустой 200)
