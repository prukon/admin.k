Районы (админка): /admin/districts

Файл: /docs/documentation/admin-districts.html · обзор: directories-hierarchy · объекты: admin-locations · лендинг: school-leads-landing
Глоссарий UI: в интерфейсе «Район» = модель District, «Объект» = модель Location.
Пункт сайдбара справочников (виден при districts.view, locations.view, groups.view или sport_types.view): при единственном districts.view подпись «Районы», при 2+ правах — «Справочники» (DirectoriesMenu). Вкладка «Районы» — третья: Группы → Объекты → Районы → Виды спорта ($activeTab = 'districts'). URL пункта меню — первая доступная вкладка по приоритету: группы → объекты → районы → виды спорта.
Все маршруты раздела — под middleware('can:districts.view'). Право districts.view покрывает просмотр и CRUD (отдельного manage нет).

2) Модель и БД

  • Таблица districts: partner_id, name, is_enabled, sort_order.
  • Уникальность: unique(partner_id, name) — после удаления имя можно использовать снова.
  • Связь: District hasMany Location (locations.district_id nullable).

3) Интерфейс

View: resources/views/admin/districts/index.blade.php, вкладки: admin/directories/_section_tabs.blade.php.

  • Поля формы: name, is_enabled, sort_order.
  • Колонка «Объектов» — locations_count (withCount).
  • Фильтры: название/id, статус active/inactive.
  • Сортировка по умолчанию: sort_order, затем name.

4) Удаление

  • Если к району привязан хотя бы один объект (locations.district_id, включая отключённые) — 422: «Нельзя удалить район: к нему привязаны объекты».
  • Без объектов — удаление из БД (hard delete). У заявок school_leads.district_id обнуляется (nullOnDelete).

5) Лендинг и заявки

  • На лендинге родитель выбирает район → объект → услугу (см. school-leads-landing).
  • Объекты без district_id на лендинге не показываются.
  • В school_leads сохраняются и district_id, и location_id (история заявки).
  • В CRM заявок: колонка/фильтр «Район» при districts.view; «Объект» — при locations.view.
