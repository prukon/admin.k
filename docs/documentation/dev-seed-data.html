Dev-фикстуры: SEED_DEV_DATA и цепочка DatabaseSeeder

Файл: /docs/documentation/dev-seed-data.html · права admin: partners-permissions · юр. лица: admin-legal-entities · T‑Bank: tbank · платёжные системы: settings-payment-systems
Только dev/test. Блок dev-сидеров в DatabaseSeeder выполняется только при SEED_DEV_DATA=true в .env. На production этот флаг должен быть false или отсутствовать. Сидеры не вызывают API T‑Bank и не выполняют sm-register — ShopCode в фикстурах задаётся строками вида SHOP-DEV-* для ручной проверки UI и резолверов.

1) Включение и запуск

# .env (локальная / тестовая БД)
SEED_DEV_DATA=true

# Полный прогон (миграции + prod-сидеры + dev-блок)
php artisan migrate:fresh --seed

# Один dev-сидер (тоже требует SEED_DEV_DATA=true)
php artisan db:seed --class=Database\\Seeders\\DevPartnerLegalEntitiesSeeder

Trait Database\Seeders\Concerns\GuardsDevSeedData: метод abortUnlessDevSeedEnabled() — при false сидер с trait молча завершается (без записи в БД). Используют все Dev*-сидеры, включая DevPartnersSeeder, DevPaymentSystemsSeeder, DevPricesSeeder, DevAdminsSeeder.

2) Порядок в DatabaseSeeder

Сначала всегда идут «prod»-справочники (дни недели, роли, permissions, соцсети, статусы занятий). Dev-блок — только при флаге:

#СидерНазначение
1DevPartnersSeeder3 партнёра (factory), ожидаются id 1–3 на чистой БД
2DevPartnerLegalEntitiesSeeder5 юр. лиц (id 1–5), multi-entity у партнёров 2 и 3; обнуляет partners.tinkoff_partner_id
3DevSportTypesSeeder9 видов спорта (id 1–9, по 3 на партнёра)
4DevDistrictsSeeder9 районов (id 1–9)
5DevScheduleStatusesSeederстатусы посещаемости журнала /schedule
6–8Dev*RoleBasePermissionsSeederсинхронизация permission_role для user/admin/trainer по config/role_base_permissions.php
9DevTeamsSeeder20 групп; trait AssignsTeamsToLegalEntities, AssignsTeamDirectoryLinks
10DevLocationsSeederобъекты с district_id, привязка к группам
11–14DevAdminsSeeder, DevUsersSeeder, DevTrainersSeeder, DevTrainerSalaryDefaultsSeederперсонал и ученики
15DevScheduleJournalSeederжурнал посещаемости за ~6 мес.
16DevSchoolLeadsSeederзаявки с district_id, sport_type_id, объект по району
17–19DevLessonPackagesSeeder, DevSchoolScheduleSeeder, DevLessonPackageAssignmentsSeederабонементы, слоты, назначения с user_lesson_packages.team_id

22DevTbankHistorySeederистория T‑Bank / выплат / чеков (см. §4)

Не входит в dev-seed: клиентские договоры (contracts) — намеренно не создаются фабриками/сидерами dev-слоя.

3) Юр. лица и группы в фикстурах

  • DevPartnerLegalEntitiesSeeder — фиксированные id 1–5, ShopCode SHOP-DEV-1 … SHOP-DEV-3-2; у каждого multi-entity партнёра ровно одно is_default=true.
  • DevTeamsSeeder — после создания групп вызывает assignLegalEntitiesToAllTeams(): у каждой группы явный legal_entity_id своего партнёра (для multi-entity — распределение по юр. лицам партнёра).
  • assignSportTypesToAllTeams() и assignLocationsToAllTeams() — у групп заполнены sport_type_id и location_id из справочников партнёра.
  • Legacy partners.tinkoff_partner_id в dev-данных всегда null; PartnerFactory по умолчанию тоже не генерирует legacy ShopCode (state legacyTinkoffPartnerId() — только для тестов legacy).

Поведение резолвера и multi-entity guard в runtime — admin-legal-entities §1.1, payments §2.3.5.

4) T‑Bank: комиссии и история

4.1) DevTinkoffCommissionRulesSeeder

9 записей в tinkoff_commission_rules (id 1–9): для каждого dev-партнёра 1–3 — правила card, sbp, method=null. У партнёра 1 правило card: auto_payout_enabled=true, auto_payout_delay_hours=2 (для проверки UI автовыплат; cron на test-стенде обычно не запущен).

4.2) DevTbankHistorySeeder

  • До 40 записей из оплаченных payment_intents (provider tbank, status paid) + детерминированные standalone-платежи dev-seed-tp-{partner}-{slot}.
  • У каждого tinkoff_payments заполнен legal_entity_id (через LegalEntityResolver / payable / группу).
  • Выплаты только в финальных статусах COMPLETED / REJECTED — статус INITIATED не создаётся (без риска для cron).
  • Фiscal receipts с idempotency keys; повторный прогон идемпотентен по order_id.

Админка комиссий и автовыплата — tbank §5.

5) DevPaymentSystemsSeeder

  • Robokassa — детерминированно для партнёров 1 и 2 (не случайный партнёр).
  • T‑Bank — одна глобальная запись payment_systems(partner_id=NULL, name=tbank, test_mode=true); повторный прогон не дублирует.

Архитектура платёжных систем — settings-payment-systems §4.

6) Назначения абонементов и team_id

DevLessonPackageAssignmentsSeeder создаёт назначения с явным user_lesson_packages.team_id:

  • для multi-entity партнёров ученики выбираются в разрезе юр. лица;
  • слоты расписания фильтруются по группе;
  • валидация teamBelongsToPartner() перед записью.

Runtime и UI колонки «Группа» — lesson-packages § назначения.

7) Базовые права admin в dev

DevAdminRoleBasePermissionsSeeder проставляет для роли admin на всех партнёрах права из config/role_base_permissions.php, в том числе (с 2026‑07):

  • sport_types.view — просмотр справочника видов спорта (без sport_types.manage);
  • legal_entities.view, legal_entities.manage — CRUD юр. лиц в CRM;
  • не включает legal_entities.sm_register — регистрация в T‑Bank только при явной выдаче права.

Полный список и исключения — partners-permissions §7.

8) Демо-партнёр «Исток»

DevIstokMenuSeeder (последний в цепочке) обновляет partner id=1 («Школа футбола "Исток"»), объект «Стадион Исток», 5 групп с legal_entity_id=1, sport_type_id, location_id и демо-меню. Не перезаписывает legacy ShopCode на partners — только tinkoff_partner_id=null.

9) Автотесты

  • tests/Feature/Database/DevSeedStackFeatureTest.php — guard SEED_DEV_DATA, целостность цепочки (юр. лица, справочники, комиссии, payment systems, history, права admin).
  • HTTP-разделы справочников — SportTypes*FeatureTest, Districts*FeatureTest, LegalEntities*FeatureTest (см. tests-standards).
php artisan test --env=testing tests/Feature/Database/DevSeedStackFeatureTest.php

10) Ограничения и типичные ошибки

  • Сидеры с фиксированными id партнёров 1–3 пропускаются, если в БД нет ровно трёх партнёров с этими id (warning в консоль).
  • DevPricesSeeder при повторном db:seed без migrate:fresh может дублировать начисления — используйте fresh или точечные сидеры.
  • На test-стенде без cron безопасны dev-выплаты в статусах COMPLETED/REJECTED; INITIATED в dev-истории не создаются намеренно.
  • Feature-тесты dev-сидеров сбрасывают AUTO_INCREMENT таблицы partners между кейсами (MySQL + транзакционный rollback).
