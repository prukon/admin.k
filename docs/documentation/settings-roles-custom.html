Настройки: кастомные роли и стартовый набор прав

Файл: /docs/documentation/settings-roles-custom.html · см. также партнёры и базовые роли user/admin
Назначение. В админке на вкладке «Настройки → Права и роли» можно создать новую несистемную роль для текущего партнёра. При создании ей автоматически назначается стартовый набор permissions по списку roles.admin из конфига /config/role_base_permissions.php — в том же смысле «как у администратора по умолчанию», но только для новой роли и только в контексте partner_id (см. ниже).

1) Где в интерфейсе и маршруты

  • Страница: GET /admin/settings/rules (внутри общей страницы настроек, вкладка «Права и роли»).
  • Создание роли: POST /admin/setting/role/create (AJAX, имя маршрута admin.setting.role.create).
  • Переключение одного права у роли: POST /admin/setting/rule/toggle.
  • Удаление кастомной роли: DELETE /admin/setting/role/delete.
  • Логи вкладки (DataTables): GET /admin/setting/rules/logs-data.
Системную роль через эту форму создать нельзя: в UI создаётся обычная роль с is_sistem = 0.

2) Доступ (авторизация)

  • Все перечисленные маршруты обёрнуты в middleware can:settings.roles.view (см. /routes/web.php).
  • Право settings.roles.view входит в базовый набор для роли admin в role_base_permissions.php и проставляется в разрезе партнёра через permission_role, как описано в документе про партнёров и базовые роли.
  • Пользователь с ролью superadmin проходит проверки Gate без ограничений (см. Gate::before в AuthServiceProvider).

3) Как назначаются права при создании роли

  • Источник имён: config('role_base_permissions.roles.admin') — тот же массив, что используется для базовой роли admin при сидировании (AdminRoleBasePermissionsSeeder).
  • Записи создаются в permission_role с полями role_id (новая роль), permission_id, partner_id (текущий партнёр из контекста). Так же, как при ручном включении чекбокса на той же странице.
  • Вставка идемпотентна: используется insertOrIgnore, уникальность по тройке (partner_id, role_id, permission_id) (см. раздел про UNIQUE в документе по партнёрам).

3.1 Отличия от сидеров и от создания партнёра

  • Сидеры (AdminRoleBasePermissionsSeeder): при отсутствии в БД хотя бы одного имени из конфига сидер падает с ошибкой (fail-fast).
  • Создание партнёра: при создании записи партнёра также fail-fast, если в справочнике нет нужных permissions.
  • Создание кастомной роли в UI: если имя из конфига не найдено в таблице permissions, оно пропускается, создание роли не отменяется; в Laravel-лог пишется предупреждение (Log::warning) с перечислением пропущенных имён. Отдельные записи в my_logs на массовое назначение прав не создаются.

4) Реализация в коде

  • Контроллер: /app/Http/Controllers/Admin/Setting/RuleController.php, метод createRole.
  • Сервис: /app/Services/Roles/AssignPartnerRolePermissionsFromConfig.php — разбор конфига, сопоставление с permissions, предупреждение по пропускам, вставка в permission_role.
  • Разметка и скрипт вкладки: /resources/views/admin/setting/rule.blade.php.

5) Ответ API при создании роли

JSON содержит поля success, role (модель роли) и permission_ids — список идентификаторов permissions, реально назначенных для текущего партнёра. Клиент обновляет таблицу чекбоксов и список ролей в модальном окне без полной перезагрузки страницы (модал успеха без location.reload()).
