Пользователи в админке

Файл: /docs/documentation/admin-users.html · URL в CRM: /docs/documentation/admin-users · оглавление: /doc

1) Назначение и доступ

Страница списка: GET /admin/users (маршрут admin.user1, UserController@index), view resources/views/admin/user.blade.php.
Все маршруты раздела обёрнуты в middleware('can:users.view') (routes/web.php). Без users.view страница и AJAX раздела отвечают 403, даже если у роли есть locations.view.
Внутренняя документация (/doc) требует отдельное право documentations.view — это не доступ к /admin/users.

Основные endpoint’ы раздела

Модалки create/edit отдают и принимают JSON только при AJAX-запросе (X-Requested-With: XMLHttpRequest / ожидание JSON), иначе store может завершиться без JSON-тела.

2) Список пользователей (DataTables)

  • Фильтры (кнопка «Найти»): name, team_id (none = без группы), status (active / inactive).
  • Фильтр «Локация» — только при locations.view (см. §3). Параметр: location_id (none = без локации, id = активная локация партнёра). Без права параметр в запросе игнорируется.
  • Колонки настраиваются через «Поля списка»; настройки per user в user_table_settings.

Индексы колонок DataTables (фиксированные)

Колонка «Локация» всегда присутствует в разметке таблицы (стабильные индексы); без locations.view она принудительно скрыта и не попадает в UI настроек.

3) Локация пользователя (users.location_id)

Необязательная привязка учётной записи к справочнику locations текущего партнёра. Миграция: database/migrations/2026_05_19_120000_add_location_id_to_users_table.php.
Поле: users.location_id (nullable, FK → locations.id, ON DELETE SET NULL). Модели: User::location(), Location::users().
Один пользователь — не более одной локации (в отличие от many-to-many). Справочник локаций ведётся на GET /admin/locations (права locations.view / locations.manage) — отдельно от привязки к ученику.
Не путать с расписанием: слоты team_schedule_slots имеют свой location_id. Поле users.location_id — справочная привязка ученика в карточке; автоматической синхронизации со слотами расписания нет.

3.1) Права

ПравоЧто даёт в разделе «Пользователи»
users.view Доступ к странице и всем endpoint’ам раздела (обязательно). Без него — 403.
locations.view Селект локации в модалках create/edit; фильтр «Локация»; столбец и ключ location в columns-settings; чтение/запись location_id в store/update; поле location в JSON /admin/users/data.
locations.manage Не требуется для привязки ученика к локации — только для CRUD справочника на /admin/locations.
users.group.update Только поле «Группа» при редактировании (аналогия: для локации отдельного users.location.update нет).

Без locations.view: входящий location_id удаляется в StoreRequest / UpdateRequest; в ответе data поле location — пустая строка; фильтр по локации на сервере не применяется.

3.2) UI

  • Create — #create-location, опции: «Без локации» + активные локации партнёра (is_enabled = true).
  • Edit — #edit-location: те же активные + текущая локация, если она отключена (в JS подпись «(отключена)», опция добавляется динамически).
  • Фильтр — #filter-location: пусто / none / id активной локации.
  • Столбец — после «Группа»; по умолчанию виден при первом заходе, если есть locations.view.

В index во view всегда передаётся $activeLocations (активные локации партнёра) — для фильтра и селектов; без права блоки в Blade не рендерятся (@can('locations.view')).

3.3) Валидация и сохранение

ОперацияRequestПравила
Создание StoreRequest При locations.view: location_id nullable, exists:locations,id с partner_id текущего партнёра и is_enabled = true. Пустая строка → null. Ошибки — ключ location_id (вывод под полем в модалке).
Обновление UpdateRequest При locations.view: location_id nullable integer. В withValidator: локация того же партнёра; если is_enabled = false — разрешена только если уже назначена этому пользователю.

3.4) API: data и edit

  • GET /admin/users/data — в каждой строке при locations.view: "location": "Название" (из связи user.location), иначе "".
  • GET /admin/users/{id}/edit (AJAX) — в user: location_id; объект location: { id, name, is_enabled } или null.
  • Изменение локации пишется в пользовательский лог (action 22) строкой «Локация: … → …», если у актора есть locations.view.

3.5) Использование в отчётах

Поле users.location_id задаётся в этом разделе и используется, в частности, в отчёте «Задолженности» (фильтр по текущей привязке ученика). В отчётах по оплатам используется снимок payments.location_id на момент платежа — см. reports-payments.html и payments.html. Смена users.location_id не пересчитывает старые строки payments.

4) Создание пользователя (POST /admin/users)

Маршрут admin.user.store. Валидация: App\Http\Requests\User\StoreRequest. Создание через UserService::store() в DB::transaction с логом (action 21) и опциональными user_field_values.

Пароль

Поле password необязательно. Пустая строка → password = NULL в БД. Если указан: min:8 / max:255; хеш — каст hashed модели User.

Телефон

Поле phone с маской +7 (999) 999-99-99 (Inputmask). Хранение: канон +7XXXXXXXXXX в users.phone. Право на запись: users.phone.update (без права поле игнорируется на бэкенде и disabled в UI).

Дополнительные поля (custom)

В модалке создания — те же доп. поля партнёра, что в edit (фильтр по роли админа, payload fields в JSON edit).

ТемаПоведение
Телоcustom[{slug}] — значения для user_field_values
Валидацияcustom.* nullable string; slug должен существовать у UserField текущего партнёра
ЗаписьТолько поля с editable для роли актора

custom не передаётся в User::create — извлекается до UserService::store().

5) Редактирование (PATCH /admin/users/{user})

Валидация: UpdateRequest. Поля с отдельными правами: имя (users.name.update), группа (users.group.update), email, телефон, роль, активность, дата рождения, дата начала — см. правила в Request. Доп. поля: custom[{slug}] с той же логикой партнёра и editable, что при создании. Локация — §3 (locations.view).
