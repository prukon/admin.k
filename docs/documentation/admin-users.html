Пользователи в админке

Файл: /docs/documentation/admin-users.html · URL в CRM: /docs/documentation/admin-users · оглавление: /doc

1) Назначение и доступ

Страница списка: GET /admin/users (маршрут admin.user1, UserController@index), view resources/views/admin/user.blade.php.
Все маршруты раздела обёрнуты в middleware('can:users.view') (routes/web.php). Без users.view страница и AJAX раздела отвечают 403, даже если у роли есть locations.view.
Внутренняя документация (/doc) требует отдельное право documentations.view — это не доступ к /admin/users.

Основные endpoint’ы раздела

МетодURLНазначение

Модалки create/edit отдают и принимают JSON только при AJAX-запросе (X-Requested-With: XMLHttpRequest / ожидание JSON), иначе store может завершиться без JSON-тела.

2) Список пользователей (DataTables)

  • Фильтры (кнопка «Найти»): name — ищет по ФИО ученика, ФИО родителя в parents (whereHas parentProfile), email, телефону, дате рождения; team_id (none = без группы); status (active / inactive).
  • Фильтр «Локация» — только при locations.view (см. §3). Параметр: location_id (none = без локации, id = активная локация партнёра). Без права параметр в запросе игнорируется.
  • Колонки настраиваются через «Поля списка»; настройки per user в user_table_settings (ключ parent для колонки «Родитель»).

Индексы колонок DataTables (фиксированные)

Колонка «Локация» всегда присутствует в разметке таблицы (стабильные индексы); без locations.view она принудительно скрыта и не попадает в UI настроек.

2.1) Данные родителя (справочник parents)

ФИО родителя / подписанта договора хранится в таблице parents (модель ParentProfile). Ученик связывается через users.parent_id. Один родитель — несколько детей (1:N). Полный обзор (семейный кабинет, личный кабинет, изоляция): parents-and-family-cabinet.

2.1.1) Поля в БД

Таблица / полеОписание
parents.lastname, firstname, middlenameФИО (nullable, max 100)
parents.partner_idОрганизация (обязательно)
users.parent_idFK на родителя ученика; общий parent_id у детей — переключатель в кабинете
parent_full_name (аксессор User)ФИО из parentProfile для списков и договоров
users.phone, users.emailКонтакты для SMS/оплат на учётке ученика, не дублируются в parents

В GET /admin/users/data: "parent": "Фамилия Имя Отчество" из связи parentProfile. Сортировка col=3: join parents, orderBy parents.lastname, parents.firstname.

2.1.2) UI и поиск (модалки create / edit)

Partial: resources/views/admin/users/_parent_form.blade.php (подключается в createUser / editUser).

Видимость блока. Секция «Данные родителя» показывается только если в модалке выбрана роль user (ученик). При другой роли блок скрывается, поля parent_* отключаются и не уходят в сохранение. На сервере StudentParentSyncService для не-user сбрасывает users.parent_id в null.
  • Если у партнёра есть хотя бы один родитель в parents — переключатель режима (две кнопки: «Из справочника» / «Новый родитель»), затем либо Select2 parent_id, либо три поля ФИО (взаимоисключающие панели).
  • Если справочник пуст — переключатель и Select2 не рендерятся; сразу только поля ФИО нового родителя.
  • GET /admin/users/parents/search?q=... (и опционально id=) — JSON для Select2; только parents.partner_id = current.
  • Выбор из справочника подставляет ФИО в скрытые поля формы (для PATCH/POST); режим «Новый родитель» очищает parent_id и показывает пустые поля ФИО.
  • Отвязка: очистка Select2 + пустые parent_* в запросе → parent_id = null (см. isAdminParentBlockCleared в сервисе).
  • В «Поля списка» — колонка «Родитель» (data-column-key="parent").
  • Право на раздел: users.view (отдельного permission на справочник родителей нет).

2.1.4) Валидация, сервис, логи

Trait ValidatesStudentParent в StoreRequest / UpdateRequest:

  • parent_id — nullable, exists в parents с partner_id = current.
  • parent_lastname, parent_firstname, parent_middlename — ключи HTTP-формы (не колонки users), nullable|string|max:100.

Сохранение: StudentParentSyncService::syncForStudent из UserService::update/store. При обновлении вызывается, если в теле есть ключи parent_* или у пользователя после сохранения роль не user (сброс parent_id).

JSON edit: user.parent_lastname и т.д. — из User::parentFormFields() (значения из parents).

2.1.5) Связь с договорами и личным кабинетом

Договоры: parent_full_name и предзаполнение подписанта — см. contracts §2–3.
Личный кабинет ученика: блок «Данные родителя» (общая карточка parents для всех детей с тем же parent_id), право account.user.parent.update — см. parents-and-family-cabinet §4.

2.2) Сведения об ученике (мед./особенности)

Три необязательных флага на учётной записи ученика (roles.name = user): индивидуальные особенности, учёт у медспециалистов, наличие инвалидности. Хранятся в таблице users (не отдельная таблица, не user_fields).
Миграция: database/migrations/2026_05_28_140000_add_student_health_flags_to_users_table.php.
Право на чтение и запись в админке: users.other.update (отдельного права «только просмотр» нет — без него блок в UI не рендерится). Доступ к странице раздела по-прежнему только через users.view (§1).

2.2.1) Поля в БД

Колонка users.*Смысл в UIТип
is_individual_traitsИндивидуальные особенности воспитанника (физические, психологические)boolean NULL
is_on_medical_registerСостоит на учёте у медицинских специалистовboolean NULL
is_with_disabilityНаличие инвалидностиboolean NULL

Семантика значений: NULL — «не указано», 1 — да, 0 — нет. В модели User поля в $casts как boolean. Префикс is_ — единый стиль для булевых флагов в users (аналогично is_enabled).

2.2.2) UI (только модалка edit)

Partial: resources/views/includes/modal/_student_health_fields.blade.php (подключается в editUser.blade.php внутри @can('users.other.update')).

Где есть / где нет. Блок «Сведения об ученике» — только при редактировании существующего пользователя. В модалке создания (createUser) полей нет — при создании все три флага остаются NULL до первого edit.
Блок виден только если у актора есть users.other.update и в форме выбрана роль user (ученик); для тренера, админа и др. — скрыт (js-user-health-wrap, JS syncEditUserHealthFields).
  • Три поля — <select> с вариантами: «Не указано» (value="" → NULL), «Да» (1), «Нет» (0).
  • Имена полей формы: is_individual_traits, is_on_medical_register, is_with_disability.
  • В списке DataTables (/admin/users/data) и в «Поля списка» колонок для этих флагов нет (редкое использование, только карточка ученика).

2.2.3) Права

ПравоЧто даёт
users.view Доступ к разделу и endpoint’ам (§1). Не даёт видеть или менять мед./особенности — без users.other.update блок не показывается.
users.other.update Рендер блока в edit, приём и сохранение трёх флагов в PATCH /admin/users/{user}. Gate: users.other.update в AuthServiceProvider. В config/role_base_permissions.php для роли admin включено по умолчанию; миграция 2026_05_28_140001_add_users_other_update_permission.php выдаёт право существующим admin по всем партнёрам.

2.2.4) Валидация и сохранение

Только UpdateRequest (не StoreRequest).

  • Правила добавляются, если у актора users.other.update и эффективная роль редактируемого пользователя — user (учитывается входящий role_id в том же запросе).
  • Каждое поле: nullable|boolean. В prepareForValidation: пустая строка → null; 0/1 → false/true; прочие значения не нормализуются → 422.
  • Без users.other.update ключи is_* удаляются из запроса (offsetUnset) — значения в БД не меняются.
  • Для роли не user ключи также снимаются; при смене роли с ученика на другую сохранённые флаги в БД не обнуляются автоматически (остаются в строке users, но в UI не редактируются).
  • Сохранение: через UserService::update() в общем массиве полей User (не user_field_values).

2.2.5) API: edit и update

  • GET /admin/users/{id}/edit (AJAX) — в объекте user: is_individual_traits, is_on_medical_register, is_with_disability (true / false / null в JSON).
  • PATCH /admin/users/{user} — тело с теми же ключами (строки "", "0", "1" до нормализации).
  • Изменения пишутся в пользовательский лог (action 22), если у актора есть users.other.update: строки «Индивидуальные особенности», «Учёт у медспециалистов», «Инвалидность» в формате «Не указано / Да / Нет».

2.2.6) Изоляция партнёра

Те же правила scope, что для ученика в §2.1.3: users.partner_id = current_partner в UserController.

2.2.7) Договоры и отчёты

Пока не реализовано: предзаполнение полей договора (ContractTemplatePrefillSources / ContractPrefillResolver) для этих флагов не подключено. При необходимости добавляются отдельные prefill_source (например, «Да»/«Нет»/«—» для шаблона DOCX).

3) Локация пользователя (users.location_id)

Необязательная привязка учётной записи к справочнику locations текущего партнёра. Миграция: database/migrations/2026_05_19_120000_add_location_id_to_users_table.php.
Поле: users.location_id (nullable, FK → locations.id, ON DELETE SET NULL). Модели: User::location(), Location::users().
Один пользователь — не более одной локации (в отличие от many-to-many). Справочник локаций ведётся на GET /admin/locations (права locations.view / locations.manage) — отдельно от привязки к ученику.
Не путать с расписанием: слоты team_schedule_slots имеют свой location_id. Поле users.location_id — справочная привязка ученика в карточке; автоматической синхронизации со слотами расписания нет.

3.1) Права

ПравоЧто даёт в разделе «Пользователи»
users.view Доступ к странице и всем endpoint’ам раздела (обязательно). Без него — 403.
locations.view Селект локации в модалках create/edit; фильтр «Локация»; столбец и ключ location в columns-settings; чтение/запись location_id в store/update; поле location в JSON /admin/users/data.
locations.manage Не требуется для привязки ученика к локации — только для CRUD справочника на /admin/locations.
users.group.update Только поле «Группа» при редактировании (аналогия: для локации отдельного users.location.update нет).
users.other.update Три флага мед./особенностей ученика в edit (§2.2). Не даёт доступ к разделу без users.view.

Без locations.view: входящий location_id удаляется в StoreRequest / UpdateRequest; в ответе data поле location — пустая строка; фильтр по локации на сервере не применяется.

3.2) UI

  • Create — #create-location, опции: «Без локации» + активные локации партнёра (is_enabled = true).
  • Edit — #edit-location: те же активные + текущая локация, если она отключена (в JS подпись «(отключена)», опция добавляется динамически).
  • Фильтр — #filter-location: пусто / none / id активной локации.
  • Столбец «Локация» — после «Группа»; по умолчанию виден при первом заходе, если есть locations.view.

В index во view всегда передаётся $activeLocations (активные локации партнёра) — для фильтра и селектов; без права блоки в Blade не рендерятся (@can('locations.view')).

3.3) Валидация и сохранение

ОперацияRequestПравила
Создание StoreRequest При locations.view: location_id nullable, exists:locations,id с partner_id текущего партнёра и is_enabled = true. Пустая строка → null. Ошибки — ключ location_id (вывод под полем в модалке).
Обновление UpdateRequest При locations.view: location_id nullable integer. В withValidator: локация того же партнёра; если is_enabled = false — разрешена только если уже назначена этому пользователю.

3.4) API: data и edit

  • GET /admin/users/data — в каждой строке при locations.view: "location": "Название" (из связи user.location), иначе "".
  • GET /admin/users/{id}/edit (AJAX) — в user: location_id; объект location: { id, name, is_enabled } или null.
  • Изменение локации пишется в пользовательский лог (action 22) строкой «Локация: … → …», если у актора есть locations.view.

3.5) Использование в отчётах

Поле users.location_id задаётся в этом разделе и используется, в частности, в отчёте «Задолженности» (фильтр по текущей привязке ученика). В отчётах по оплатам используется снимок payments.location_id на момент платежа — см. reports-payments.html и payments.html. Смена users.location_id не пересчитывает старые строки payments.

4) Создание пользователя (POST /admin/users)

Маршрут admin.user.store. Валидация: App\Http\Requests\User\StoreRequest. Создание через UserService::store() в DB::transaction с логом (action 21) и опциональными user_field_values. Флаги мед./особенностей (§2.2) при создании не задаются — только в edit после появления ученика.

Пароль

Поле password необязательно. Пустая строка → password = NULL в БД. Если указан: min:8 / max:255; хеш — каст hashed модели User.

Телефон

Поле phone с маской +7 (999) 999-99-99 (Inputmask). Хранение: канон +7XXXXXXXXXX в users.phone. Право на запись: users.phone.update (без права поле игнорируется на бэкенде и disabled в UI).

Дополнительные поля (custom)

В модалке создания — те же доп. поля партнёра, что в edit (фильтр по роли админа, payload fields в JSON edit).

ТемаПоведение
Телоcustom[{slug}] — значения для user_field_values
Валидацияcustom.* nullable string; slug должен существовать у UserField текущего партнёра
ЗаписьТолько поля с editable для роли актора

custom не передаётся в User::create — извлекается до UserService::store().

Создание из заявки с сайта (school_lead_id)

На странице «Заявки с сайта» (/admin/school-leads) при schoolLeads.view + users.view в колонке «Договор» открывается та же модалка createUser.blade.php с предзаполнением имени, телефона и локации из лида и скрытым полем school_lead_id.
  • Опциональное поле school_lead_id в StoreRequest: заявка текущего партнёра, не удалена, school_leads.user_id ещё null.
  • После успешного store в UserController: school_leads.user_id = id нового ученика (в той же транзакции, что создание пользователя).
  • Отдельного маршрута у заявок для создания клиента нет — только POST /admin/users с middleware can:users.view.

5) Редактирование (PATCH /admin/users/{user})

Валидация: UpdateRequest. Поля с отдельными правами: имя (users.name.update), группа (users.group.update), email, телефон, роль, активность, дата рождения, дата начала — см. правила в Request. Сведения об ученике (три флага is_*) — §2.2 (users.other.update, только роль user). Доп. поля: custom[{slug}] с той же логикой партнёра и editable, что при создании (это user_fields, не путать с §2.2). Локация — §3 (locations.view).

5.1) Роль «Тренер» и группы (team_ids)

При trainers.view и роли trainer в модалке редактирования скрывается селект основной группы (team_id), показываются чекбоксы team_ids[] (те же, что на странице /admin/trainers). UserService::update синхронизирует team_trainer только при trainers.view у актора и наличии ключа team_ids в запросе. Смена роли на trainer создаёт trainer_profiles через TrainerProfileSyncService; смена на другую роль — soft delete профиля. Подробно: Тренеры (админка).
  • JSON edit: trainer_team_ids — id групп для предзаполнения чекбоксов.
  • Для роли trainer поле team_id в update не сохраняется (сбрасывается на сервере).
