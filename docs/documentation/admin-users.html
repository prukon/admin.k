Пользователи в админке

Файл: /docs/documentation/admin-users.html · URL в CRM: /docs/documentation/admin-users (список разделов: /doc)

Назначение и доступ

Страница списка: GET /admin/users (UserController@index), view resources/views/admin/user.blade.php. Маршруты раздела обёрнуты в middleware('can:users.view') (см. routes/web.php). Просмотр внутренней документации по адресу /doc / /docs/documentation требует отдельного права documentations.view — это не то же самое, что доступ к /admin/users.

Создание пользователя (POST /admin/users)

Имя маршрута: admin.user.store. Валидация: App\Http\Requests\User\StoreRequest. Создание учётной записи выполняется через UserService::store() внутри DB::transaction вместе с записью лога (тип действия создания учётки) и, при наличии данных, значений дополнительных полей в user_field_values.

Пароль (без изменений)

Поле password необязательно. Если пароль не указан или передана пустая строка, в таблице users сохраняется password = NULL (учётные записи без входа в систему). Если пароль указан, действуют ограничения min:8 / max:255; хеширование — каст password → hashed у модели User.

Телефон при создании (новый функционал)

В форме создания есть поле phone (тип tel) с маской +7 (999) 999-99-99 (Inputmask). На бэкенде телефон хранится и валидируется в каноническом виде: +7XXXXXXXXXX (11 цифр, включая 7), поле users.phone.
Тема Поведение в коде
Формат хранения users.phone — строка (до 20 символов), канон: +7 + 10 цифр. Модель User дополнительно нормализует входящие значения через mutator setPhoneAttribute().
Нормализация masked-ввода StoreRequest приводит ввод (например, +7 (999) 111-22-33) к +79991112233. Если номер не удаётся привести к валидному канону — в запрос подставляется null.
Валидация Если у актора есть право users.phone.update, применяется правило: regex:/^\+7\d{10}$/ (валидируем уже нормализованное значение).
Права доступа Поле телефона управляется правом users.phone.update. Если права нет, бэкенд игнорирует входящий phone (значение не сохранится и не может «случайно» попасть в БД). На фронте поле в модалке создания disabled и показывается подсказка об отсутствии прав.

Дополнительные поля пользователя при создании (новый функционал)

В модальном окне «Создание пользователя» выводятся те же дополнительные поля текущего партнёра, что и в модалке редактирования (текстовые поля), если для партнёра настроены записи в user_fields и они попадают в выборку по роли текущего администратора (аналогично GET /admin/users/{id}/edit в JSON-ключе fields).

Тема Поведение в коде
Тело запроса Вложенный массив custom: ключи — slug поля из user_fields, значения — строки (в форме: name="custom[{slug}]").
Партнёр Все определения полей и сохранение значений привязаны к partner_id текущего контекста (PartnerContext / SetPartner). Поля другого партнёра передать нельзя: валидация отклонит неизвестный для текущего партнёра slug (ошибка вида custom.{slug}).
Валидация (StoreRequest) custom — необязательный массив; каждый элемент — nullable|string|max:255. Дополнительно проверяется, что каждый переданный ключ существует как slug у UserField с partner_id текущего партнёра.
Права на запись значения В транзакции сохраняются только поля, помеченные как доступные для редактирования текущим актором (editable в payload полей — та же логика, что для формы редактирования: супер-админ или роль из связи user_field_role). Значения для неразрешённых полей в запросе не создают записей в user_field_values (в т.ч. при подборе slug вручную).
Пустые значения Пустая строка для доп. поля допустима; при сохранении может быть записана как пустое значение в user_field_values для разрешённых полей.
Транзакция Создание строки в users и вставка/обновление строк в user_field_values выполняются в одной транзакции с логом создания пользователя.
Ответ AJAX При успешном запросе с заголовком, ожидающим JSON (как в модалке), возвращается JSON с сообщением и краткими данными пользователя (как раньше); структура не расширялась специально для доп. полей.

Ключ custom не передаётся в User::create: он извлекается из проверенных данных до вызова UserService::store(), чтобы не попасть в массовое заполнение модели пользователя.

Редактирование (PATCH /admin/users/{user})

Дополнительные поля при обновлении по-прежнему передаются в custom[{slug}], валидация в UpdateRequest (custom.*). Разрешение полей по slug при сохранении учитывает partner_id текущего партнёра. Подробности логирования изменений по доп. полям — в коде UserController@update.
