Заявки с сайта: виджет (iframe) и CRM

Файл: /docs/documentation/school-leads-widget.html · оглавление: /doc · иерархия: directories-hierarchy · см. также страница заявки партнёра (лендинг)
Назначение: сбор заявок клиентов школы в CRM (school_leads) двумя публичными каналами:
  • Виджет iframe (/widget/{widgetKey}) — короткая форма для встраивания на сайт партнёра (§4).
  • Страница заявки (/lead/{landingSlug}) — брендированный лендинг с полной формой; slug задаёт партнёр в CRM; подробно в school-leads-landing.html.
Заявки изолированы по partner_id и отображаются в кабинете школы (/admin/school-leads).
Не путать с разделом «Лиды партнёров» (/admin/partner-leads, таблица partner_leads) — там B2B-заявки с лендинга kidscrm «подключить школу к платформе».

1) Две разные сущности заявок

Раздел CRMТаблицаКто оставляет заявкуИзоляцияПраво
/admin/partner-leads partner_leads Школа хочет подключиться к kidscrm Глобальный список (платформа) partnerLeads.view
/admin/school-leads school_leads Клиент школы с лендинга партнёра Только current_partner schoolLeads.view

2) Модель данных

  • partner_widgets — одна запись на партнёра: widget_key (iframe, 48 символов), landing_slug (публичный адрес лендинга, 3–40 символов, задаётся в CRM), landing_key (служебный, не в URL), is_active, is_landing_active. Создаётся при создании партнёра (Partner::created → PartnerWidgetService::ensureForPartner).
  • school_lead_statuses — справочник статусов заявок (§5.4): системный «Новый» (partner_id = null, code = new, read-only) + кастомные статусы партнёра (имя, цвет, sort_order, is_default_in_filter). Soft delete.
  • school_leads — заявки: partner_id, source (widget | landing), name, phone, school_lead_status_id (FK на school_lead_statuses, nullable), поля родителя/ребёнка (лендинг), location_id, sport_type_id (лендинг, nullable), team_id (лендинг), флаги особенностей, needs_contact_help, user_id (nullable — ученик после конвертации), UTM, page_url, referrer, согласие ПДн (consent_accepted_at, policy_url — у лендинга null), soft delete. Миграции: 2026_05_29_100001_add_extended_fields_to_school_leads_table.php, 2026_05_29_120002_add_sport_type_id_to_school_leads_table.php, 2026_06_16_140000_create_school_lead_statuses_and_migrate_school_leads.php (таблица статусов, FK, удаление колонки status enum).
  • partner_telegram_link_tokens — одноразовые токены для привязки Telegram через /start (срок жизни ~30 мин).
  • partners.school_leads_telegram_chat_id — Chat ID для уведомлений партнёра в Telegram.

Поля лендинга и публичные маршруты /lead/… — school-leads-landing.html.

3) Права и меню

В /config/role_base_permissions.php для роли admin (в разрезе партнёра):
  • schoolLeads.view — страница «Заявки с сайта», datatable, update/delete заявок, CRUD настраиваемых статусов (§5.4).
  • schoolLeadLanding.view — вкладка «Страница заявки», настройка landing_slug, ссылка /lead/{landingSlug}.
  • schoolWidget.view — вкладка «Виджет для сайта», код iframe, подключение Telegram.
  • Для колонки «Договор» и конвертации в ученика дополнительно нужны users.view и/или contracts.view (§5.6) — в базовый набор роли admin партнёра обычно уже входят.
Роль user этих прав не получает по умолчанию. Назначение — через «Права и роли» или сидеры (см. партнёры: базовые права).

Меню: resources/views/includes/sidebar.blade.php — пункт «Заявки с сайта» под @can('schoolLeads.view'). Вкладки «Страница заявки» и «Виджет для сайта» — под @can('schoolLeadLanding.view') и @can('schoolWidget.view') соответственно.

4) Публичный виджет (iframe)

Отдельная полная форма на /lead/{landingSlug} — не этот раздел; см. school-leads-landing.html.

МетодURLНазначение
GET/widget/{widgetKey}HTML формы (имя, телефон, согласие, reCAPTCHA v3)
POST/widget/{widgetKey}/submitПриём заявки, JSON-ответ
  • Контроллер: App\Http\Controllers\SchoolLeadWidgetController.
  • Валидация: App\Http\Requests\SubmitSchoolLeadRequest (ошибки по полям в JSON, 422).
  • reCAPTCHA: App\Services\RecaptchaVerificationService, конфиг services.recaptcha / .env RECAPTCHA_*.
  • Throttle: 60 запросов/мин на группу маршрутов виджета.
  • Заголовок Content-Security-Policy: frame-ancestors * (middleware widget.embed) — разрешено встраивание на чужие домены.
  • Согласие ПДн: ссылка на route('policy') (/policy, политика конфиденциальности).
  • UTM: из query iframe; page_url / referrer — из JS (страница родителя, если доступна).
  • При создании заявки автоматически проставляется системный статус «Новый» (school_lead_status_id = id записи с code = new).

Пример embed-кода для партнёра

<iframe src="https://example.com/widget/{widgetKey}"
        width="100%" height="420" style="border:0;" loading="lazy" title="Заявка"></iframe>

Готовый код копируется на GET /admin/school-widget (SchoolWidgetController).

5.1) Интерфейс страницы списка

View списка заявок: resources/views/admin/school-leads/index.blade.php (вкладка «Заявки» → tabs/leads.blade.php), стили toolbar — resources/css/admin-list-toolbar.css (см. переиспользуемые UI-блоки и отчёт «Платежи»).

  • Toolbar — заголовок «Заявки», счётчики по всему партнёру (не зависят от фильтров таблицы), кнопки «Настройки» (модалка статусов, §5.4) и «История»:
    • Новых — заявки в системном статусе «Новый» (school_lead_status_id = id системной записи code = new);
    • Всего — все неудалённые заявки партнёра.
    При первой загрузке значения из leadStats (контроллер). При каждом ответе GET …/data поле stats в JSON обновляет счётчики на клиенте.
  • Фильтры (collapse, паттерн как в отчёте «Платежи») — кнопки «Применить» / «Сброс». Фильтры не применяются при изменении полей: только после «Применить».
    • Статус — filter multiselect (status_ids[], §5.1.2); при открытии страницы предвыбраны статусы с is_default_in_filter = true (включая системный «Новый»).
    • Район — при districts.view: «Все районы» / «Без района» / активные районы партнёра.
    • Объект — при locations.view: «Все объекты» / «Без объекта» / активные объекты партнёра.
    • Секция — «Все секции» / «Без секции» / активные группы партнёра (team_id).
    • Есть особые условия — чекбокс (has_special_conditions): is_individual_traits или is_on_medical_register или is_with_disability.
    • Сброс — возвращает статусы по умолчанию (is_default_in_filter), очищает район, объект, секцию и чекбокс особых условий.
  • Колонки (dropdown) — видимость колонок таблицы; настройки сохраняются на пользователя через API columns-settings. Порядок по умолчанию: после «ФИО родителя» идёт «Статус», затем телефон, email и остальные поля (см. §5.5.1).
  • Таблица — KidsCrmDataTable.create('#leads-table', …) (reusable-ui-partials §3): server-side DataTables, поиск, сортировка, пагинация; редактирование и удаление — в колонке «Действия» (иконка карандаша → модалка §5.5.2); договор — в колонке «Договор» (§5.6–5.8). Колонка «Статус» — type: 'inline-select' с кастомным рендером renderLeadStatusInlineSelect (§5.5.1): цветные бейджи и выпадающее меню выбора статуса без нативного <select>. В /admin/partner-leads — другой inline-select с фиксированным enum (не путать).
  • Колонка «Имя» (ФИО родителя из заявки) — type: 'link', класс edit-lead; клик открывает модалку «Редактирование лида» (§5.5.2), как кнопка «Изменить» в «Действиях». См. reusable-ui-partials §3.2.

Ключи колонок в настройках: name, phone, district (при districts.view), location (при locations.view), utm, page_url, status, comment, contract (при users.view и/или contracts.view), actions. Обзор иерархии — directories-hierarchy.

5.1.3) Аудит (my_logs)

Запись — через AuditLogger (audit-my-logs). События: school_lead.updated, school_lead.deleted (категория school_lead). Создание заявки с лендинга/виджета в CRM-лог не пишется — только действия на вкладке «Заявки».

  • update — diff «старое → новое» по названию статуса заявки, комментарию, району, объекту; лог только при изменениях.
  • delete — «Заявка удалена: Заявка #{id}: {ФИО}.» (level = security, soft delete).
  • CRUD статусов (§5.4) — отдельные события: school_lead.status_created, school_lead.status_updated, school_lead.status_deleted (категория school_lead).
  • UI — кнопка «История» (иконка fa-clock-rotate-left) в toolbar → GET /admin/school-leads/logs-data (route('logs.data.school-lead')), модалка includes.logModal.

5.3) Район и объект

  • Миграции: 2026_05_19_160000_add_location_id_to_school_leads_table.php, 2026_06_13_100002_add_district_id_to_school_leads_table.php.
  • Заявки из виджета — location_id = null, district_id = null, source = widget; район и объект назначают в CRM.
  • Заявки со страницы заявки — district_id и location_id из формы; см. school-leads-landing §5.4.
  • При districts.view: столбец «Район», фильтр («Все районы» / «Без района» / активные районы), поле в модалке редактирования.
  • При locations.view: столбец «Объект», фильтр («Все объекты» / «Без объекта» / активные объекты), поле в модалке. Объект должен соответствовать выбранному району; иначе 422.
  • Без соответствующего права: UI скрыт; поле в data/update игнорируется.

Район и объект заявки — поля school_leads.district_id и school_leads.location_id; в карточке ученика объекта нет (см. Объекты ↔ группы).

5.4) Настраиваемые статусы заявок

Справочник school_lead_statuses в разрезе партнёра. Управление — кнопка «Настройки» (иконка шестерёнки) в toolbar вкладки «Заявки» → модалка «Настройки статусов заявок».

ТипОписаниеОграничения
Системный «Новый» Одна запись на платформу (partner_id = null, code = new, is_system = true). Цвет по умолчанию #a0fe62 (SchoolLeadStatus::DEFAULT_NEW_COLOR; миграция 2026_06_17_120000_update_system_school_lead_new_status_color.php). В фильтре по умолчанию (is_default_in_filter = true). Нельзя переименовать, изменить или удалить (403). Рядом с названием — иконка fa-info-circle с подсказкой (KidsCrmTooltip).
Кастомные статусы партнёра Имя, цвет (#RRGGBB или #RGB), порядок (sort_order, авто при создании если не указан), флаг «Отображается заявки в этом статусе по умолчанию» (is_default_in_filter) — в таблице модалки колонка «Отображать». CRUD через API §5; удаление запрещено, если на статусе есть неудалённые заявки (422).
  • Новые заявки (виджет, лендинг) получают school_lead_status_id системного «Новый» автоматически.
  • UI модалки настроек — таблица статусов (колонки: название, сортировка, цвет, «Отображать», действия); для кастомных — иконки редактирования (fa-edit) и удаления (fa-trash); удаление через showConfirmDeleteModal (не window.confirm). В колонке «Цвет» — превью badge с фактическим цветом статуса.
  • Форма статуса — отдельная модалка поверх настроек; выбор цвета: color picker + палитра пресетов (как на /admin/lesson-packages/occurrence-statuses).
  • Валидация — StoreSchoolLeadStatusRequest, UpdateSchoolLeadStatusRequest; ошибки 422 в JSON, тексты на русском под полями формы.
  • Изоляция — кастомный статус другого партнёра → 404 на update/delete.
  • Права — те же, что на весь раздел: schoolLeads.view (отдельного permission нет).
  • Фильтр таблицы — filter multiselect в collapse (§5.1.2); после CRUD статусов опции в фильтре и inline-select в таблице обновляются без перезагрузки страницы (rebuild + setValues).

Файлы: модель App\Models\SchoolLeadStatus, контроллер Admin\SchoolLeadStatusController, partial admin/school-leads/partials/status-settings.blade.php, JS в tabs/leads.blade.php. Enum App\Enums\SchoolLeadStatus удалён.

Миграция с enum: миграция 2026_06_16_140000_… создаёт справочник и колонку school_lead_status_id, но не переносит значения из старого поля status автоматически. Существующие заявки могут остаться с school_lead_status_id = NULL до ручного обновления в БД (например, проставить системный «Новый» по school_lead_statuses.code = 'new').

5.7) Создание клиента из заявки (лид → ученик)

Связь хранится в school_leads.user_id (nullable FK на users, nullOnDelete). У старых учеников и лидов без конвертации поле null. Повторная конвертация одной заявки блокируется валидацией school_lead_id (заявка уже с user_id) и PUT заявки (422). Unique-индекс school_leads_user_id_unique на колонке user_id (миграция 2026_05_19_170000_add_user_id_to_school_leads_table.php) — один ученик не может быть привязан к двум заявкам.
  • UI — кнопка «Создать клиента» в модалке §5.5.2 (не в колонке таблицы). Доступна при users.view; disabled, пока не заполнены имя и фамилия ученика (tooltip с подсказкой).
  • Поток: сначала PUT /admin/school-leads/{id} (сохранение полей модалки), затем POST /admin/users (admin.user.store, middleware can:users.view) с телом из collectCreateClientPayload(): ФИО ученика, дата рождения, team_id, блок родителя, флаги особенностей, custom, скрытые role_id (роль user), is_enabled=1, school_lead_id.
  • Район и объект заявки (district_id, location_id) в карточку ученика не переносятся (как и раньше). См. admin-users §2.1.
  • Валидация: school_lead_id в App\Http\Requests\User\StoreRequest — заявка того же партнёра, без user_id, не удалена. Иначе 422.
  • После успешного store: school_leads.user_id = {новый user id}; модалка закрывается, таблица перезагружается.

Отдельного маршрута у заявок для создания клиента нет. Partial includes/modal/createUser.blade.php на странице заявок не подключается (используется только на /admin/users).

5.8) Создание договора без ухода со страницы заявок

  • При contracts.view на страницу заявок подключается partial contracts/partials/create-modal.blade.php (тот же UI, что на /client-contracts).
  • Кнопки «Создать договор» в колонке таблицы и в read-only модалке лида — type="button", класс js-open-create-contract-from-lead; вызывают openCreateContractFromLead(rowData) → window.KidsCrmContractCreate.openModal(preselectedUser). Страница не меняется.
  • Предзаполнение ученика (preselectedUser) собирается из строки DataTables: user_id, ФИО ученика, team_id / team_title, parent_full_name. При открытии из модалки лида она сначала закрывается, затем открывается модалка договора.
  • В JSON data по-прежнему отдаётся create_contract_url = route('contracts.index', ['user_id' => …]) для совместимости и прямых ссылок; прямой GET /client-contracts?user_id=… по-прежнему открывает модалку на странице договоров (§contracts §5).
  • Списание с баланса — только при POST /client-contracts из модалки (предварительно POST /client-contracts/check-balance), не при открытии модалки.
  • Связь «заявка ↔ договор» в БД не дублируется: договор ищется по user_id ученика, привязанного к лиду (LatestUserContractLookup).

6.1) Подключение Telegram без ручного Chat ID

  1. Admin нажимает «Подключить Telegram» в CRM → создаётся одноразовый токен в partner_telegram_link_tokens.
  2. Открывается deep-link на бота платформы (по умолчанию @kidscrmLeadFormBot, см. TELEGRAM_BOT_USERNAME).
  3. Пользователь нажимает «Старт» → Telegram шлёт webhook → сохраняется partners.school_leads_telegram_chat_id.

Webhook: POST /webhooks/telegram/school-leads (TelegramSchoolLeadWebhookController). Защита: заголовок X-Telegram-Bot-Api-Secret-Token = TELEGRAM_WEBHOOK_SECRET.

php artisan school-leads:telegram-webhook

Команда вызывает setWebhook у Telegram API. URL должен быть публичным HTTPS.

Важно: у одного бота Telegram — один webhook URL. Команда school-leads:telegram-webhook перезаписывает webhook бота. Глобальные B2B-уведомления на TELEGRAM_CHAT_ID (лиды с лендинга kidscrm через LandingPageController::notifyTelegram) после этого из Telegram API приходить не будут, пока не настроен общий обработчик или отдельный бот для B2B.

7) Уведомления о новой заявке

После успешного submit (POST /widget/…/submit или POST /lead/…/submit) вызывается SchoolLeadNotificationService:

  • Email — всем пользователям с ролью admin текущего партнёра (с email) + email организации из partners.email. Mailable: App\Mail\NewSchoolLeadSubmission; для лендинга с заполненными полями — строки «Район», «Объект», «Услуга» в теле письма.

B2B-лиды по-прежнему используют глобальный TELEGRAM_CHAT_ID в LandingPageController — это другой канал.
