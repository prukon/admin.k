Заявки с сайта: виджет (iframe) для партнёра

Файл: /docs/documentation/school-leads-widget.html · оглавление: /doc
Назначение: партнёр (школа) встраивает на свой лендинг iframe-форму сбора заявок. Заявки клиентов школы сохраняются в CRM в разрезе partner_id и отображаются в кабинете этой школы.
Не путать с разделом «Лиды партнёров» (/admin/partner-leads, таблица partner_leads) — там B2B-заявки с лендинга kidscrm «подключить школу к платформе».

1) Две разные сущности заявок

Раздел CRMТаблицаКто оставляет заявкуИзоляцияПраво
/admin/partner-leads partner_leads Школа хочет подключиться к kidscrm Глобальный список (платформа) partnerLeads.view
/admin/school-leads school_leads Клиент школы с лендинга партнёра Только current_partner schoolLeads.view

2) Модель данных

  • partner_widgets — один активный виджет на партнёра: widget_key (48 символов), is_active. Создаётся при создании партнёра (Partner::created → PartnerWidgetService::ensureForPartner).
  • school_leads — заявки: partner_id, name, phone, location_id (nullable, назначается в CRM), user_id (nullable — связанный ученик после конвертации в CRM), статус, UTM, page_url, referrer, согласие ПДн (consent_accepted_at, policy_url), soft delete.
  • partner_telegram_link_tokens — одноразовые токены для привязки Telegram через /start (срок жизни ~30 мин).
  • partners.school_leads_telegram_chat_id — Chat ID для уведомлений партнёра в Telegram.

3) Права и меню

В /config/role_base_permissions.php для роли admin (в разрезе партнёра):
  • schoolLeads.view — страница «Заявки с сайта», datatable, update/delete.
  • schoolWidget.view — страница «Виджет заявок», код iframe, подключение Telegram.
  • Для колонки «Договор» и конвертации в ученика дополнительно нужны users.view и/или contracts.view (§5.8) — в базовый набор роли admin партнёра обычно уже входят.
Роль user этих прав не получает по умолчанию. Назначение — через «Права и роли» или сидеры (см. партнёры: базовые права).

Меню: resources/views/includes/sidebar.blade.php — пункты под @can('schoolLeads.view') и @can('schoolWidget.view').

4) Публичный виджет (iframe)

МетодURLНазначение
GET/widget/{widgetKey}HTML формы (имя, телефон, согласие, reCAPTCHA v3)
POST/widget/{widgetKey}/submitПриём заявки, JSON-ответ
  • Контроллер: App\Http\Controllers\SchoolLeadWidgetController.
  • Валидация: App\Http\Requests\SubmitSchoolLeadRequest (ошибки по полям в JSON, 422).
  • reCAPTCHA: App\Services\RecaptchaVerificationService, конфиг services.recaptcha / .env RECAPTCHA_*.
  • Throttle: 60 запросов/мин на группу маршрутов виджета.
  • Заголовок Content-Security-Policy: frame-ancestors * (middleware widget.embed) — разрешено встраивание на чужие домены.
  • Согласие ПДн: ссылка на route('policy') (/policy, политика конфиденциальности).
  • UTM: из query iframe; page_url / referrer — из JS (страница родителя, если доступна).

Пример embed-кода для партнёра

<iframe src="https://example.com/widget/{widgetKey}"
        width="100%" height="420" style="border:0;" loading="lazy" title="Заявка"></iframe>

Готовый код копируется на GET /admin/school-widget (SchoolWidgetController).

5.1) Интерфейс страницы списка

View: resources/views/admin/school-leads.blade.php, стили toolbar — resources/css/admin-list-toolbar.css (как у отчёта «Платежи»).

  • Toolbar — заголовок «Заявки с сайта», счётчики по всему партнёру (не зависят от фильтров таблицы):
    • Новых — заявки со статусом new;
    • В обработке — processing;
    • Всего — все неудалённые заявки партнёра.
    При первой загрузке значения из leadStats (контроллер). При каждом ответе GET …/data поле stats в JSON обновляет счётчики на клиенте.
  • Фильтры (collapse) — чекбоксы статусов, кнопки «Применить» / «Сброс». Фильтры не применяются при изменении чекбокса: только после «Применить». Сброс возвращает статусы new + processing и очищает локацию.
  • Колонки (dropdown) — видимость колонок таблицы; настройки сохраняются на пользователя через API columns-settings.
  • Таблица — server-side DataTables, поиск, сортировка, пагинация; редактирование и удаление — в колонке «Действия»; конвертация в клиента и работа с договором — в колонке «Договор» (§5.5–5.7).

Ключи колонок в настройках: name, phone, location (только при locations.view), utm, page_url, status, comment, contract (при users.view и/или contracts.view), actions.

5.3) Локации

  • Миграция 2026_05_19_160000_add_location_id_to_school_leads_table.php — nullable FK на locations.
  • Заявки из виджета — location_id = null; назначение в CRM (модалка редактирования).
  • При locations.view: столбец «Локация», обычный <select> в фильтре («Все локации» / «Без локации» / активные локации партнёра), поле в модалке. Отключённые и чужие локации недоступны; назначить их через API → 422.
  • Без locations.view: UI локации скрыт; location_id в data и update игнорируется.

Поведение как у пользователей (filter-location).

5.4) Статусы и редактирование

Статусы: enum App\Enums\SchoolLeadStatus — new, processing, sale, rejected, spam.

PUT /admin/school-leads/{id} — JSON: status, comment, опционально location_id. Ошибки валидации — 422, тексты на русском (UpdateSchoolLeadRequest). Статус лида при создании клиента или договора не меняется автоматически.

5.5) Колонка «Договор» и права на UI

Отдельная колонка таблицы «Договор» показывается, если у пользователя есть users.view или contracts.view (в Blade: canShowLeadClientColumn).

Состояние лидаНужное правоЧто в колонке «Договор»
Нет user_id users.view Кнопка «Создать клиента» (btn-primary) — открывает модалку создания ученика на этой же странице
Есть user_id, у ученика нет договоров contracts.view Ссылка-кнопка «Создать договор» → GET /client-contracts/create?user_id={id} (без списания на этом шаге)
Есть user_id, есть договор(ы) contracts.view Текстовая ссылка на последний договор: «Договор №{id} ({статус})» → GET /client-contracts/{id}
Нет user_id, нет users.view — В ячейке «—»
Есть user_id, нет contracts.view — В ячейке «—» (клиент уже создан, договор — из раздела «Договоры»)

«Последний договор» — по created_at (затем id) среди записей contracts с school_id = partner_id и user_id ученика. Сервис: App\Services\SchoolLeads\LatestUserContractLookup. Подпись статуса — из Contract::$STATUS_RU (любой статус: Черновик, Отправлено, Подписано и т.д.).

5.6) Создание клиента из заявки (лид → ученик)

Связь хранится в school_leads.user_id (nullable FK на users, nullOnDelete). У старых учеников и лидов без конвертации поле null. Повторная конвертация одной заявки блокируется валидацией school_lead_id (заявка уже с user_id). Unique-индекс school_leads_user_id_unique на колонке user_id (миграция 2026_05_19_170000_add_user_id_to_school_leads_table.php) — один ученик не может быть привязан к двум заявкам.
  • Модалка — общий partial resources/views/includes/modal/createUser.blade.php (как на /admin/users), подключается на странице заявок при users.view.
  • При открытии из строки лида подставляются: имя (всё поле name заявки в «Имя ученика»), телефон (контакт в users.phone), локация (если locations.view). Фамилию ученика и блок «Данные родителя» (создание записи в parents) оператор заполняет вручную — см. admin-users §2.1 и parents-and-family-cabinet.
  • Скрытое поле формы: school_lead_id. Submit → POST /admin/users (admin.user.store, middleware can:users.view), не отдельный маршрут заявок.
  • Валидация: опциональный school_lead_id в App\Http\Requests\User\StoreRequest — заявка того же партнёра, без user_id, не удалена. Иначе 422 (school_lead_id.exists).
  • После успешного store в транзакции: school_leads.user_id = {новый user id}. Таблица заявок перезагружается без полной перезагрузки страницы.

Повторная привязка к другому лиду или второй клиент с того же лида блокируется валидацией (заявка уже с user_id).

5.7) Переход к созданию договора

  • Из колонки «Договор» — прямой переход на /client-contracts/create?user_id=… (middleware can:contracts.view).
  • На странице создания договора ученик предзаполняется в Select2, если передан валидный user_id своего партнёра (ContractsController@create, view-data preselectedUser).
  • Списание с баланса за создание договора происходит только при сохранении PDF на форме создания (POST /client-contracts), как и при обычном сценарии — не при клике «Создать договор» в таблице заявок. Подтверждение оплаты — модалка на странице /client-contracts/create (см. реализацию договоров в CRM).
  • Связь «заявка ↔ договор» в БД не дублируется в contracts: договор находится по user_id ученика, привязанного к лиду.

6.1) Подключение Telegram без ручного Chat ID

  1. Admin нажимает «Подключить Telegram» в CRM → создаётся одноразовый токен в partner_telegram_link_tokens.
  2. Открывается deep-link на бота платформы (по умолчанию @kidscrmLeadFormBot, см. TELEGRAM_BOT_USERNAME).
  3. Пользователь нажимает «Старт» → Telegram шлёт webhook → сохраняется partners.school_leads_telegram_chat_id.

Webhook: POST /webhooks/telegram/school-leads (TelegramSchoolLeadWebhookController). Защита: заголовок X-Telegram-Bot-Api-Secret-Token = TELEGRAM_WEBHOOK_SECRET.

php artisan school-leads:telegram-webhook

Команда вызывает setWebhook у Telegram API. URL должен быть публичным HTTPS.

Важно: у одного бота Telegram — один webhook URL. Команда school-leads:telegram-webhook перезаписывает webhook бота. Глобальные B2B-уведомления на TELEGRAM_CHAT_ID (лиды с лендинга kidscrm через LandingPageController::notifyTelegram) после этого из Telegram API приходить не будут, пока не настроен общий обработчик или отдельный бот для B2B.

7) Уведомления о новой заявке

После успешного POST /widget/…/submit вызывается SchoolLeadNotificationService:

  • Email — всем пользователям с ролью admin текущего партнёра (с email) + email организации из partners.email. Mailable: App\Mail\NewSchoolLeadSubmission.
  • Telegram — в partners.school_leads_telegram_chat_id, если заполнен. Бот: TELEGRAM_BOT_TOKEN.

B2B-лиды по-прежнему используют глобальный TELEGRAM_CHAT_ID в LandingPageController — это другой канал.
