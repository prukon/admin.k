Партнёр‑контекст и middleware SetPartner

Файл: /docs/documentation/partner-context.html
Эта страница описывает каноническое поведение партнёр‑контекста (школы/организации) в CRM, а также ожидаемое поведение middleware App\Http\Middleware\SetPartner.

1) Бизнес‑логика: зачем нужен партнёр‑контекст

  • Изоляция данных: каждый партнёр (школа) — отдельное юр. лицо, данные не должны “протекать” между партнёрами.
  • Права в разрезе партнёра: многие permissions назначаются на роль в рамках партнёра (через permission_role.partner_id).
  • Безопасность: пользователь не должен иметь возможность “подменить” партнёра, передав чужой partner_id в запросе/форме.

2) Термины и источники истины

  • Текущий партнёр — это значение session('current_partner').
  • Партнёр объекта запроса — это экземпляр модели Partner, зарегистрированный в контейнере как app('current_partner').
  • Источник истины по доступам и роли супер‑админа — сервис App\Services\PartnerContext (в т.ч. определение superadmin).
Anti‑leak правило: переключать current_partner через session может только superadmin. Для всех остальных текущий партнёр всегда фиксируется как user->partner_id.

3) Где включено

  • Middleware SetPartner подключён в группе middleware web (глобально для web‑запросов).
  • Файл: /app/Http/Kernel.php → $middlewareGroups['web'].

4) Поведение SetPartner (ожидаемые сценарии)

Ключевая идея: если контекст партнёра не может быть установлен корректно, пользователь не должен оставаться в админке/CRM.
Сценарий Кто актор Ожидаемое поведение
Пользователь не авторизован guest Middleware ничего не делает и пропускает запрос дальше (return $next($request)).
Нет current_partner и у пользователя нет partner_id non‑superadmin
  • Принудительный logout + invalidate session + regenerate CSRF token.
  • Redirect на route('login').
  • Ошибка в сессии: withErrors(['email' => 'Ваша организация недоступна.']).
Почему ключ email: форма логина отображает ошибку под полем Email (Blade использует @error('email')).
current_partner указывает на несуществующего партнёра non‑superadmin
  • То же поведение, что и выше: logout + redirect на логин.
  • Ошибка: email → “Ваша организация недоступна.”
Суперадмин без выбранного партнёра superadmin
  • Есть allow‑list роутов, которые можно открыть без контекста: список партнёров, переключение партнёра, 2FA‑роуты, logout.
  • Для остальных страниц: redirect на route('admin.partner.index') и ошибка withErrors(['partner' => 'Партнёр не выбран.']).
Суперадмин с битым current_partner (партнёр удалён/не найден) superadmin
  • session()->forget('current_partner') (сбрасываем битый контекст).
  • Redirect на route('admin.partner.index').
  • Ошибка: partner → “Текущий партнёр недоступен. Выберите другого.”

5) Пост‑условие: доступ к current_partner в коде

Если партнёр найден, middleware регистрирует его в контейнере приложения:
app()->instance('current_partner', $partner)
После этого код (контроллеры/сервисы) может безопасно использовать app('current_partner').

6) Тестируемость и контракт тестов

  • Нельзя подменять app('current_partner') вручную в тестах (это маскирует реальную механику).
  • Правильный путь: задавать current_partner через session и проверять эффекты middleware.
  • Канонические ожидания для кейса “non‑superadmin без партнёра”: 
    $resp->assertStatus(302);
$this->assertGuest();
$resp->assertSessionHasErrors(['email' => 'Ваша организация недоступна.']);

7) Связанные файлы

  • /app/Http/Middleware/SetPartner.php — реализация middleware
  • /app/Services/PartnerContext.php — определение superadmin и работа с контекстом
  • /resources/views/auth/login.blade.php — вывод ошибок по ключу email
  • /tests/Feature/Crm/* — feature‑тесты, фиксирующие контракт
