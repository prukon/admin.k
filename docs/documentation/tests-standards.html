Требования к единообразию Feature-тестов (партнёр, авторизация, права)

Файл: /docs/documentation/tests-standards.html
Эта страница фиксирует канонические правила для Feature-тестов проекта. Цель: чтобы тесты проверяли реальное поведение приложения (партнёр-контекст, авторизация, права), не “подменяя” его заглушками.

1) Партнёр-контекст (Partner)

  1. Источник истины по партнёру — сервис App\Services\PartnerContext.
    В тестах нельзя изобретать “третий способ” определения партнёра.
  2. Feature-тесты должны проверять использование PartnerContext через реальный HTTP-путь: session (current_partner) → middleware → PartnerContext → Gate/permissions → контроллер → ответ.
    Т.е. тесты не “дергают” PartnerContext вручную, а проверяют эффект/изоляцию на уровне запросов.
  3. Единый способ задать текущего партнёра — только через session: withSession(['current_partner' => $partnerId]) (или то, что делает CrmTestCase).
  4. Запрещено использовать app()->instance('current_partner', ...) в тестах.
    Это маскирует реальную механику SetPartner/PartnerContext и ломает безопасность/изоляцию.
  5. Anti-leak правило обязательно: переключать current_partner через session может только superadmin. Для всех остальных текущий партнёр должен быть равен user->partner_id.
    Это должно быть зафиксировано отдельным тестом “нельзя переключить партнёра не-superadmin’у”.
  6. Изоляция партнёров: для списков/CRUD/отчётов обязательно есть проверки, что данные чужого партнёра не “просвечивают” (ожидаемый результат: 404/403/пусто — в зависимости от контрактов контроллера).
  7. Middleware SetPartner — обязательная часть контракта админки/CRM: пользователь без partner_id (и/или без корректного выбора партнёра) должен быть заблокирован.
    В тестах должна быть минимум 1 проверка этого поведения для ключевых разделов (settings/reports/users и т.п.).
    Каноническое ожидаемое поведение для non‑superadmin без партнёра: 302, актор становится guest, и в сессии присутствует ошибка под ключом email с текстом “Ваша организация недоступна.” (чтобы вывести её под полем Email на логине). 
    $resp->assertStatus(302);
$this->assertGuest();
$resp->assertSessionHasErrors(['email' => 'Ваша организация недоступна.']);

2) Авторизация пользователя (Actor)

  1. Базовый сетап — только через tests/Feature/Crm/CrmTestCase.php.
    Тесты не дублируют seed + создание партнёра/пользователя/сессии, а используют уже созданные сущности.
  2. Переключение акторов — через хелперы CrmTestCase (например asAdmin(), asSuperadmin(), asForeignUser()).
    Не создавать “ручных” пользователей без причины, чтобы не терять связь с реальными правами/seed’ами.
  3. Тесты должны явно показывать, кто актор и в каком контексте он действует: роль/права, текущий партнёр в session, и ожидаемый статус ответа.

3) Реальные права (Permissions / Roles)

  1. Права — только настоящие: permissions + permission_role + правила из AuthServiceProvider.
    Например: permission setPrices.view соответствует ability setPrices.view.
  2. Глобальный bypass для суперадмина (обходит все permissions):
    В проекте суперадмин (роль roles.name = superadmin) проходит любой ability через Gate::before(...). Это влияет на всё: middleware can:*, вызовы Gate::allows(), директивы @can и т.п.
    Код: /app/Providers/AuthServiceProvider.php
    // Всё подряд разрешаем суперадмину (role.name = superadmin)
Gate::before(function (User $user, string $ability) use ($roleNameById) {
    return $roleNameById((int)$user->role_id) === 'superadmin' ? true : null;
});
    Важно для тестов: если в тесте используется asSuperadmin(), то проверки “нет permission → 403” становятся нерелевантны (суперадмин всё равно пройдёт). Для тестирования запретов используйте акторов без superadmin-роли.
  3. Запрещены заглушки авторизации для бизнес-прав: Gate::define()/Gate::before() нельзя использовать, чтобы “дать доступ”.
    Исключение — тесты, которые проверяют сам механизм Gate как отдельную тему, но это должно быть обосновано.
  4. Запрещены “магические id”: нельзя писать role_id = 1, ?? 1, “если не нашли — возьмём 1”. Роли/права должны извлекаться строго (например через firstOrFail() или через хелперы CrmTestCase).
  5. Если нужен пользователь без права — создавать его через общий хелпер вроде createUserWithoutPermission('users.view'), который гарантирует отсутствие записи в permission_role именно для текущего партнёра.

4) Маршруты и стабильность тестов

  1. Предпочитать route('...') вместо хардкода URL ('/admin/...'), если роут именован.
  2. Отключение middleware (withoutMiddleware()) допускается только точечно для проверки конкретного поведения, а не как способ “всё разрешить”.
  3. Статусы ответов должны соответствовать реальному контракту контроллера/мидлварей. Если контракт меняется — тесты обновляются, а не “подгоняются” заглушками.
Итог: тест считается “каноничным”, если он повторяет реальный путь приложения: session → middleware → PartnerContext → Gate/permissions → контроллер, и проверяет отсутствие утечек данных между партнёрами.

5) Юр. лица в Feature-тестах (T‑Bank, CloudKassir, выплаты)

С 2026‑06‑30 legacy-поля на partners (tinkoff_partner_id, tax_id, organization_name, vat и др.) в тестах не имитируют работу оплат и чеков. Приложение читает ShopCode и фискальные реквизиты только из partner_legal_entities через LegalEntityResolver. Подробнее о продуктовой логике — admin-legal-entities §1.1, payments §2.3.2.

5.1) Обязательная цепочка (как на проде)

partner_legal_entities (активное, tinkoff_shop_code / tax_id / vat / organization_name)
        ↑
teams.legal_entity_id
        ↑
payables.meta.team_id  (+ ученик в группе для Init)
        ↑
LegalEntityResolver::forPayable / forFiscalReceipt / forTinkoffPayment

Без meta.team_id на payable (или без группы в resolver) forPayable не падает обратно на forPartner — Init T‑Bank, автовыплата и сборка чека CloudKassir не сработают. При ≥2 активных юр. лицах у группы в цепочке обязателен teams.legal_entity_id (без fallback на default) — §5.4. Тест, который ставит только partners.tinkoff_partner_id, проверяет устаревший сценарий.

5.2) Хелперы CrmTestCase

Файл: tests/Feature/Crm/CrmTestCase.php. Использовать вместо ручного $partner->tinkoff_partner_id = ….

МетодНазначение
seedRegisteredLegalEntityForPartner(?Partner, shopCode, overrides) Активное юр. лицо с ShopCode; partners.tinkoff_partner_id обнуляется. Для выплат без payable (ручной e2c, job по forPartner).
seedTbankTeamChainForStudent(?Partner, ?User, shopCode, overrides) ShopCode → юр. лицо → новая группа с legal_entity_id → ученик в группе. Для Init T‑Bank / СБП / webhook автовыплаты.
seedFiscalTeamChainForStudent(?Partner, ?User, overrides) Юр. лицо с tax_id / organization_name / vat → группа → ученик. Для CloudKassirReceiptBuilder и SendCloudKassirReceiptJob; на payable нужен meta.team_id.
bindTeamsToLegalEntity(PartnerLegalEntity, Team ...) Привязка уже созданных групп (multi-team monthly и т.п.) к юр. лицу с ShopCode.
seedGlobalTbank(…) Глобальный терминал payment_systems(partner_id=NULL, name=tbank).

5.3) Примеры сценариев

  • Init карты / СБП — seedGlobalTbank + seedTbankTeamChainForStudent или seedRegisteredLegalEntityForPartner + bindTeamsToLegalEntity для существующих teamA/teamB.
  • Автовыплата webhook — цепочка через группу + meta.team_id на Payable и связанный PaymentIntent (TbankAutoPayoutOnWebhookTest).
  • Ручная выплата / дубли — достаточно seedRegisteredLegalEntityForPartner, если платёж без payable (TbankPayoutManualTest, TbankPayoutDuplicatePreventionTest).
  • CloudKassir — seedFiscalTeamChainForStudent, payable с meta.team_id; на партнёре оставить email (обязателен), при agent-схеме — phone, website (CloudKassirReceiptBuilderAgentTest).
  • Доступность T‑Bank — PaymentServiceTbankAvailabilityFeatureTest: legacy tinkoff_partner_id без юр. лица → false; отдельный кейс test_legacy_tinkoff_partner_id_is_ignored_without_legal_entity; unbound team в multi-entity → false + tbankAvailabilityError.

5.4) Multi-entity guard (2026‑06‑30)

При ≥2 активных partner_legal_entities группа обязана иметь teams.legal_entity_id с активным юр. лицом и ShopCode — иначе Init T‑Bank возвращает «Для выбранной группы не настроено юр. лицо». Продуктовая документация — payments §2.3.5.

Комплексный feature-тест раздела:

  • tests/Feature/Crm/PartnerLegalEntities/MultiEntityTbankGuardAccessFeatureTest.php
    • доступы: guest / 403 / 200 на admin.team.*, clubFee, payment.tinkoff.*;
    • AJAX JSON store/update группы с legal_entity_id → 200 / 422;
    • non-AJAX store → 302 + БД; non-AJAX validation → session errors;
    • клубный взнос: teamTbankCheckout, teamTbankCardAvailable, блок «Поставщик услуг», скрытие T‑Bank в HTML, Init по привязанной/непривязанной группе;
    • месячная витрина и Init: tbankAvailable, serviceProviderLabel, payable с meta.team_id;
    • single-entity: группа без привязки — T‑Bank по-прежнему доступен.
  • Inline JS клубного взноса — tests/Feature/Crm/Ui/BladeInlineJsSyntaxTest.php (payment/clubFee.blade.php).
  • tests/Feature/Crm/Payments/PaymentCheckoutServiceProviderFeatureTest.php — блок «Поставщик услуг»: PaymentCheckoutLegalEntityPresenter, PaymentCheckoutContextResolver, доступы POST /payment / clubFee, multi-entity, custom/lesson package (payments §3.1.2).
