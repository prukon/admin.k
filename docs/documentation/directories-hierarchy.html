Справочники: иерархия «Район → Объект → Группа»

Файл: /docs/documentation/directories-hierarchy.html · детали: группы, объекты, районы, виды спорта, лендинг, CRM заявки
Глоссарий UI (единый по проекту):
«Район» = модель District, таблица districts.
«Объект» = модель Location, таблица locations (в старой документации могло быть «локация» — в UI CRM с 2026-06 заменено на «Объект»).
«Группа» / «Услуга» = модель Team (секция на лендинге).
«Вид спорта» = модель SportType, таблица sport_types — плоский справочник партнёра (не в иерархии район→объект), но в CRM объединён с районами и объектами в одном разделе меню.
«Юр. лицо» = модель PartnerLegalEntity, таблица partner_legal_entities — реквизиты и ShopCode T‑Bank (sm-register); привязка к группам через teams.legal_entity_id (см. admin-legal-entities).
Меню CRM: один пункт сайдбара (иконка «книга») — при одном доступном справочнике называется как он (Группы, Объекты, Районы, Виды спорта, Юр. лица), при двух и более — «Справочники» (App\Support\DirectoriesMenu). Для пункта меню дополнительно нужно право directories.view. На страницах CRM заголовок раздела всегда «Справочники»; вкладки: Группы · Объекты · Районы · Виды спорта · Юр. лица (каждая — по своему permission).

1) Единый UI раздела «Справочники»

Пять справочных страниц CRM объединены одним заголовком, общим partial вкладок и одним пунктом сайдбара. URL страниц не менялись. С 2026-06 добавлена вкладка «Юр. лица» (/admin/legal-entities).

1.1) Подпись пункта сайдбара — DirectoriesMenu

Класс App\Support\DirectoriesMenu, метод forUser(?Authenticatable $user): ?array. Возвращает null, если нет ни одного из прав справочников; иначе ['url' => …, 'label' => …].

Приоритет URL (независимо от подписи): группы → объекты → районы → виды спорта → юр. лица. Заголовок страницы vs сайдбар: на HTML-страницах справочников (<h4>Справочники</h4>) заголовок не меняется — динамическая подпись только у пункта левого меню. У партнёра с единственным правом, например, groups.view, в сайдбаре «Группы», на странице /admin/teams — «Справочники» и вкладка «Группы».

2) Зачем и общая схема данных

Партнёр (школа) может иметь несколько районов города; в каждом районе — один или несколько объектов (филиалы, залы); к объектам привязаны группы (секции). Родитель на странице заявки выбирает цепочку район → объект → услуга; оператор в CRM видит те же поля в заявке.

districts (Район)
    └── locations.district_id (Объект, nullable в CRM)
            ├── location_admin_user → users (M2M, роль admin, до 5 на объект)
            └── teams.location_id (Группа, nullable)
                    └── school_leads.team_id (из лендинга)

school_leads.district_id + school_leads.location_id — снимок на момент заявки

5) CRM: страницы справочников

5.1) Группы — /admin/teams

Подробно: admin-teams.html.

  • Первая вкладка раздела «Справочники»; URL без изменений (/admin/teams).
  • DataTables, фильтры, модалки create/edit, привязка объектов и тренеров — см. admin-teams. Клик по названию группы — модалка edit (edit-team-link).
  • При locations.view — фильтры по объекту, району объекта и администратору объекта (location_id, district_id, admin_user_id — id админа в pivot; значение none для «без привязки» / «без района» / «без администратора»).
  • Право groups.view обязательно для доступа к странице и всем её endpoint'ам.

5.2) Объекты — /admin/locations

Подробно: admin-locations.html (§3 — M2M location_admin_user, до 5 админов на объект).

  • Select «Район» в create/edit — только активные районы (is_enabled = true).
  • Multiselect «Администратор» (admin_user_ids[], до 5) в create/edit — при locations.manage; колонка и фильтр — при locations.view. Фильтр таблицы передаёт admin_user_id (id или none) в /admin/locations/data.
  • Колонка и фильтр «Район» в таблице; фильтр «Без района» (district_id=none).
  • Сортировка по district_name (join к districts) и admin_user_label (минимальное ФИО среди администраторов объекта).
  • Multiselect групп (team_ids) при locations.manage.
  • Назначение админов объекта не используется для ограничения доступа к разделу — только отображение и фильтрация.

5.3) Районы — /admin/districts

Подробно: admin-districts.html.

  • DataTables: поиск по имени/id, фильтр active/inactive, сортировка (sort_order, locations_count, is_enabled).
  • Multiselect «Объекты» (location_ids[]) в модалках create и edit — при locations.view; sync через LocationDistrictSyncService.
  • Колонка «Объекты» — список названий с tooltip (locations_label), не только счётчик.
  • Удаление: 422, если есть объекты с district_id этого района (включая отключённые). Иначе — hard delete.

5.4) Виды спорта — /admin/sport-types

Подробно: admin-sport-types.html.

  • Четвёртая вкладка раздела «Справочники»; URL без изменений (/admin/sport-types).
  • DataTables: фильтры name, status; колонка teams_count.
  • Привязка к группам — на странице /admin/teams, не в этом справочнике.
  • На лендинге заявки — необязательный селект «Вид спорта» (если есть активные записи).

5.5) Юр. лица — /admin/legal-entities

Подробно: admin-legal-entities.html.

  • Пятая вкладка раздела «Справочники»; CRUD — модалки на списке (legal_entities.manage); sm-register T‑Bank — карточка /admin/legal-entities/{id} (legal_entities.sm_register).
  • DataTables: фильтры поиск / active-inactive; колонки «Регистрация в банке» (бейдж + hint при «Нет»), наименование через displayTitle(), teams_count, «Основное»; колонка ShopCode и кнопка «Регистрация» — только при legal_entities.sm_register.
  • Привязка к группам — селект в модалках /admin/teams (если ≥2 активных юр. лиц).
  • Оплаты T‑Bank: ShopCode через LegalEntityResolver (с legacy fallback на partners.tinkoff_partner_id).

6) Публичный лендинг — каскад выбора

Подробно: school-leads-landing §5.4.

  1. Район — активные районы с ≥1 активным объектом с заполненным district_id.
  2. Объект — GET /lead/{slug}/locations?district_id= (только enabled, с district_id).
  3. Услуга — GET /lead/{slug}/teams?location_id= (группы объекта через TeamLocationAvailabilityService).
  4. Submit — обязательны district_id и location_id; cross-check: объект принадлежит району; partner scope по slug.
AJAX vs submit: для чужого или неактивного district_id endpoint /locations возвращает 200 и пустой массив (не 422). Ошибки partner scope на submit — 422 с текстами под полями.

7) CRM: заявки с сайта

Подробно: school-leads-widget §5.3.

  • Виджет iframe (source=widget) — район и объект обычно null; назначают вручную в CRM.
  • Лендинг (source=landing) — district_id и location_id из формы.
  • GET /admin/school-leads/data: параметры district_id, location_id (значение none — без района/объекта); поиск по имени района/объекта при соответствующих правах.
  • PUT /admin/school-leads/{id} — UpdateSchoolLeadRequest: нельзя назначить отключённый район/объект; объект должен соответствовать району.
  • Уведомления: email (emails/new_school_lead.blade.php) и Telegram — для лендинга включают район и объект.

8) Связанные разделы (без district_id, но в той же цепочке)

  • Виды спорта — плоский справочник; не заменяет район/объект на лендинге.
  • Группы — необязательная привязка к объекту через location_id; опционально sport_type_id.
  • Объекты ↔ группы — расписание, отчёты.
  • Отчёт «Платежи» — фильтр по объекту (filter_location_id); в UI отчётов термин «Локация» может сохраняться исторически — имеется в виду объект (Location).
