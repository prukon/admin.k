Абонементы (lesson packages): назначение ученику и даты периода

Запись user_lesson_packages — полноценная бизнес-сущность сразу после создания на странице «Назначение абонементов». Поля starts_at и ends_at могут быть NULL до первой привязки к календарю расписания школы.

Правило периода

При первой привязке назначения к календарю (одна и та же точка для гибкого и фиксированного типа расписания шаблона):

  • starts_at — дата якоря привязки (дата выбранного занятия в календаре).
  • ends_at — starts_at плюс duration_days из шаблона абонемента (lesson_packages.duration_days), по той же логике, что использовалась раньше при ручном вводе даты начала на форме назначения.

Если период уже задан (обе даты не NULL), повторно через этот механизм он не перезаписывается. Для фиксированного абонемента повторная привязка того же назначения к календарю блокируется, если у записи уже есть даты периода или уже есть строки в user_team_schedule_slots.

Гибкое расписание

Эндпоинт привязки одного занятия (assign-flexible) выставляет период при первом успешном сохранении привязки, если даты ещё были NULL. Одна успешная операция добавляет одну строку в user_team_schedule_slots; при достижении лимита по числу строк (равному lessons_total) следующая привязка возвращает ошибку валидации.

UI календаря школы: привязка выполняется inline в модалке слота (без отдельной модалки «Гибкий абонемент»). Список подходящих назначений на дату — в GET …/slot-user-bind-actions → flexible.existing_assignments; одно назначение — один клик, несколько — select в #schoolCalSlotFlexFormWrap. Подробности — в разделе «Расписание школы» (§ «Гибкие абонементы в модалке слота»).

Фиксированное расписание

В модалке календаря выбирается уже существующее назначение (user_lesson_package_id) без периода; затем по якорной дате и шаблону (patterns[]: день недели и интервал времени) строится цепочка занятий и создаются строки user_team_schedule_slots. Период назначения задаётся в той же транзакции перед построением цепочки (UserLessonPackageCalendarPeriodService::applyFirstCalendarAnchor).

Длина цепочки ограничена доступным «слотом» в календаре: берётся lessons_total − уже существующее число строк по этому назначению. Если за период абонемента не удаётся разместить все вхождения шаблона или есть конфликты — привязка отклоняется с сообщением об ошибке (TeamScheduleCalendarService: проверка шаблона, построение цепочки, пересечения с другими занятиями ученика).

Важно: при успешной привязке фиксированного (как и гибкого или разового) абонемента поле lessons_remaining в user_lesson_packages не уменьшается — см. раздел ниже.

Остаток занятий (lessons_remaining) и календарь

Баланс абонемента хранится в user_lesson_packages.lessons_remaining и lessons_total.

  • Привязка к календарю школы (POST привязки гибкого, фиксированного или разового занятия) только создаёт записи в user_team_schedule_slots и при необходимости задаёт период назначения; lessons_remaining при этом не меняется.
  • Списание и возврат остатка выполняются при сохранении статуса занятия в календаре школы для записи с абонементом: учитывается флаг consumes_lesson у статуса и предыдущий статус по тому же слоту/дате/ученику (дельта −1 / +1 / 0). Подробности и таблицы статусов — в разделе про календарь школы.
  • Лимит строк календаря на одно назначение: количество строк user_team_schedule_slots с данным user_lesson_package_id не может превысить lessons_total. Для гибкого и разового это проверяется перед каждой новой привязкой; для фикса длина создаваемой цепочки считается от оставшегося «бюджета» занятий.

Сервис

Логика выставления дат инкапсулирована в App\Services\UserLessonPackageCalendarPeriodService::applyFirstCalendarAnchor.

Пробные занятия (trial) в календаре школы

Пробное занятие не является назначением абонемента. Это отдельная запись в user_team_schedule_slots с is_trial_lesson = true и user_lesson_package_id = NULL; у записи задаётся пробный баланс trial_lessons_remaining / trial_lessons_total (обычно 1/1). Списание пробного баланса и флаг «пробное уже использовано» у ученика описаны в разделе про календарь школы.

Разовое занятие (no_schedule)

Шаблон абонемента с lesson_packages.schedule_type = no_schedule. Назначение ученику — строка user_lesson_packages с lessons_total / lessons_remaining (обычно 1/1) и индивидуальной стоимостью fee_amount.

  • Страница «Назначения» (POST /admin/lesson-packages/assignments): создаётся назначение без периода (starts_at/ends_at = NULL). Привязка к календарю — отдельным шагом.
  • Модалка слота календаря школы (POST …/school-schedule/single-lesson-registration): если у ученика уже есть свободное назначение (mode=bind_existing), выполняется только привязка к слоту; иначе (mode=create_new) в одной транзакции создаётся назначение и строка календаря. Подробности UI и API — в разделе «Расписание школы».
  • Legacy-модалка #schoolCalSingleModal: привязка уже существующего назначения через POST …/assign-single-lesson (выбор user_lesson_package_id). Основной сценарий для администратора — модалка слота.

Период назначения (starts_at/ends_at) задаётся при первой привязке к календарю через UserLessonPackageCalendarPeriodService::applyFirstCalendarAnchor, как у гибкого абонемента. Списание lessons_remaining — только через статус занятия с consumes_lesson.

Отмена записи в календаре (DELETE …/single-lesson-registration/{userTeamScheduleSlot}) удаляет строку user_team_schedule_slots, но не удаляет назначение user_lesson_packages — его можно снова привязать к другому слоту. Если статус уже списывал занятие, остаток восстанавливается (+1).
