Журнал расписания: /schedule

Файл: /docs/documentation/schedule-journal.html · вкладка «Журнал»: GET /schedule (schedule.index) · см. также Нагрузка тренеров
Не путать с разделом «Расписание школы» (абонементы, календарь слотов, lesson_occurrence_statuses, lessonPackages.view).
Журнал /schedule — отдельная посещаемость по дням месяца для учеников CRM; данные в schedule_users и справочнике statuses (не в статусах занятий абонементов).
Пункт меню «Журнал расписания» — resources/views/includes/sidebar.blade.php (@can('schedule.view') или schedule.trainerSalary.view; ссылка на /schedule или /schedule/trainer-salary).
Маршруты вкладки «Журнал» и статусов в routes/web.php — middleware('can:schedule.view') внутри auth + 2fa. Вкладки «ЗП тренеров» / «Листы ЗП» — отдельные права schedule.trainerSalary.*.

1) Право доступа schedule.view

СитуацияРезультат
ГостьGET /schedule → редирект на логин; JSON-эндпоинты → 401
Авторизован, нет schedule.viewСтраница и все эндпоинты журнала → 403
Есть schedule.viewСтраница и перечисленные ниже API → 200 при валидных данных

То же permission дополнительно включает колонку «Расписание» (дни недели группы) на /admin/teams — см. Группы: schedule.view. Право trainers.view для журнала не обязательно: выбор тренера в ячейке «Посетил» доступен при schedule.view, список берётся из активных trainer_profiles текущего партнёра.

В config/role_base_permissions.php для роли admin строка schedule.view по умолчанию закомментирована — на новом партнёре право нужно включить в матрице прав.

2.1) Список строк в журнале: только активные ученики

В таблице вкладки «Журнал» и во всех user-эндпоинтах журнала (cell-context, update, user-schedule, sync-teams, set-group, update-schedule-range) участвуют только пользователи, которые одновременно:
  • принадлежат current_partner;
  • имеют users.is_enabled = 1;
  • имеют системную роль user (roles.name = 'user' и roles.is_sistem = 1, сидер RolesSeeder, id=3).

Кого нет в списке

Тип пользователяПочему не отображается
Администратор (roles.name = admin)Не ученик; системная роль admin
Тренер (roles.name = trainer)Не ученик; учёт тренеров — через trainer_profiles и вкладки «Нагрузка» / «ЗП» (см. Тренеры)
Пользователь с кастомной ролью партнёра (roles.is_sistem = 0)Даже если в UI роль называется «ученик» или «VIP» — в журнал не попадает. Кастомные роли: settings-roles-custom
Отключённый ученик (is_enabled = 0)Фильтр активности
СуперадминДругая системная роль
Согласованность с /admin/users: и журнал, и вкладка «Пользователи» в админке показывают только учеников (системная роль user). Отличие журнала — дополнительно is_enabled = 1 и контекст посещаемости; тренеры, администраторы и кастомные роли — на других вкладках раздела (см. admin-users-section).

Реализация

  • Scope модели User::scopeWithSystemRoleUser() — whereHas('role') с name = user и is_sistem = true.
  • ScheduleController::index() — partner_id + is_enabled = 1 + withSystemRoleUser(); затем фильтры год / месяц / группа (all / none / id) — по pivot team_user (User::filterByStudentTeam).
  • API: findScheduleStudentForPartner() и assertScheduleStudent() — те же условия; иначе 404.

Записи в schedule_users для пользователей, которые больше не проходят фильтр (смена роли, отключение), в БД могут остаться, но строка в таблице журнала не выводится и изменить ячейку через API нельзя. Отчёт «Нагрузка тренеров» агрегирует schedule_users по включённым пользователям партнёра (is_enabled = 1) без проверки роли — устаревшие строки для не-учеников теоретически могут учитываться в нагрузке, пока не удалены из БД.

3) Данные: таблицы и модели

3.1) schedule_users

Одна запись на пару ученик + дата (updateOrCreate по user_id + date).

ПолеОписание
user_idУченик: users с системной ролью user (roles.is_sistem = 1), is_enabled = 1, partner_id = current_partner — см. §2.1
dateДень (формат даты в БД, ключ ячейки Y-m-d)
status_idСсылка на statuses.id
descriptionКомментарий к ячейке (индикатор в углу ячейки на таблице)
trainer_profile_idNullable FK на trainer_profiles. Заполняется только при статусе «Посетил»; для остальных статусов при сохранении сбрасывается в null

Модель: App\Models\ScheduleUser, связи statusRelation, trainerProfile.

3.2) statuses (справочник журнала)

Типpartner_idis_systemПримерыКто создаёт
Системные (общие на проект) NULL true «Посетил», «Не был» Один раз на проект (сидер на dev, на prod — вручную или data-миграция; см. §8)

  • Сортировка: колонка sort_order; в UI настроек — колонка «Сортировка» (редактируется только у кастомных).
  • Цвет: HEX в поле color (например #75EB81); в ячейках журнала и в модалке настроек используется как background-color иконки (прозрачность в БД не хранится).

Выборка для журнала и API: Status::forSchedulePartner($partnerId) — кастомные партнёра + глобальные системные. ID «Посетил» для логики тренера: Status::globalVisitedId() (имя Status::VISITED_NAME).

6) Тренер при статусе «Посетил»

Назначение тренера — только в модалке ячейки, не в самой таблице журнала (в ячейке сетки тренер не отображается). Эти данные использует отчёт «Нагрузка тренеров» (агрегация по trainer_profile_id и дню недели).

6.1) Поведение в модалке

  1. При открытии ячейки — GET /schedule/cell-context?user_id=&date=.
  2. Блок «Тренер» (#cell-trainer-wrap) показывается, если выбран статус «Посетил».
  3. В select — активные тренеры партнёра (trainer_profiles.is_enabled, user включён) + пункт «Без тренера» (value="").
  4. Если запись «Посетил» ещё не сохранялась — в UI подставляется первый тренер группы из team_trainer (минимальный team_trainer.id для context_team_id — фильтр группы на странице журнала или query-параметр в cell-context).
  5. Если «Посетил» уже сохранён с trainer_profile_id = null — это явное «Без тренера» (не подменяется дефолтом группы).
  6. При смене статуса с «Посетил» на другой — поле тренера в запрос не уходит; в БД trainer_profile_id обнуляется на сервере.

6.2) API GET /schedule/cell-context

Валидация: GetScheduleCellContextRequest (user_id, date, опционально context_team_id). user_id должен быть активным учеником партнёра (§2.1), иначе 404.

Ответ (основные поля):

  • visited_status_id — id системного «Посетил»
  • current_status_id — статус сохранённой ячейки, если есть
  • team_ids, teams_label — все группы ученика из pivot (подпись в модалке ячейки)
  • team_id, team_default_trainer_profile_id — для группы context_team_id (или первой группы ученика)
  • trainer_profile_id_for_select — null (ещё не «Посетил»), строка id или "" (без тренера)
  • trainers — массив { id, name } (ФИО из users.full_name)

6.3) API POST /schedule/update

Валидация: UpdateScheduleCellRequest. user_id — тот же круг учеников, что в §2.1; иначе 404.

  • trainer_profile_id — nullable; значения "", none, 0 приводятся к null
  • Тренер должен существовать в trainer_profiles с partner_id = current_partner
  • Пишется в БД только если status_id = «Посетил»

В аудит my_logs (event = schedule.day_updated) добавлены строки «Тренер до» / «Тренер после».

6.4) Связь с тренерами и группами

Дефолтный тренер ячейки = первый в team_trainer для группы context_team_id (фильтр журнала или query) — тот же порядок, что при одном тренере на группу в привязке тренера к группе. Если у ученика нет групп в pivot или у выбранной группы нет тренера — дефолт пустой, можно выбрать любого тренера партнёра или «Без тренера». Полное описание M:N групп: student-team-membership.

7) Массовое «Личное расписание ученика»

POST /schedule/user/{user}/update-schedule-range — только для активного ученика с системной ролью user (§2.1); иначе 404. Удаляет записи ученика в диапазоне дат и вставляет «Посетил» на выбранные дни недели (weekdays: 1–7 ISO). Поле trainer_profile_id при массовой вставке не заполняется (остаётся null). Тренера для таких дней при необходимости задают вручную через модалку ячейки.

8.2) AJAX-контракт (основной путь UI)

Формы #createStatusForm и #editStatusForm — preventDefault + fetch с заголовком X-Requested-With: XMLHttpRequest.

МетодУспехТело ответаОшибки
GET /schedule/statuses 200 { "statuses": [ { id, partner_id, name, icon, color, is_system, sort_order }, … ] } 401 / 403
POST /schedule/statuses 200 { "success": true, "status": { … } } 422 — валидация (name обязателен и др.)
PATCH /schedule/statuses/{id} 200 { "success": true, "status": { … } } 422 валидация; 403 legacy-системный; 404 чужой / глобальный системный
DELETE /schedule/statuses/{id} 200 { "success": true } 403 legacy-системный; 404 чужой / глобальный системный

Валидация: StoreScheduleStatusRequest, UpdateScheduleStatusRequest (обязателен name; для update — sort_order 0…65535). Тексты ошибок — в JSON errors, на фронте выводятся под полями (data-error-for). Если sort_order не передан при create — назначается автоматически (nextCustomSortOrder).

8.3) Safety-net: non-AJAX POST/PATCH

Если форма ушла нативным submit без X-Requested-With / без ожидания JSON (StatusController::store, update — ветка !ajax() && !expectsJson()):

  • запись в БД создаётся или обновляется (как при AJAX);
  • ответ — 302 redirect на route('schedule.index') (/schedule), не пустой 200.

Удаление (DELETE) в UI только через fetch; отдельного non-AJAX redirect для destroy нет.

8.4) Инициализация данных

  • Dev: php artisan db:seed --class=DevScheduleStatusesSeeder (при SEED_DEV_DATA=true — в составе DatabaseSeeder): глобальные «Посетил»/«Не был» + демо-кастомные по партнёрам.
  • Production: глобальные системные статусы должны существовать с partner_id IS NULL, is_system = 1. Отдельной миграции с вставкой (как у school_lead_statuses) пока нет — при пустом списке проверьте БД.
  • Метод DevScheduleStatusesSeeder::ensureGlobalSystemStatuses() — идемпотентное создание/обновление глобальных системных; можно вызвать вручную на prod при необходимости.
  • DevScheduleStatusesSeeder::removeLegacyPerPartnerSystemDuplicates() — удаляет ошибочные копии «Посетил»/«Не был» с partner_id; запускать только после появления глобальных записей.
