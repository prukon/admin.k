Журнал расписания: /schedule

Файл: /docs/documentation/schedule-journal.html · вкладка «Журнал»: GET /schedule (schedule.index) · см. также Нагрузка тренеров
Не путать с разделом «Расписание школы» (абонементы, календарь слотов).
Журнал /schedule — отдельная посещаемость по дням месяца (schedule_users). Справочник статусов общий: lesson_occurrence_statuses (вкладка «Статусы занятий» на /schedule и в разделе абонементов). Списание занятий абонемента из журнала не выполняется — только календарь школы.
Пункт меню «Журнал расписания» — resources/views/includes/sidebar.blade.php (@can('schedule.view') или schedule.trainerSalary.view; ссылка на /schedule или /schedule/trainer-salary).
Маршруты вкладки «Журнал» / «Нагрузка» — middleware('can:schedule.view'). Вкладка и CRUD «Статусы занятий» — Gate lessonOccurrenceStatuses.manage (= schedule.view OR lessonPackages.view), см. §8. Вкладки «ЗП тренеров» / «Листы ЗП» — schedule.trainerSalary.*.

То же permission дополнительно включает колонку «Расписание» (дни недели группы) на /admin/teams — см. Группы: schedule.view. Право trainers.view для журнала не обязательно: выбор тренера в ячейке «Посетил» доступен при schedule.view, список берётся из активных trainer_profiles текущего партнёра.

В config/role_base_permissions.php для роли admin строка schedule.view по умолчанию закомментирована — на новом партнёре право нужно включить в матрице прав.

2.1) Список строк в журнале: только активные ученики

В таблице вкладки «Журнал» и во всех user-эндпоинтах журнала (cell-context, update, user-schedule, sync-teams, set-group, update-schedule-range) участвуют только пользователи, которые одновременно:
  • принадлежат current_partner;
  • имеют users.is_enabled = 1;
  • имеют системную роль user (roles.name = 'user' и roles.is_sistem = 1, сидер RolesSeeder, id=3).

Кого нет в списке

Тип пользователяПочему не отображается
Администратор (roles.name = admin)Не ученик; системная роль admin
Тренер (roles.name = trainer)Не ученик; учёт тренеров — через trainer_profiles и вкладки «Нагрузка» / «ЗП» (см. Тренеры)
Пользователь с кастомной ролью партнёра (roles.is_sistem = 0)Даже если в UI роль называется «ученик» или «VIP» — в журнал не попадает. Кастомные роли: settings-roles-custom
Отключённый ученик (is_enabled = 0)Фильтр активности
СуперадминДругая системная роль
Согласованность с /admin/users: и журнал, и вкладка «Пользователи» в админке показывают только учеников (системная роль user). Отличие журнала — дополнительно is_enabled = 1 и контекст посещаемости; тренеры, администраторы и кастомные роли — на других вкладках раздела (см. admin-users-section).

Реализация

  • Scope модели User::scopeWithSystemRoleUser() — whereHas('role') с name = user и is_sistem = true.
  • ScheduleController::index() — partner_id + is_enabled = 1 + withSystemRoleUser(); затем фильтры год / месяц / группа (all / none / id) — по pivot team_user (User::filterByStudentTeam).
  • API: findScheduleStudentForPartner() и assertScheduleStudent() — те же условия; иначе 404.

Записи в schedule_users для пользователей, которые больше не проходят фильтр (смена роли, отключение), в БД могут остаться, но строка в таблице журнала не выводится и изменить ячейку через API нельзя. Отчёт «Нагрузка тренеров» агрегирует schedule_users по включённым пользователям партнёра (is_enabled = 1) без проверки роли — устаревшие строки для не-учеников теоретически могут учитываться в нагрузке, пока не удалены из БД.

Ответы API для «не-ученика»

СитуацияКод

3) Данные: таблицы и модели

3.1) schedule_users

Одна запись на пару ученик + дата (updateOrCreate по user_id + date).

ПолеОписание
user_idУченик: users с системной ролью user (roles.is_sistem = 1), is_enabled = 1, partner_id = current_partner — см. §2.1
dateДень (формат даты в БД, ключ ячейки Y-m-d)
lesson_occurrence_status_idFK на lesson_occurrence_statuses.id (активный статус партнёра)
descriptionКомментарий к ячейке (индикатор в углу ячейки на таблице)
trainer_profile_idNullable FK на trainer_profiles. Заполняется только при статусе «Посетил»; для остальных статусов при сохранении сбрасывается в null

Модель: App\Models\ScheduleUser, связи statusRelation, trainerProfile.

3.2) lesson_occurrence_statuses (единый справочник)

Один справочник на партнёра для журнала (/schedule) и календаря школы (абонементы). Таблица statuses и модалка «Настройки статусов» на журнале удалены (обратная совместимость не нужна).
Типpartner_idis_systemПримеры (code)Кто создаёт
Системные id партнёра (не глобальные) true «Посетил» (attended), «Не посетил» (not_attended), «Запись» (scheduled), «Отмена», «Заморозка» LessonOccurrenceStatusesSeeder::ensureForPartner при создании партнёра и при открытии UI
Кастомные id партнёра false произвольное название, code = custom_… Админ во вкладке «Статусы занятий» (оба экрана)
  • Поля: title, color, icon, sort_order, is_active, consumes_lesson.
  • В селекте ячейки журнала — только is_active = true. Неактивный статус на уже сохранённой ячейке всё ещё отображается (lookup по полной выборке партнёра).
  • «Посетил» для тренера / нагрузки / массового расписания: LessonOccurrenceStatus::attendedIdForPartner($partnerId) (code = attended).
  • Флаг consumes_lesson влияет на списание абонемента только в календаре школы. Журнал /schedule его не применяет (не пишет в user_lesson_occurrence_status_events).

6) Тренер при статусе «Посетил»

Назначение тренера — только в модалке ячейки. Данные использует отчёт «Нагрузка тренеров» (фильтр по lesson_occurrence_status_id = attended партнёра).

6.1) Поведение в модалке

  1. При открытии ячейки — GET /schedule/cell-context?user_id=&date=.
  2. Блок «Тренер» показывается, если выбран статус с code = attended (id из visited_status_id).
  3. В select — активные тренеры партнёра + «Без тренера».
  4. Если «Посетил» ещё не сохранялся — дефолт = первый тренер группы (team_trainer по context_team_id).
  5. Если «Посетил» сохранён с trainer_profile_id = null — явное «Без тренера».
  6. Смена статуса с «Посетил» на другой — trainer_profile_id обнуляется на сервере.

6.2) API GET /schedule/cell-context

Валидация: GetScheduleCellContextRequest. Ученик — круг §2.1, иначе 404.

  • visited_status_id — id attended текущего партнёра
  • current_status_id — schedule_users.lesson_occurrence_status_id, если есть
  • team_ids, teams_label, team_default_trainer_profile_id, trainer_profile_id_for_select, trainers

6.3) API POST /schedule/update

Валидация: UpdateScheduleCellRequest — поле lesson_occurrence_status_id (активный статус партнёра; alias status_id в request принимается для совместимости клиента).

  • trainer_profile_id — nullable; "" / none / 0 → null
  • Тренер пишется в БД только если статус = attended
  • Списания абонемента / событий user_lesson_occurrence_status_events нет

Аудит: schedule.day_updated — в тексте «Статус до/после» берётся title из справочника.

6.4) Связь с тренерами и группами

Дефолтный тренер = первый в team_trainer для context_team_id — привязка тренера к группе. M:N групп: student-team-membership.

7) Массовое «Личное расписание ученика»

POST /schedule/user/{user}/update-schedule-range — активный ученик §2.1, иначе 404. Удаляет записи в диапазоне и вставляет «Посетил» (attended партнёра) на выбранные дни недели. trainer_profile_id при массовой вставке не заполняется.

8.3) AJAX-контракт

МетодУспехТелоОшибки

PUT …/occurrence-statuses/{id} 200 { message: "Статус обновлён" } 422 / 404

Requests: StoreLessonOccurrenceStatusRequest, UpdateLessonOccurrenceStatusRequest, ReorderLessonOccurrenceStatusesRequest — authorize через lessonOccurrenceStatuses.manage. Ошибки валидации в JSON errors, на фронте под полями (data-err).

8.4) Safety-net: non-AJAX

Без Accept: application/json / без ожидания JSON (!$request->expectsJson()):

  • store / update / destroy — запись создаётся/обновляется/удаляется, ответ 302 на admin.lesson-packages.occurrence-statuses.index (не пустой 200);
  • ошибка валидации FormRequest — 302 back + session errors.

8.5) Инициализация

  • LessonOccurrenceStatusesSeeder::ensureForPartner — при Partner::created, в DatabaseSeeder, при открытии вкладок статусов / журнала.
  • Отдельного DevScheduleStatusesSeeder / таблицы statuses больше нет.
