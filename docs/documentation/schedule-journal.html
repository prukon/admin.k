Журнал расписания: /schedule

Файл: /docs/documentation/schedule-journal.html · страница CRM: GET /schedule (schedule.index)
Не путать с разделом «Расписание школы» (абонементы, календарь слотов, lesson_occurrence_statuses, lessonPackages.view).
Журнал /schedule — отдельная посещаемость по дням месяца для учеников CRM; данные в schedule_users и справочнике statuses (не в статусах занятий абонементов).
Пункт меню «Журнал расписания» — resources/views/includes/sidebar.blade.php (@can('schedule.view')).
Все маршруты журнала и статусов журнала в routes/web.php обёрнуты в middleware('can:schedule.view') внутри группы auth + 2fa.

1) Право доступа schedule.view

СитуацияРезультат
ГостьGET /schedule → редирект на логин; JSON-эндпоинты → 401
Авторизован, нет schedule.viewСтраница и все эндпоинты журнала → 403
Есть schedule.viewСтраница и перечисленные ниже API → 200 при валидных данных

То же permission дополнительно включает колонку «Расписание» (дни недели группы) на /admin/teams — см. Группы: schedule.view. Право trainers.view для журнала не обязательно: выбор тренера в ячейке «Посетил» доступен при schedule.view, список берётся из активных trainer_profiles текущего партнёра.

В config/role_base_permissions.php для роли admin строка schedule.view по умолчанию закомментирована — на новом партнёре право нужно включить в матрице прав.

3) Данные: таблицы и модели

3.1) schedule_users

Одна запись на пару ученик + дата (updateOrCreate по user_id + date).

ПолеОписание
user_idУченик (users, роль user, is_enabled = 1 в выборке журнала)
dateДень (формат даты в БД, ключ ячейки Y-m-d)
status_idСсылка на statuses.id
descriptionКомментарий к ячейке (индикатор в углу ячейки на таблице)
trainer_profile_idNullable FK на trainer_profiles. Заполняется только при статусе «Посетил»; для остальных статусов при сохранении сбрасывается в null

Модель: App\Models\ScheduleUser, связи statusRelation, trainerProfile.

3.2) statuses (справочник журнала)

  • Системные (общие на проект): partner_id IS NULL, is_system = true — «Посетил», «Не был». Редактирование и удаление через API запрещено (403).
  • Кастомные партнёра: partner_id = current_partner, is_system = false — например «Заморозка», «Болезнь» (на dev создаёт DevScheduleStatusesSeeder).
  • Сортировка: колонка sort_order; в UI настроек журнала — колонка «Сортировка» (только для несистемных статусов).

Выборка для журнала: Status::forSchedulePartner($partnerId) — кастомные партнёра + системные глобальные. ID «Посетил» для логики тренера: Status::globalVisitedId() (имя Status::VISITED_NAME).

4) UI страницы

  • Фильтры: год, месяц, группа (all / none / id группы).
  • Таблица: строки — ученики, столбцы — дни месяца; цвет/иконка из статуса.
  • Клик по ячейке → модалка #cellEditModal: статус (радио), комментарий, кнопка «Сохранить».
  • Иконка у строки ученика → «Личное расписание» (диапазон дат + дни недели).
  • Кнопка «Настройки» → список статусов, создание/редактирование кастомных.
  • Полноэкранный режим, поиск по таблице (DataTables).

Шаблон: resources/views/admin/schedule.blade.php, стили/скрипты: resources/css/schedule.css, resources/js/schedule.js.

5) Тренер при статусе «Посетил»

Назначение тренера — только в модалке ячейки, не в самой таблице (в ячейке сетки тренер не отображается). Задумано как основа для будущих отчётов по тренерам и расчёта ЗП (количество тренировок по отметкам «Посетил») — отдельная итерация.

5.1) Поведение в модалке

  1. При открытии ячейки — GET /schedule/cell-context?user_id=&date=.
  2. Блок «Тренер» (#cell-trainer-wrap) показывается, если выбран статус «Посетил».
  3. В select — активные тренеры партнёра (trainer_profiles.is_enabled, user включён) + пункт «Без тренера» (value="").
  4. Если запись «Посетил» ещё не сохранялась — в UI подставляется первый тренер группы из team_trainer (минимальный team_trainer.id для users.team_id ученика).
  5. Если «Посетил» уже сохранён с trainer_profile_id = null — это явное «Без тренера» (не подменяется дефолтом группы).
  6. При смене статуса с «Посетил» на другой — поле тренера в запрос не уходит; в БД trainer_profile_id обнуляется на сервере.

5.2) API GET /schedule/cell-context

Валидация: GetScheduleCellContextRequest (user_id, date).

Ответ (основные поля):

  • visited_status_id — id системного «Посетил»
  • current_status_id — статус сохранённой ячейки, если есть
  • team_id, team_default_trainer_profile_id
  • trainer_profile_id_for_select — null (ещё не «Посетил»), строка id или "" (без тренера)
  • trainers — массив { id, name } (ФИО из users.full_name)

5.3) API POST /schedule/update

Валидация: UpdateScheduleCellRequest.

  • trainer_profile_id — nullable; значения "", none, 0 приводятся к null
  • Тренер должен существовать в trainer_profiles с partner_id = current_partner
  • Пишется в БД только если status_id = «Посетил»

В лог (MyLog, action 93) добавлены строки «Тренер до» / «Тренер после».

5.4) Связь с тренерами и группами

Дефолтный тренер ячейки = первый в team_trainer для группы ученика — тот же порядок, что при одном тренере на группу в привязке тренера к группе. Если у ученика нет team_id или у группы нет тренера — дефолт пустой, можно выбрать любого тренера партнёра или «Без тренера».

6) Массовое «Личное расписание ученика»

POST /schedule/user/{user}/update-schedule-range удаляет записи ученика в диапазоне дат и вставляет «Посетил» на выбранные дни недели (weekdays: 1–7 ISO). Поле trainer_profile_id при массовой вставке не заполняется (остаётся null). Тренера для таких дней при необходимости задают вручную через модалку ячейки.

7) Настройки статусов

Модалка «Настройки» на странице журнала. Таблица статусов с сервера (в т.ч. колонка сортировки); CRUD только для кастомных.

Запросы: StoreScheduleStatusRequest, UpdateScheduleStatusRequest (обязательны name, для update — sort_order).

На dev: php artisan db:seed --class=DevScheduleStatusesSeeder (при SEED_DEV_DATA=true — в составе сидов).
