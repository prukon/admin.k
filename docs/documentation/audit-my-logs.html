Аудит CRM: таблица my_logs (колонки event и level)

Файл: /docs/documentation/audit-my-logs.html · UI вкладки «Настройки → Логи»: settings-logs.html
Суть. Журнал аудита CRM хранится в my_logs. С 2026-06 каноническими полями являются event (строка domain.action) и level (info | security | integration). Запись в приложении — только через App\Services\Audit\AuditLogger; в runtime-фильтрах и подписях UI legacy-колонки type и action не используются.
Legacy type / action. В старых строках БД коды могут оставаться (архив). Для новых записей AuditLogger явно пишет NULL. Колонки не удалялись; backfill-миграции заполнили event/level по старым кодам. Не путать с полями type в других таблицах (payables, contract_events, фискальные чеки и т.д.).

1) Реестр событий AuditEvent

Enum App\Enums\AuditEvent — единый справочник:

  • Значение — канонический event, например user.updated, schedule.user_range_updated.
  • Уровень — AuditLevel: info (изменения в CRM), security (входы), integration (платежи, webhook, подписание).
  • location — домен для модалок «История изменений»: pricing, user, team, location, district, sport_type, school_lead, contract_template, schedule, role, partner, settings, payment, auth, contract и др.
  • Методы legacyType() / legacyAction() / fromLegacy() — только для миграций backfill и unit-тестов реестра, не для HTTP/UI.

Примеры событий расписания (вместо старых «type=6, action=60» в документации):

eventНазначение (кратко)
schedule.user_range_updatedИзменение индивидуального расписания ученика
schedule.trial_cancelledОтмена пробного занятия (legacy action 601)
schedule.single_lesson_registration_cancelledОтмена записи разового занятия (legacy action 602)
schedule.slot_occurrence_skippedПропуск даты слота (legacy action 461)

Примеры пользователей: user.created (legacy action 21), user.updated (22).

2) Запись: AuditLogger

app(AuditLogger::class)->record(
    AuditEvent::SettingsUpdated,
    AuditContext::make('Описание изменения')
        ->withPartnerId($partnerId)
        ->withAuthorId($authorId)
        ->withTarget($model, $label)
);
  • Пишутся: event, level, description, author_id, partner_id, user_id, target_*, created_at.
  • type и action — всегда NULL для новых строк.
  • Прямой MyLog::create([...]) в прод-коде не используется (остались только закомментированные фрагменты).

3) Чтение и фильтры: AuditLogQueryScopes

Трейт App\Support\BuildsLogTable подключает scope-фильтры только по event и level:

Метод / параметрПоведение
applyCategoryScope($category)whereIn('event', …) по категории из AuditEvent::eventValuesForCategory()
filter_action = domain.actionwhere event = …
filter_action=unknownevent IS NULL, пустой event или значение вне реестра AuditEvent
filter_levelwhere level = info|security|integration
hide_authorizations=1исключить event = auth.login
hide_integrations=1исключить level = integration и integration-event
Числовой filter_action (legacy-код)игнорируется (фильтр не применяется)

Строки без event (только legacy type/action в БД) в категорийные модалки и в фильтры по событию не попадают. На проде после backfill таких строк быть не должно.

4) Отображение в DataTables

  • Колонка «Действие» в JSON — ключ action (историческое имя поля DataTables), значение — MyLog::eventLabel() → AuditEvent::resolveLabel($event).
  • Если event пустой или неизвестный — подпись «Неизвестное событие».

5.1) Единый паттерн UI «История»

На страницах разделов CRM (см. таблицу ниже) используется один и тот же UX: toolbar → кнопка с иконкой fa-clock-rotate-left и подписью «История» (на узких экранах — только иконка) → Bootstrap-модалка #historyModal из includes/logModal.blade.php → при открытии вызывается showLogModal(url) с URL GET …/logs-data. Контроллер подключает трейт BuildsLogTable и метод log() / schoolScheduleLogs(): JSON DataTables с колонками дата, автор, действие (eventLabel), описание; фильтр по категории через applyCategoryScope() — только события одного домена, без legacy type/action.
  • Права — endpoint logs-data находится в той же middleware-группе, что и index страницы (например contracts.view, districts.view). Без права — 403; гость — 302/401.
  • Partner-scope — строки только текущего партнёра (partner_id из current_partner).
  • Запись без изменений — для update-событий лог не пишется, если diff пустой (справочники, шаблоны, заявки).
  • Удаление — события delete обычно с level = security; при блокировке удаления (422) лог не создаётся.
