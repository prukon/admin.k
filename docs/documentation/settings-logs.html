Настройки → Логи: журнал изменений (my_logs)

Файл: /docs/documentation/settings-logs.html
Назначение. Отдельная вкладка «Настройки → Логи» (/admin/settings/logs) — единый журнал аудита по таблице my_logs для текущего партнёра (все домены событий). Раньше тот же контент открывался модальным окном «История изменений» на вкладке «Общие»; модалка убрана, просмотр только через вкладку. Каноническая модель данных и запись — в разделе Аудит CRM: my_logs (event / level).
Permission viewing.all.logs (в справочнике: «Просмотр всех логов») означает доступ к журналу всех типов событий в рамках разрешённого partner-scope, а не «логи всех партнёров» для обычного админа. Cross-partner (все школы сразу) — только у роли superadmin на этой вкладке. На отчёте «Платежи» то же право даёт кнопку «История» T‑Bank — см. отчёт «Платежи».

1) Доступ и права

ЧтоPermission / рольHTTPПримечание

Контекст CRM: auth + 2fa + SetPartner. Гость и пользователь без прав получают 403/редирект на всех endpoint’ах раздела. См. партнёр-контекст.

2) Маршруты

  • GET /admin/settings/logs — HTML (вкладка внутри admin.setting.index, activeTab = logs).
  • GET /admin/settings/logs-data — JSON для jQuery DataTables (server-side).

Имена маршрутов: admin.setting.logs, settings.logs.data (routes/web.php).

3) Интерфейс

  • Шапка в стиле отчёта «Платежи» / «Комиссии Т‑Банк»: карточка payments-report-*, CSS resources/css/admin-list-toolbar.css (подключается через Vite на вкладке).
  • Кнопка «Фильтры» — сворачиваемая панель с полями фильтрации и кнопками «Применить» / «Сброс» (перезагрузка DataTables без перезагрузки страницы).
  • Таблица — server-side DataTables, по умолчанию 10 строк на странице, сортировка по умолчанию: «Дата создания» по убыванию.

3.1 Колонки таблицы

  1. ID
  2. Дата создания (created_at, формат d.m.Y / H:i:s)
  3. Партнёр — только для superadmin (колонка partner_title)
  4. Действие — подпись из MyLog::eventLabel() по колонке event (реестр AuditEvent); ключ JSON — action (историческое имя колонки DataTables)
  5. Автор (author → ФИО пользователя)
  6. Что меняли (target_label)
  7. Описание (description, переносы строк → <br> в ячейке)

3.3 Логика фильтров на сервере

Реализация: App\Support\AuditLogQueryScopes + BuildsLogTable. Все условия по событиям — только колонка event, по уровню — level. Подробнее: audit-my-logs §3.
ФильтрПоведение
Скрыть суперадмина При hide_superadmin=1 исключаются записи, у которых автор — пользователь с ролью superadmin.
Скрыть авторизации Исключаются записи с event = auth.login.
Скрыть интеграции Исключаются записи с level = integration и события уровня integration из реестра.
Действие → «Неизвестный тип» filter_action=unknown — пустой/отсутствующий event или event вне реестра AuditEvent.
Уровень filter_level — точное совпадение с колонкой level.
  • Комбинация фильтров — условия объединяются через AND (дата + действие + уровень + автор + partner-scope + исключения).
  • Кнопка «Сброс» — «Скрыть суперадмина» вкл., «Скрыть авторизации» и «Скрыть интеграции» выкл., остальные поля пусто; для superadmin партнёр снова all.
  • DataTables AJAX всегда передаёт hide_superadmin, hide_authorizations, hide_integrations (1 или 0).

3.4 Отличие от модалок логов на других страницах

Параметры hide_superadmin, hide_authorizations, hide_integrations и расширенный селект «Действие» есть только на вкладке «Логи». Модалки «История изменений» на users/teams/schedule и др. вызывают тот же BuildsLogTable, но с ограничением по категории event и без этих чекбоксов — см. §5.1 и audit-my-logs.

4) Partner-scope: режим SUPERADMIN_ALL_OR_FILTER

На этой вкладке для API логов включён режим App\Support\PartnerScopeMode::SUPERADMIN_ALL_OR_FILTER (реализация в PartnerContext::applyPartnerScope(), вызов из SettingController::logsData).
  • STRICT_CURRENT (по умолчанию в модалках) — where partner_id = current_partner.
  • SUPERADMIN_ALL_OR_FILTER (только вкладка «Логи»):
    • superadmin + filter_partner_id пусто / all → без фильтра по partner_id;
    • superadmin + числовой filter_partner_id → один партнёр;
    • не superadmin → как STRICT_CURRENT.

Подробнее: partner-context.html, partner-scope-guide.

5) Данные и код

  • Таблица БД: my_logs — рабочие поля event, level, partner_id, author_id, target_*, description, created_at; legacy type/action только в архивных строках.
  • Запись: AuditLogger — см. audit-my-logs §2.
  • Чтение: BuildsLogTable + AuditLogQueryScopes — только event и level.
  • Вкладка «Логи»: buildLogDataTable(null, SUPERADMIN_ALL_OR_FILTER) — без ограничения категории (все события).
  • View: resources/views/admin/setting/logs.blade.php.
  • Контроллер: SettingController::showLogs(), logsData().

5.1 Модалки «История» на других экранах

Тот же билдер с категорией AuditEvent (фильтр whereIn(event, …)):

  • пользователи — GET /admin/user/logs-data (user);
  • группы — GET /admin/teams/logs-data (team);
  • расписание — GET /schedule/logs-data (schedule);
  • права и роли — GET /admin/setting/rules/logs-data (role);
  • партнёры — GET /admin/partner/logs-data (partner);
  • цены — GET /admin/setting-prices/logs-data (pricing).

6) Отличие от вкладки «Общие» и от queue.log

  • На вкладке «Общие» кнопка «История изменений» и модалка #historyModal удалены.
  • «Настройки → Очереди» и storage/logs/queue.log — операционный лог воркера, не my_logs (очереди в админке).
