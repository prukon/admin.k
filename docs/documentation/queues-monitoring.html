Очереди в админке: мониторинг и управление

Файл: /docs/documentation/queues-monitoring.html
Раздел описывает вкладку Настройки → Очереди, которая показывает реальное состояние database-очередей и даёт безопасные инструменты операционного контроля.

1) Доступ и права

  • Просмотр вкладки и данных: permission settings.queues.view.
  • Управление (перезапуск worker): permission settings.queues.manage.
  • Маршруты вкладки остаются внутри стандартного CRM-контекста (auth + 2fa + SetPartner). Метрики jobs/failed_jobs и heartbeat worker — глобальные. Счётчик и пример просроченных отложенных выплат T‑Bank для пользователя без роли superadmin ограничены текущим партнёром; для superadmin — по всем партнёрам.

2) Маршруты

  • GET /admin/settings/queues — страница вкладки (HTML).
  • GET /admin/settings/queues/status — JSON со статусом и метриками.
  • GET /admin/settings/queues/logs — JSON с последними строками queue-лога.
  • POST /admin/settings/queues/restart — выполняет queue:restart (graceful restart).

3) Что показывает вкладка

  • jobs_count — количество задач в таблице jobs.
  • failed_jobs_count — количество задач в failed_jobs.
  • oldest_job_age_seconds — возраст самой старой задачи в jobs.
  • last_success_at — время последней успешной обработки job.
  • last_failed_at — время последней неуспешной обработки job.
  • worker_status — вычисляемый статус worker: alive, stale, dead, no_data.
  • scheduler_last_tick_at — время последнего успешного тика планировщика (см. ниже).
  • scheduler_status — «живость» планировщика по последнему тику.
  • overdue_scheduled_payouts_count — число просроченных отложенных выплат T‑Bank (условие совпадает с TinkoffRunScheduledPayoutsJob).
  • overdue_scheduled_payouts_sample — до 15 строк для быстрого перехода в карточку выплаты.

4) Логика статуса worker (heartbeat)

  • heartbeat обновляется на Queue::looping и на событиях Queue::after/Queue::failing.
  • alive: heartbeat не старше 60 сек.
  • stale: heartbeat от 61 до 300 сек.
  • dead: heartbeat старше 300 сек.
  • no_data: heartbeat ещё не зафиксирован.

4.1) Планировщик Laravel (cron → schedule:run)

Отложенные выплаты T‑Bank не обязаны постоянно висеть в таблице jobs: срок хранится в tinkoff_payouts.when_to_run, а периодически джоба TinkoffRunScheduledPayoutsJob ставится в очередь планировщиком (после php artisan schedule:run). Поэтому в админке добавлен отдельный индикатор «последний тик планировщика».
  • В App\Console\Kernel зарегистрировано событие everyMinute, которое пишет timestamp в cache-ключ scheduler:last_tick_at (класс App\Support\SchedulerHeartbeat).
  • Если на сервере нет cron-строки вида * * * * * php artisan schedule:run, тик не обновляется — в UI будет no_data или статус «мертв».
  • Пороги для scheduler_status: до 120 сек — ок; 121–600 сек — предупреждение; больше 600 сек — критично.

4.2) Просроченные отложенные выплаты (T‑Bank)

  • Условие совпадает с выборкой в TinkoffRunScheduledPayoutsJob: when_to_run <= now(), status = INITIATED, tinkoff_payout_payment_id IS NULL, completed_at IS NULL.
  • На странице /admin/tinkoff/payouts выводится тот же блок (до 25 строк) для оперативного реагирования.

5) Разбивки в UI

  • По очередям: агрегируется по колонке queue в jobs/failed_jobs.
  • По типам задач: агрегируется по шаблонам class-name в payload: возвраты, CloudKassir-чеки, AI-блог, выплаты T-Bank, smoke-test.
В текущей архитектуре большинство задач отправляется в очередь default (без onQueue(...)), поэтому физическая разбивка по очередям может быть однотипной. Для оперативного анализа используйте также разбивку по типам задач.

6) Логирование очередей

  • Выделен отдельный канал queue в config/logging.php.
  • Файл канала: storage/logs/queue.log.
  • Во вкладке выводятся последние строки именно этого канала (не общий laravel.log).

7) Перезапуск worker

  • Кнопка запускает php artisan queue:restart через контроллер.
  • Это мягкий (graceful) перезапуск: текущая job завершается, затем worker перезапускается.
  • Действие логируется в канал queue с user_id.

8) Контроль корректности (feature-тесты)

  • Покрыты проверки доступа: отдельные права settings.queues.view и settings.queues.manage.
  • Покрыты happy-path сценарии: страница 200, статус 200, логи 200, restart 200.
  • Покрыта структура JSON и базовые метрики статуса очередей.
  • Покрыты поля scheduler_status, overdue_scheduled_payouts_count; в расписании присутствует событие * * * * * для heartbeat планировщика.
  • Страница выплат: отображение блока при наличии просроченной записи.
