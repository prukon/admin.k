Юр. лица (админка): /admin/legal-entities

Файл: /docs/documentation/admin-legal-entities.html · T‑Bank sm-register: tbank §2.1 · группы: admin-teams · обзор справочников: directories-hierarchy
Справочник «Юр. лица» — реквизиты и ShopCode T‑Bank (sm-register) для каждого юридического лица партнёра (школы). Один партнёр может иметь несколько юр. лиц; одна группа привязана к одному юр. лицу (teams.legal_entity_id), одно юр. лицо — ко многим группам.
Страница входит в раздел CRM «Справочники» — вкладка «Юр. лица» ($activeTab = 'legal-entities'). Пункт сайдбара «Справочники» / «Юр. лица» — через directories.view + право конкретного справочника (App\Support\DirectoriesMenu).
Регистрация в sm-register выполняется только здесь, не на карточке партнёра. Legacy-маршруты tinkoff/partners/sm-* отключены (410 JSON / redirect с предупреждением).
Переходный период: поле legacy partners.tinkoff_partner_id и реквизиты на партнёре сохранены. При оплатах ShopCode берётся из активного юр. лица (partner_legal_entities.tinkoff_shop_code), при отсутствии — fallback на partners.tinkoff_partner_id (LegalEntityResolver::shopCode()). Данные из partners в partner_legal_entities не мигрируются автоматически — заполняются вручную в CRM.

1) Модель данных

Миграции (DDL only, без INSERT из partners):

  • 2026_06_27_*_create_partner_legal_entities_table.php — таблица partner_legal_entities (soft delete);
  • 2026_06_27_*_add_legal_entity_id_to_teams_table.php — teams.legal_entity_id, FK nullOnDelete;
  • снимки legal_entity_id на tinkoff_payments, tinkoff_payouts, fiscal_receipts;
  • 2026_06_27_160000_add_legal_entities_permissions.php — права legal_entities.view, legal_entities.manage.
Таблица / полеНазначение
partner_legal_entities partner_id, business_type (enum: OOO, IP, ANO, NKO), title, реквизиты, tinkoff_shop_code, sm_register_status, taxation_system, vat, is_default, is_enabled, soft delete. Уникальность ИНН: (partner_id, tax_id) среди не удалённых.
teams.legal_entity_id Необязательная явная привязка группы к юр. лицу. При удалении юр. лица — SET NULL. См. admin-teams.
is_default Основное юр. лицо партнёра (ровно одно при нескольких активных). Первое созданное становится default автоматически.

1.1) Выбор юр. лица для оплаты (LegalEntityResolver)

группа.legal_entity_id (активное) → это юр. лицо
иначе, если активных юр. лиц = 1 → единственное
иначе, если есть is_default → default (fallback=true)
иначе → первое активное (fallback=true)

ShopCode = entity.tinkoff_shop_code ?? partners.tinkoff_partner_id (legacy)

Сервис: App\Services\PartnerLegalEntities\LegalEntityResolver. Используется в PaymentService, Init T‑Bank, выплатах, CloudKassir. DTO: LegalEntityResolution (entity, usedDefaultFallback).

4) Интерфейс

4.1) Список (admin/legal-entities/index.blade.php)

  • DataTables (KidsCrmDataTable), фильтры: поиск, статус active/inactive.
  • Колонки: наименование (ссылка на карточку), форма, ИНН, ShopCode, зарегистрировано, основное, число групп, активность, действия (legal_entities.manage).
  • Модалки create/edit — AJAX (fetch + X-Requested-With); ошибки 422 под полями.
  • Non-AJAX safety-net: нативный POST/PUT без JS → redirect 302, запись в БД создана/обновлена (см. тесты LegalEntitiesNonAjaxSafetyNetFeatureTest).

4.2) Карточка (admin/legal-entities/show.blade.php)

  • Блок CRM-полей (legalEntityCrudForm, PUT через AJAX) — налогообложение, НДС, default, enabled.
  • Блок sm-register (legalEntitySmForm) — полный профиль для T‑Bank; после получения ShopCode часть полей CRM блокируется (guardrails).
  • Кнопки «Обновить статус» / «Подтянуть из банка» — для зарегистрированных записей.

4.3) sm-register / sm-patch

Сервис: App\Services\Tinkoff\PartnerLegalEntitySmRegisterService, клиент: SmRegisterClient.

  • После успешной регистрации сохраняются tinkoff_shop_code, sm_register_status, реквизиты, registered_at.
  • Форма организации в CRM (OOO/IP/ANO/NKO) маппится в тип API sm-register (company / individual_entrepreneur / non_commercial_organization) — см. PartnerLegalEntityBusinessType::smRegisterApiType().
  • AJAX-ответ register: { "ok": true, "shopCode": "…", "status": "…" }; patch: { "ok": true }; ошибки — 422 JSON с message и errors.
  • Non-AJAX sm-register → redirect на show с flash ok.

6) Аудит (my_logs)

События: legal_entity.created, legal_entity.updated, legal_entity.deleted (категория legal_entity). Кнопка «История» на index → GET /admin/legal-entities/logs-data (logs.data.legal-entity).

7) Связь с группами и оплатами

  • Селект юр. лица в модалках create/edit группы — только если у партнёра ≥2 активных юр. лиц (legal_entities.view). Подробнее — admin-teams § legal-entities.
  • В таблице групп колонка «Юр. лицо»: явная привязка или бейдж fallback «По умолчанию: {title}».
  • Доступность T‑Bank на витрине: PaymentService::isTbankAvailable() — глобальный терминал + ShopCode через resolver (см. settings-payment-systems §4.3).
  • Интеграция с договорами — отдельная фаза (prefill из юр. лица в модалке договора пока не реализован).

7.1) Снимки legal_entity_id в UI T‑Bank

При Init, webhook и создании выплаты в tinkoff_payments, tinkoff_payouts, fiscal_receipts сохраняется снимок legal_entity_id (миграции 2026‑06). В админке отображение идёт по снимку, а не через повторный вызов LegalEntityResolver.
ЭкранИсточникПодписьДокументация
Список выплат /admin/tinkoff/payouts, колонка «Организация» tinkoff_payouts.legal_entity_id → JOIN partner_legal_entities organization_name → title → «—» tbank-admin-payouts §4.1
Карточка платежа /admin/tinkoff/payments/{id}, строка «Организация» tinkoff_payments.legal_entity_id, eager legalEntity (withTrashed) то же tbank §5.7.1
Колонка «Партнёр» на выплатах / «Партнёр» на карточке платежа partners.title название клуба в CRM, не юр. лицо —

Карточка выплаты /admin/tinkoff/payouts/{id} строку «Организация» не выводит.

8) Legacy partner sm-register

Маршруты POST tinkoff/partners/{id}/sm-register и аналоги возвращают redirect/410 с указанием перейти в «Юр. лица». UI sm-register на карточке партнёра скрыт. Поле partners.tinkoff_partner_id остаётся fallback для оплат до полного перехода.
