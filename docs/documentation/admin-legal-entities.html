Юр. лица (админка): /admin/legal-entities

Файл: /docs/documentation/admin-legal-entities.html · T‑Bank sm-register: tbank §2.1 · группы: admin-teams · обзор справочников: directories-hierarchy
Справочник «Юр. лица» — реквизиты и ShopCode T‑Bank (sm-register) для каждого юридического лица партнёра (школы). Один партнёр может иметь несколько юр. лиц; одна группа привязана к одному юр. лицу (teams.legal_entity_id), одно юр. лицо — ко многим группам.
Страница входит в раздел CRM «Справочники» — вкладка «Юр. лица» ($activeTab = 'legal-entities'). Пункт сайдбара «Справочники» / «Юр. лица» — через directories.view + право конкретного справочника (App\Support\DirectoriesMenu).
Регистрация в sm-register выполняется только здесь, не на карточке партнёра. Legacy-маршруты tinkoff/partners/sm-* отключены (410 JSON / redirect с предупреждением).
Переходный период: поле legacy partners.tinkoff_partner_id и реквизиты на партнёре сохранены. При оплатах ShopCode берётся из активного юр. лица (partner_legal_entities.tinkoff_shop_code), при отсутствии — fallback на partners.tinkoff_partner_id (LegalEntityResolver::shopCode()). Данные из partners в partner_legal_entities не мигрируются автоматически — заполняются вручную в CRM.

Таблица / полеНазначение
partner_legal_entities partner_id, business_type (enum: OOO, IP, ANO, NKO), title (внутреннее; при создании из partners.title, в модалках CRM не показывается), organization_name (публичное наименование, обязательно в CRM), реквизиты (ИНН, КПП, ОГРН, адрес, банк), ceo (JSON), sm_details_template, sms_name, tinkoff_shop_code, sm_register_status, vat (ставка НДС для онлайн-чека), is_default, is_enabled, soft delete. is_default Основное юр. лицо партнёра (ровно одно при нескольких активных). Первое созданное становится default автоматически.

1.1) Выбор юр. лица для оплаты (LegalEntityResolver)

Init / выплата по payable:
  team_id из PayableTeamResolver → teams.legal_entity_id (активное) → это юр. лицо
  без team_id → entity = null (forPartner для нового payable не используется)

Общая логика выбора entity по группе:
  группа.legal_entity_id (активное) → это юр. лицо
  иначе, если активных юр. лиц = 1 → единственное
  иначе, если есть is_default → default (fallback=true)
  иначе → первое активное (fallback=true)

ShopCode = entity.tinkoff_shop_code ?? partners.tinkoff_partner_id (legacy)

Сервис: App\Services\PartnerLegalEntities\LegalEntityResolver. Используется в PaymentService, Init T‑Bank, выплатах, CloudKassir. DTO: LegalEntityResolution (entity, usedDefaultFallback). Группа платежа — payments §2.3.

4) Интерфейс

4.1) Список (admin/legal-entities/index.blade.php)

  • DataTables (KidsCrmDataTable), фильтры: поиск (название, ИНН; при sm_register — также ShopCode), статус active/inactive.
  • Колонки: наименование (публичное — displayTitle(): organization_name → title), форма, ИНН, ShopCode (только legal_entities.sm_register), Регистрация в банке (бейдж Да/Нет; при «Нет» — иконка i + kids-tooltip-hint, текст «Обратитесь к администратору платформы», scope hint после draw.dt), основное, группы, активен, действия (legal_entities.manage).
  • В колонке «Действия»: «Изменить» (модалка) и «Регистрация» (ссылка на карточку) — последняя только при legal_entities.sm_register.
  • Модалки create/edit — partial crud-fields.blade.php, AJAX (fetch + X-Requested-With); ошибки 422 под полями.
  • Удаление из модалки edit: подтверждение showConfirmDeleteModal → DELETE; при успехе edit-модалка закрывается (снятие hidden.bs.modal.return у confirm); при 422 (есть привязанные группы) — showErrorModal('Удаление юр. лица', …), edit-модалка остаётся закрытой (без повторного открытия).
  • Non-AJAX safety-net: нативный POST/PUT/DELETE без JS → redirect 302, запись в БД создана/обновлена/удалена (см. LegalEntitiesNonAjaxSafetyNetFeatureTest).

4.2.1) Публичное наименование (displayTitle())

Метод модели PartnerLegalEntity::displayTitle(): organization_name (если не пусто) → иначе title. Используется в:

  • колонке «Наименование» списка юр. лиц (PartnerLegalEntityController::data());
  • колонке «Юр. лицо» таблицы групп (TeamController::data(), поле JSON legal_entity_label);
  • селектах legal_entity_id в модалках create/edit группы (displayTitle() в Blade).

Внутреннее title (часто совпадает с названием клуба partners.title) в этих местах не выводится, если заполнено organization_name. Для оплат и T‑Bank по-прежнему действует LegalEntityResolver (см. §1.1).

4.3) Карточка регистрации (admin/legal-entities/show.blade.php)

  • Доступ: legal_entities.sm_register (HTML). Без него — 403, даже при manage.
  • Инфо-блок: ShopCode, статус sm-register, версия/дата реквизитов, основное, число групп.
  • Форма sm-register (legalEntitySmForm, partial sm-form-fields.blade.php) — полный профиль для T‑Bank; доступна при legal_entities.manage. Без manage — alert «Редактирование доступно при праве…».
  • Кнопки «Обновить статус» / «Подтянуть из банка» — для зарегистрированных записей (sm-refresh, sm-pull, manage).
  • Блок «Реквизиты CRM» на карточке удалён — изменение через модалку на списке.

4.4) sm-register / sm-patch

Сервис: App\Services\Tinkoff\PartnerLegalEntitySmRegisterService, клиент: SmRegisterClient.

  • После успешной регистрации сохраняются tinkoff_shop_code, sm_register_status, реквизиты, registered_at.
  • Форма организации в CRM (OOO/IP/ANO/NKO) маппится в тип API sm-register (company / individual_entrepreneur / non_commercial_organization) — см. PartnerLegalEntityBusinessType::smRegisterApiType().
  • AJAX-ответ register: { "ok": true, "shopCode": "…", "status": "…" }; patch: { "ok": true }; ошибки — 422 JSON с message и errors.
  • Non-AJAX sm-register / sm-patch → redirect на show с flash ok.
  • Форма sm-register включает поля CEO (ФИО, телефон) и страну адреса; приоритет данных CEO из формы (PartnerLegalEntitySmRegisterService::resolveCeo()).

5.1) НДС и CloudKassir

  • Ставка НДС (partner_legal_entities.vat) — задаётся в модалке CRM, поле «Ставка НДС (онлайн-чек)»; используется в чеках CloudKassir через LegalEntityResolver::fiscalVat() (CloudKassirReceiptBuilder::resolveVat()).
  • Система налогообложения (СНО) в чеке — только из env платформы CLOUDKASSIR_TAXATION_SYSTEM (CloudKassirReceiptBuilder::resolvePlatformTaxationSystem()). Поля taxation_system у партнёра и юр. лица удалены (миграция 2026-06-29).

6) Аудит (my_logs)

События: legal_entity.created, legal_entity.updated, legal_entity.deleted (категория legal_entity). Кнопка «История» на index → GET /admin/legal-entities/logs-data (logs.data.legal-entity).

7) Связь с группами и оплатами

  • Селект юр. лица в модалках create/edit группы — только если у партнёра ≥2 активных юр. лиц (legal_entities.view). Подписи опций — displayTitle(). Подробнее — admin-teams §5.2.
  • В таблице групп колонка «Юр. лицо»: значение — displayTitle() выбранного или default-юр. лица; при fallback (legal_entity_fallback: true) рядом иконка i с подсказкой «Юр лицо установленное по умолчанию» (без текста «По умолчанию:» в ячейке).
  • Доступность T‑Bank на витрине: PaymentService::isTbankAvailable() — глобальный терминал + ShopCode через resolver (см. settings-payment-systems §4.3).
  • Интеграция с договорами — отдельная фаза (prefill из юр. лица в модалке договора пока не реализован).

7.1) Снимки legal_entity_id в UI T‑Bank

При Init, webhook и создании выплаты в tinkoff_payments, tinkoff_payouts, fiscal_receipts сохраняется снимок legal_entity_id (миграции 2026‑06). В админке отображение идёт по снимку, а не через повторный вызов LegalEntityResolver.

8) Legacy partner sm-register

Маршруты POST tinkoff/partners/{id}/sm-register и аналоги возвращают redirect/410 с указанием перейти в «Юр. лица». UI sm-register на карточке партнёра скрыт. Поле partners.tinkoff_partner_id остаётся fallback для оплат до полного перехода.
