T‑Bank: возврат платежа и отмена отложенной выплаты партнёру

Файл: /docs/documentation/tbank-refunds-payout-cancel.html · список разделов (/doc)
Описывает актуальную логику отчёта «Платежи»: кнопка «Возврат», API POST /admin/reports/payments/{payment}/refund и согласованность с таблицей tinkoff_payouts. Общий обзор T‑Bank и e2c: tbank.html, отчёт и права: reports-payments.html, админка списка выплат: tbank-admin-payouts.html.

1) Два разных «PaymentId»

  • Платёж эквайринга (оплата ученика): идентификатор в нотификациях T‑Bank, в CRM хранится в payments.payment_id / payments.payment_number и в payment_intents (tbank_payment_id, provider_inv_id).
  • Выплата партнёру (e2c): после вызова банка e2c/v2/Init в tinkoff_payouts.tinkoff_payout_payment_id появляется отдельный идентификатор операции выплаты. Пока поле пустое, запрос Init в банк ещё не выполнялся (или Init завершился без успешного присвоения id — см. код).

2) Связь «оплата → выплата» в базе

  • Строка оплаты в смысле мультирасчётов: tinkoff_payments (внутренний PK id, поле tinkoff_payment_id — строка, как отдаёт банк).
  • Строка выплаты: tinkoff_payouts. Поле tinkoff_payouts.payment_id — это FK на tinkoff_payments.id, а не на payments.id отчёта.
  • tinkoff_payouts.deal_id дублирует сделку мультирасчёта для API и отображения; отмена выплаты при возврате не опирается на deal_id (ни из CRM payments, ни иначе) — только на цепочку через tinkoff_payments.

3) Политика доступности возврата

Запрет возврата, если для текущего партнёра найдены выплаты, привязанные к тем же tinkoff_payments, что и возвращаемый платёж, и среди них есть строка со статусом не REJECTED и с непустым tinkoff_payout_payment_id (выплата уже ушла в банк на этапе e2c Init).
  • Разрешён возврат, если таких «активных с банковским id» выплат нет: в том числе когда выплата в статусе INITIATED с пустым tinkoff_payout_payment_id (отложена по when_to_run или ещё не подобрана джобой).
  • Перед созданием возврата все подходящие выплаты (тот же tinkoff_payments.id, не REJECTED, без банковского PaymentId) переводятся в REJECTED, обнуляется when_to_run, в payload_state пишется cancelled_by_refund (время, актор, id строки отчёта payments).
  • Если по возвращаемому платежу не удаётся найти строку tinkoff_payments (по tinkoff_payment_id из CRM и при необходимости по tbank_order_id из intent), автоматическая отмена выплат не выполняется — возврат в эквайринг может пройти, а «висящая» выплата без корректной связи потребует ручной разборки данных.

4) Как находится tinkoff_payments при возврате

  • Из строки отчёта берётся числовой T‑Bank PaymentId оплаты (payments.payment_id или payment_number).
  • Загружается payment_intents (tbank, тот же партнёр, совпадение по provider_inv_id / tbank_payment_id).
  • Кандидаты tinkoff_payments.id: все строки партнёра с tinkoff_payment_id = эта строка (как в БД), плюс строки с order_id = intent.tbank_order_id (если заполнен) — на случай расхождения строкового tinkoff_payment_id в tinkoff_payments с полем в отчёте.
  • По объединённому списку id оплат выбираются все tinkoff_payouts с payment_id из этого списка; строки лочатся в одной транзакции с блокировкой платежа отчёта.

5) Согласованность с джобой отложенных выплат

  • TinkoffRunScheduledPayoutsJob перед runPayout берёт строку выплаты в транзакции с lockForUpdate и проверяет, что она всё ещё INITIATED, без tinkoff_payout_payment_id, с наступившим when_to_run.
  • После успешного возврата выплата становится REJECTED и не попадает в выборку джобы — повторный Init в банк по отменённой записи не выполняется.

6) UI отчёта «Платежи»

  • Колонка refund_action: кнопка «Возврат» отключается по той же логике, что и серверный запрет: через tinkoff_payments → выплаты с непустым tinkoff_payout_payment_id и статусом не REJECTED.
  • Только INITIATED без банковского id кнопку не блокирует.

7) Права доступа

  • reports.view — страница отчёта, getPayments, настройки колонок, POST …/refund. Без права — 403.
  • Платёж должен принадлежать current_partner (иначе 403 на возврате).
Легаси и «сироты»: если в tinkoff_payouts указан payment_id, не совпадающий ни с одной строкой tinkoff_payments для этой оплаты, такая выплата при возврате не отменяется автоматически. Поле deal_id для этого не используется.
