Заполнение договора родителем (личный кабинет)

Файл: /docs/documentation/account-contract-fill.html · UI: /account-settings/documents (вкладка «Мои документы») · см. также шаблоны договоров, договоры в CRM

1) Назначение и доступ

После создания договора в режиме «форма клиенту» (creation_mode=template) ученик (родитель в семейном кабинете) видит задачу «Требуется заполнение», открывает модалку, заполняет поля из схемы шаблона, формирует PDF и подписывает договор по SMS (Подпислон).

Право на раздел: account.documents.view (Gate::define('account.documents.view')).
Маршруты fill / generate / sign дополнительно защищены middleware contract.owner — только владелец договора (contracts.user_id === Auth::id()); чужой пользователь получает 404 (anti-enumeration).
Гость → 302 на login; без права → 403.

2) Модалка заполнения (#contractFillModal)

  • Ширина: класс contract-fill-modal-dialog (max-width 720px).
  • Контент подгружается AJAX: GET …/fill с заголовком X-Requested-With: XMLHttpRequest.
  • При ?fill= на index вызывается JS loadContractFill(contractId).
  • Маска телефона: jQuery Inputmask +7 (999) 999-99-99, класс js-contract-fill-phone.

2.2) Поля ФИО и даты

Составные ключи DOCX (parent_full_name, child_full_name) в форме разбиты на части: parent_lastname, parent_firstname, parent_middlename, child_lastname, child_firstname. При генерации PDF полные ФИО собираются в ContractPdfGenerationService / ContractTemplateVariablePresets::composeNameFieldsForPdf.

child_birthday — <input type="date">, валидация date + before:today (AccountContractGenerateRequest). Значения в PDF и filled_data — в формате d.m.Y; в профиле ученика — Y-m-d.

2.3) Порядок полей

Поля сортируются по fill_sort_order из fields_schema версии шаблона (или пресета из ContractTemplateVariablePresets). Рекомендуемая шкала: 10–50 основные данные, 200+ супруг(а), 300+ кастомные поля из DOCX. Редактирование порядка в CRM — только с правом contracts.templates.fillSortOrder.edit (по умолчанию только superadmin).

2.4) Предзаполнение

ContractPrefillResolver подставляет значения из карточки ученика/родителя/группы по prefill_source в схеме. После успешного generate данные синхронизируются обратно:

  • ContractParentProfileSyncService — ФИО родителя в parents
  • ContractStudentProfileSyncService — фамилия, имя, дата рождения ученика в users

3) Жизненный цикл статусов (кабинет)

СтатусUI в модалке
awaiting_client_fillФорма «Сформировать договор»; доступен fill JSON
generating_pdfСпиннер, poll: true в JSON — автообновление модалки
draft + PDFБлок подписания SMS + «Скачать PDF»
sent / openedFill JSON → 422 «Договор недоступен для заполнения»; подсказка про SMS на странице при других сценариях
Срок заполнения: fill_expires_at (по умолчанию +7 дней от создания, Contract::FILL_TTL_DAYS). После истечения в статусе awaiting_client_fill fill JSON возвращает 422: «Срок заполнения договора истёк…».

4) Валидация generate

AccountContractGenerateRequest: обязательные поля по required в схеме; системные поля (contract_date, contract_id…) не показываются родителю. Ошибки возвращаются redirect на ?fill= с withErrors под ключами fields.{key}.

5) Скрытое право «показывать ключи полей»

Право account.contracts.showFieldKeys (скрытое в БД, не в базовых ролях): при наличии под полем показывается технический ключ {{…}} из DOCX.

6) Приглашение по email

При создании договора по шаблону (ContractCreationService::notifyClientAboutFill) на email ученика уходит ContractClientFillInvitationMail. Текст берётся из версии шаблона (email_subject, email_body_html) или системных дефолтов (ContractTemplateEmailDefaults); плейсхолдеры подставляет ContractInvitationEmailRenderer ({{child_full_name}}, {{partner_name}}, {{documents_url}}, {{fill_deadline}}, {{contract_id}}). Если у ученика нет email — договор создаётся, письмо не отправляется.
