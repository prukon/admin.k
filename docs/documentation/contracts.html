Договоры (клиентские)

Файл: /docs/documentation/contracts.html · раздел CRM: /client-contracts · оглавление: /doc

1) Назначение и доступ

Раздел для работы школы с PDF-договорами учеников: список, создание, карточка, отправка на подпись (Подпислон / SMS), email, скачивание файлов.
Все маршруты обёрнуты в middleware('can:contracts.view') (routes/web.php). Без права — 403 (HTML и JSON).
Маршруты с {contract} дополнительно проверяют contract.partner (договор того же партнёра, что current_partner).
Синхронизация статуса с Подпислоном: GET …/status — отдельно can:contracts.sync.

2) Родитель и ученик на карточке договора

Договор привязан к ученику (contracts.user_id → users). ФИО родителя хранится в полях ученика — см. admin-users §2.1 (parent_lastname, parent_firstname, parent_middlename).

2.1) Блок основной информации

На GET /client-contracts/{id} в карточке (<dl>) порядок строк:

ПодписьИсточник
Родитель$student->parent_full_name или «—», если пусто
Ученик$student->full_name (фамилия + имя ученика)
Группаteams.title по user.team_id
Статусcontracts.status (бейдж)
Телефонusers.phone (контакт родителя)
Почтаusers.email

2.2) Загрузка ученика в контроллере

В ContractsController@show ученик выбирается явным select(). В выборку обязательно входят parent_lastname, parent_firstname, parent_middlename (плюс name, lastname, phone, email, team_id). Если их не включить, в UI родитель и предзаполнение подписанта будут пустыми при заполненной карточке ученика.

3) Предзаполнение подписанта (модалка «Отправка на подпись»)

Кнопки «Отправить СМС на подпись» / «Повторно отправить» открывают модалку #sendModal. Значения по умолчанию (до правки оператором):

Поле в модалкеИсточник в БДОтправка в API
Фамилия (#signerLastname)parent_lastnamesigner_lastname (ContractSendRequest)
Имя (#signerFirstname)parent_firstnamesigner_firstname
Отчество (#signerMiddlename)parent_middlenamesigner_middlename (nullable)
Телефон (#signerPhone)phone (10 цифр без +7 для маски)signer_phone → канон 7XXXXXXXXXX

3.1) Подпислон

Провайдер: App\Services\Signatures\Providers\PodpislonProvider. В PUT /add-document передаются name (имя), last_name (фамилия), second_name (отчество), phone — из данных формы отправки, не напрямую из users.parent_* после сохранения договора.
После отправки в contract_sign_requests фиксируется снимок подписанта; сменить подписанта у уже созданного договора нельзя — только новый договор или новая отправка с новыми данными в форме.

4) Скачивание PDF в интерфейсе

Правило единое для карточки договора в CRM и для вкладки «Мои документы» в кабинете ученика.
  • Статус signed (Contract::STATUS_SIGNED) — кнопка «Скачать подписанный» (signed_pdf_path).
  • Любой другой статус — «Скачать оригинал» (source_pdf_path).
  • Критерий кнопки — поле contracts.status, а не только наличие пути к файлу.
  • Файлы: диск local, storage/app/ + относительный путь из БД.

5) Создание договора и заявки с сайта

  • После привязки ученика к заявке — ссылка «Создать договор» → GET /client-contracts/create?user_id={id}.
  • Ученик предвыбирается на форме (preselectedUser); ФИО родителя на форме создания не показывается — заполняется в карточке ученика до отправки SMS.
  • Списание с баланса партнёра — только при POST /client-contracts (загрузка PDF).
  • Полный сценарий лид → клиент → договор: school-leads-widget.html (§5.5–5.7).
