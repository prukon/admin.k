Договоры (клиентские)

Файл: /docs/documentation/contracts.html · раздел CRM: /client-contracts · оглавление: /doc

1) Назначение и доступ

Раздел для работы школы с PDF-договорами учеников: список, создание, карточка, отправка на подпись (Подпислон / SMS), email, скачивание файлов.
Все маршруты обёрнуты в middleware('can:contracts.view') (routes/web.php). Без права — 403 (HTML и JSON).
Маршруты с {contract} дополнительно проверяют contract.partner (договор того же партнёра, что current_partner).
Синхронизация статуса с Подпислоном: GET …/status — отдельно can:contracts.sync.

2) Родитель и ученик на карточке договора

Договор привязан к ученику (contracts.user_id → users). ФИО родителя берётся из справочника parents (связь users.parent_id → parentProfile). См. родители и семейный кабинет и admin-users §2.1.

2.1) Блок основной информации

На GET /client-contracts/{id} в карточке (<dl>) порядок строк:

ПодписьИсточник
Родитель$student->parent_full_name (из parents через parentProfile) или «—»
Ученик$student->full_name (фамилия + имя ученика)
Группаteams.title по user.team_id
Статусcontracts.status (бейдж)
Телефонusers.phone (контакт родителя)
Почтаusers.email

2.2) Загрузка ученика в контроллере

В ContractsController@show (и legacy ContractsController) ученик загружается с ->with('parentProfile') и полями id, name, lastname, phone, email, team_id, parent_id. ФИО для UI — через $student->parentFormFields() / parent_full_name, не из колонок users (legacy parent_* на users удалены).

3) Предзаполнение подписанта (модалка «Отправка на подпись»)

Кнопки «Отправить СМС на подпись» / «Повторно отправить» открывают модалку #sendModal. Значения по умолчанию (до правки оператором):

Поле в модалкеИсточник в БДОтправка в API
Фамилия (#signerLastname)parentFormFields()['parent_lastname'] ← parents.lastnamesigner_lastname (ContractSendRequest)
Имя (#signerFirstname)parentFormFields()['parent_firstname']signer_firstname
Отчество (#signerMiddlename)parentFormFields()['parent_middlename']signer_middlename (nullable)
Телефон (#signerPhone)phone (10 цифр без +7 для маски)signer_phone → канон 7XXXXXXXXXX

3.1) Подпислон

Провайдер: App\Services\Signatures\Providers\PodpislonProvider. В PUT /add-document передаются name (имя), last_name (фамилия), second_name (отчество), phone — из данных формы отправки (оператор может изменить перед SMS), не напрямую из БД после отправки.
После отправки в contract_sign_requests фиксируется снимок подписанта; сменить подписанта у уже созданного договора нельзя — только новый договор или новая отправка с новыми данными в форме.

4) Скачивание PDF в интерфейсе

Правило единое для карточки договора в CRM и для вкладки «Мои документы» в кабинете ученика.
  • Статус signed (Contract::STATUS_SIGNED) — кнопка «Скачать подписанный» (signed_pdf_path).
  • Любой другой статус — «Скачать оригинал» (source_pdf_path).
  • Критерий кнопки — поле contracts.status, а не только наличие пути к файлу.
  • Файлы: диск local, storage/app/ + относительный путь из БД.

5) Модалка «Создать договор»

Отдельной страницы создания нет: форма в модалке #createContractModal на GET /client-contracts (resources/views/contracts/partials/create-modal.blade.php).
Кнопка «Создать договор» на списке и ссылка из заявок открывают ту же модалку. GET /client-contracts/create — только redirect на index с query-параметрами (create=1, опционально user_id).

5.1) Поля формы (порядок сверху вниз)

ПолеЭлементПоведение
Ученик#user_id (Select2, AJAX)Обязательное. Поиск по имени/фамилии/телефону/email ученика. Отправляется в POST /client-contracts как user_id.
ФИО родителя#parent_full_name_displayТолько просмотр (disabled, readonly). В форму не отправляется. По умолчанию «—»; после выбора ученика — parent_full_name из справочника parents или «—», если родитель не указан.
Группа#group_id_select + hidden group_idRead-only: подставляется из users.team_id выбранного ученика.
Партнёрdisabled inputТекущий current_partner.
Способ созданияcreation_modepdf — прикрепить PDF; template — форма клиенту (DOCX-шаблон).
PDF / шаблон#input_pdf или #contract_template_idВ зависимости от режима; см. contract-templates.

5.2) Источник ФИО родителя в модалке

Тот же источник, что на карточке договора (§2): User::parent_full_name через parentProfile (parents). Legacy-колонки parent_* в users не используются.
СценарийОткуда берётся значение
Выбор ученика в Select2AJAX GET /client-contracts/users-search → поле parent_full_name в каждом элементе results[] (ContractLookupsController::usersSearch, join parents)
Предзаполнение по ?user_id=View-data preselectedUser.parent_full_name (ContractsController::createFormContext, eager parentProfile)
Очистка выбора ученика / сброс модалкиСнова «—» (setParentFullNameDisplay в JS)

Поиск ученика в Select2 не ищет по ФИО родителя — родитель только отображается после выбора ученика. Редактировать ФИО родителя в модалке нельзя; изменение — в карточке ученика (admin-users §2.1).

5.3) Предзаполнение и открытие модалки

  • ?create=1, ?user_id={id} или ошибки валидации после POST — флаг shouldOpenCreateModal, модалка открывается при загрузке страницы.
  • Валидный user_id своего партнёра (is_enabled = 1) → preselectedUser: id, text, team_id, team_title, parent_full_name (или null).
  • Чужой партнёр / несуществующий id — preselectedUser = null, модалка всё равно может открыться по query.
  • Переход из заявок: GET /client-contracts/create?user_id=… → redirect на index с теми же параметрами (§school-leads-widget §5.7).

5.4) Сохранение и оплата

  • Submit модалки → подтверждение «Создать договор?» → POST /client-contracts/check-balance → POST /client-contracts.
  • Списание с баланса партнёра — только при успешном POST /client-contracts, не при открытии модалки.
  • После создания ученика и группы изменить нельзя — только новый договор.

6) Заявки с сайта и создание договора

  • После привязки ученика к заявке — ссылка «Создать договор» → GET /client-contracts/create?user_id={id} (redirect на список с открытой модалкой).
  • Ученик и ФИО родителя предзаполняются в модалке, если у ученика есть запись в parents.
  • На странице /admin/users — иконка PDF последнего договора в колонке «Договор» (только при contracts.view, без кнопки «Создать договор»).
  • Полный сценарий лид → клиент → договор: school-leads-widget.html (§5.5–5.7).
